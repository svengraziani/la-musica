# P33 Comp-segment and take selection rendering in the GraphCompiler/render path

> **Area:** audio-engine · **Severity:** medium · **Effort:** M · **Phase:** Phase 2 — Faithful Engine & Content Rendering
> **Depends on:** P09, P17 · **Blocks:** none

## Objective
Wire the existing comping model into the actual render and export path so a chosen composite take renders the selected segment at every timeline position. Today `ClipTake`/`ClipTakeLane`/`ClipComp`/`ClipCompSegment` (`libs/session/include/lamusica/session/AudioClipEditing.hpp:34-62`) and `renderCompedClip` (`libs/session/src/AudioClipEditing.cpp:277`) exist as a pure offline helper, but `compileProjectAudioGraph`/`makeClipNode` (`libs/session/src/GraphCompiler.cpp:175-212`) emit exactly one Sine/Sample node per clip and never consult a comp, and `ProjectManifest` (`libs/session/include/lamusica/session/ProjectManifest.hpp:122-140`) has no comp/take fields at all — so a comp is currently orphaned from both the graph and the saved project. After P33 the compiler resolves the active `ClipComp` for an audio clip into per-segment Sample sources/offsets with click-free boundary crossfades, comps/takes persist through P08 sidecars and round-trip through save/open, and offline bounce/export reflect the comp — all nondestructively (source take media byte-unchanged).

## Dependencies
- `P09 — Faithful GraphCompiler: render real user audio/MIDI content and add an instrument/synth graph node kind`
- `P17 — Live latency-compensated recording with input monitoring, punch/count-in, and take handling`

## Deliverables
- **Change** `libs/session/src/AudioClipEditing.cpp:277-311` `renderCompedClip`: apply a real crossfade across each segment boundary. Today the loop hard-cuts (writes segment B starting at its `clipStartSample` with no overlap), so the existing `renderLinearCrossfade` helper (`libs/session/src/AudioClipEditing.cpp:313-349`) is never invoked from comping. Add an equal-power (or configurable) crossfade of N samples straddling each boundary, sourced from both adjacent takes, so `maxAdjacentSampleDelta` (`libs/session/src/AudioClipEditing.cpp:351-370`) stays below a click threshold. Extend `CrossfadeRenderOptions` (`AudioClipEditing.hpp:13-15`) or add a comp-level crossfade length, and relax `validateClipTakeLane` (`AudioClipEditing.cpp:118-191`) so adjacent segments may share the boundary overlap region without tripping the non-overlap check at line 176.

Fades (`clipFadeGain`) apply only at the clip's own outer edges, not at internal segment seams.

Bump `schemaVersion` and add a migration in `migrateProjectManifest`.

## Decisions To Make
- **Persistence shape:** inline `comps`/`takeLanes` in the JSON manifest vs. P08 binary sidecar references (mirroring `MidiClipReference`). Sidecars keep the manifest small and match P08's autosave/recovery; inline is simpler to round-trip. Pick one and document the schema.
- **Graph representation:** one Sample node per segment (simple, leans on existing graph machinery and per-node fades) vs. a new comp-aware `GraphNodeKind` that owns multiple sources internally. Per-segment nodes are recommended for P33; revisit if node-count blowup hurts large comps.
- **Crossfade law and length:** linear (already implemented in `renderLinearCrossfade`) vs. equal-power (constant-power, better for correlated takes of the same performance). Choose a default crossfade length (samples or ms at project SR) and whether it is per-comp, per-segment, or global; equal-power is the safer default for comped vocal/instrument takes.
- **Boundary semantics:** whether segment boundaries store the crossfade region explicitly (overlap), or the crossfade is derived symmetrically around a zero-width boundary. This drives whether `validateClipTakeLane` must permit overlap and how the UI (P21/P23) authors boundaries.
- **Take source SR / channel mismatch:** policy when a take source's sample rate or channel count differs from the project. Defer the actual resampling to P29 (project sample-rate field + asset resampling), but P33 must validate and reject (or flag) mismatches rather than render garbage.
- **Muted takes / empty comp:** define the render when a segment's take is `muted` (currently skipped at `AudioClipEditing.cpp:292`) or when a clip has no comp — fall back to the plain single-clip node path.

## Compromises And Tradeoffs
- **Accepted:** P33 ships crossfade-at-boundaries comping over decoded full-take buffers loaded into Sample nodes; it does not add real-time take switching, comp automation, or streaming of huge takes from disk. Offline/export correctness is the bar; live RT playback of comps inherits whatever the Sample node path (P09) already does.
- **Accepted:** per-segment Sample nodes increase node and buffer count proportional to segment count. Acceptable for typical comps; flagged as a perf risk for pathological comps (revisit in P26 deadline verification).
- **Rejected MVP shortcut:** destructively flattening the comp into a new audio asset on edit. The mission requires nondestructive comping — source takes must remain byte-identical (checksum-verified), so the comp is always resolved at compile/render time.
- **Rejected:** hard-cut boundaries without crossfade. The existing `renderCompedClip` does exactly this today and would click; P33 explicitly fixes that by routing through a crossfade.
- **Risks later tasks must revisit:** SR/channel reconciliation (P29), take-source streaming for memory (relates to P10 streaming WAV), and UI authoring of comp boundaries/crossfade handles (P21/P23). This task covers the gap "comp-segment rendering uncovered" — the model exists but nothing wired it into render/export.

## Acceptance Gates
- `cmake --build --preset <default> && ctest --preset <default> --output-on-failure` passes, including the new comp-render, nondestructive-checksum, round-trip, and determinism cases in `tests/unit/bootstrap_tests.cpp`.
- A two-segment comp compiled via `compileProjectAudioGraph` and bounced through `bounceGraphToWav` renders take A for segment A's span and take B for segment B's span, with the boundary region a crossfade — assert `maxAdjacentSampleDelta` over the rendered output stays below a click threshold (the existing helper at `AudioClipEditing.cpp:351` already proves the crossfade path smooths a ±1 fixture boundary, ref test line 798).
- Edit the comp (swap which take a segment uses) → exported PCM differs; a checksum/hash of every `ClipTakeSource` PCM is byte-identical before and after the edit.
- `serializeProjectManifest` → `parseProjectManifest` round-trips comps and take lanes with bit-exact float fields (locale-independent), and `validateProjectManifest` rejects a comp referencing a missing take id / out-of-range segment (mirror the existing `validateClipTakeLane` errors at `AudioClipEditing.cpp:166-188`).
- Determinism: exporting the same comped project at block sizes 64, 128, and 512 yields byte-identical PCM (this is a hard gate carried by P26's determinism suite).
- Edge cases that must pass: zero-length / single-segment comp falls back to the plain clip node; muted take in a segment renders silence for that span without breaking adjacent crossfades; a comp on a non-audio (MIDI) clip is rejected or ignored, not crashed.

## Notes For Agents
- **Realtime/safety:** all comp resolution, decoding, validation, and crossfade math happen at compile time on the message/offline thread inside `compileProjectAudioGraph`; the audio callback only consumes the resulting validated `AudioGraph` snapshot (P05). No allocation, file IO, JSON, or comp logic on the audio thread — `renderGraph` (`libs/audio/src/AudioGraph.cpp:205`) must keep operating only on prebuilt `node.samples`.
- **Nondestructive contract:** never write to a take source. `renderCompedClip`/the compiler read from `ClipTakeSource::audio` (`AudioClipEditing.hpp:59-62`) and copy into per-segment node buffers. Add the checksum assertion to lock this in.
- **The real fix in the helper:** `renderCompedClip` at `AudioClipEditing.cpp:289-308` writes each segment's samples with a hard cut (`rendered.interleavedSamples[clipFrame*channels+channel] = source...`). There is no call to `renderLinearCrossfade`. Either fold a crossfade into this loop at boundaries or compose per-segment `renderClipRegion` results through `renderLinearCrossfade` (`AudioClipEditing.cpp:313`). The validator's non-overlap rule (`AudioClipEditing.cpp:176-178`) currently forbids the overlap a crossfade needs — relax it deliberately for the crossfade region.
- **Sample node positioning:** the Sample render branch (`libs/audio/src/AudioGraph.cpp:258-265`) reads `node.samples` via `sampleNodeValue`, gated by `nodeActiveAtSample` and shaped by `clipEnvelopeGain`/fades; per-segment nodes must set `startSample`/`sourceOffsetSamples`/`lengthSamples`/`sampleChannels`/`sampleFrames` consistently with how `makeClipNode` fills them at `GraphCompiler.cpp:207-210` from a decoded WAV.
- **Decode reuse:** segment take sources decode through `audio::readPcm16Wav` exactly like `makeClipNode` (`GraphCompiler.cpp:206`); reuse the P09 path-traversal guard and the P08 sidecar/asset resolution rather than adding a second reader.
- **Mapping to existing tasks docs:** this completes the audio-path half of comping that the UI tasks assume. The legacy roadmap covers the comping UI in `docs/tasks/029-audio-clip-editing-ui-waveforms.md` (clip editing) and recording/takes UX in `docs/tasks/033-browser-assets-import-record-export-ux.md`; performance/determinism verification lands in `docs/tasks/035-performance-stress-realtime-verification.md`. No task in P01–P27 (nor docs/tasks/026–036) renders comps — P33 is the render/export completion of that feature.
