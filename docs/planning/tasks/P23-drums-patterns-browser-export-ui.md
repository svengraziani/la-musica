# P23 Drum machine, step sequencer, clip-launcher, browser/inspector, and export UIs

> **Area:** daw-ui · **Severity:** high · **Effort:** XL · **Phase:** Interactive DAW UI surfaces (P19–P23)
> **Depends on:** P08, P16, P21, P22 · **Blocks:** P30, P34

## Objective
Replace the placeholder status-panel window with the remaining production editing surfaces: a full drum rack (pads, sample assignment, choke groups, mute/solo, velocity layers, per-pad routing), a step sequencer (variable length/resolution, swing, probability, ratchets, ties, slides, accents, per-lane length), and a clip-launcher (scene/slot grid with quantized launch), all driving the existing MIDI/instrument model, persisted to binary sidecars (P08), and rendered into exports. Ship a fully interactive piano roll (note draw/select/move/resize/delete, velocity, controller lanes, quantize, audition, fold/scale/ghost), a navigable Browser (asset/media/device tree, import/drag, tags/favorites/search, missing-file repair, background analysis) and a selection-driven editable Inspector. Wire an off-thread Export dialog (mix/loop/stems/package, format/range options, progress, result/error feedback) to the existing offline render/bounce path so the exported audio is bit-identical to the engine's offline render. This is the production surface, not an MVP demo: every edit is command-authored, undoable, persists across save/open, and plays through the live engine.

## Dependencies
- `P08 — Versioned binary sidecar format for MIDI, automation, plugin state, and waveform/analysis caches, with migration registry and autosave/recovery`
- `P16 — Live MIDI input/output ingestion and routing to instrument nodes`
- `P21 — Arrangement timeline with clip/automation editing, plus high-level audio clip editing UI`
- `P22 — Mixer, plugin-management, and recording UIs`

## Decisions To Make
- **Clip-launcher data model and launch semantics:** quantization grid set (bar/beat/1/16/none), retrigger vs legato follow-action, and whether a scene launch is one command or a batched transaction. No precedent exists in `libs/session`; design it to reuse `patternPlaybackEventsInRange` and the P21 timeline scheduler rather than a parallel clock.
- **Bundled drum-kit/sample licensing:** `DrumMachinePreset` already carries `license`, `licenseUrl`, `bundledAssetsIncluded`, and `hasClearDrumPresetRedistributionRights` (`DrumMachine.hpp:39-44,108`). Decide which factory kits ship in the app bundle, under what license (CC0 vs CC-BY vs none-bundled), and whether AGPL of the host constrains sample redistribution. Block any bundled kit that fails `hasClearDrumPresetRedistributionRights`.
- **Quantize/swing rounding policy:** piano-roll quantize and step swing must round to integer sample positions deterministically and identically across export and live playback (swing already clamps to ±0.5 and uses `std::llround`, `Pattern.cpp:24-26`). Decide strength/swing UI ranges and the single shared rounding helper.
- **Background analysis threading and cancellation:** `MediaAnalysisJob` is synchronous today (`Assets.hpp:69-75,213-217`). Decide the executor (JUCE `ThreadPool`), max concurrency, cancellation on browser navigation, and how results cross back to the message thread without blocking playback.
- **Selection model ownership:** whether the Inspector reads a central selection service or each view publishes selection; pick one to avoid drift between piano-roll/drum/clip selection and Inspector contents.
- **Export-while-playing policy:** allow concurrent offline export during live playback (separate offline graph copy) vs require transport stop; affects whether the dialog needs a snapshot of the manifest at launch time.

## Compromises And Tradeoffs
- **Accepted:** the underlying drum/pattern/piano-roll/asset logic stays in `libs/session` (headless, offline, unit-tested); this task is mostly view + command-binding + the new clip-launcher model. Reuse beats reimplementation.
- **Accepted:** clip-launcher and step sequencer share the MIDI/pattern model and the timeline scheduler; we do not build a separate Ableton-style session clock in this task — full session/arrangement-clock divergence is deferred.
- **Rejected MVP shortcut:** no read-only or "preview" panels. The Inspector and all editors must be command-authored and undoable through the unified history (P03), not direct mutation of `ApplicationSession` snapshots (`ApplicationSession.hpp:354`).
- **Rejected MVP shortcut:** export must not re-render on the message thread or use a second code path; it must call the same offline bounce as `exportProjectMixToWav` so audio is identical (the existing export assertions at `bootstrap_tests.cpp:400-467` are the floor).
- **Risk later tasks must revisit:** this surface is built before P30 (accessibility) and P34 (onboarding/templates) — every new view needs AX/keyboard hooks (P30) and must be reachable from templates (P34). The clip-launcher model added here will also need warp/stretch interaction (P28) and SR reconciliation on imported slots (P29).
- **Risk:** Browser background analysis and drag-drop touch the same `AssetCatalog` the engine reads; concurrent mutation must be serialized on the message thread, never blocking the audio callback (covers_gaps: "background analysis never blocks playback").

## Notes For Agents
- **Realtime safety:** views and the Export dialog run on the JUCE message thread; the audio engine runs on the P04 realtime thread. UI edits reach the engine only as P03 commands mirrored to the engine via the P05 validated snapshot + lock-free SPSC queue — never mutate the live `AudioGraph` directly and never allocate/log/parse-JSON/do file-IO on the audio callback. Live MIDI audition (drum pads, piano-roll, step preview) must route through the P16 live-MIDI path, not a side channel.
- **Command authoring:** route every edit through the unified `ICommand` history (P03); do not extend the whole-manifest snapshot list `undoSnapshots_` (`ApplicationSession.hpp:354`). Coalesce drag operations (note move/resize, velocity scrub) into single undo entries.
- **Persistence:** drum kits, patterns, the new scene/slot grid, and timeline organization (track folders, arranger sections, color labels — covers_gaps) persist via the P08 binary sidecar + migration registry; do not add ad-hoc JSON. Serialization helpers already exist for drums/patterns (`serializeDrumMachinePreset`/`parseDrumMachinePreset` `DrumMachine.hpp:106-107`) — wire them into the sidecar, do not duplicate.
- **Code pointers:** drum model `libs/session/include/lamusica/session/DrumMachine.hpp` (impl `libs/session/src/DrumMachine.cpp:264,321-343`); pattern model `libs/session/include/lamusica/session/Pattern.hpp` (impl `libs/session/src/Pattern.cpp:12-100`); piano-roll model+layout `libs/session/include/lamusica/session/PianoRoll.hpp`; assets/browser `libs/session/include/lamusica/session/Assets.hpp`; export `libs/session/include/lamusica/session/Export.hpp` + `libs/session/src/Export.cpp`; placeholder UI `apps/daw/src/main_juce.cpp`; smoke entry `apps/daw/src/main.cpp:697-742`; test floor `tests/unit/bootstrap_tests.cpp:400-467`.
