# P21 Arrangement timeline with clip/automation editing, plus high-level audio clip editing UI

> **Area:** daw-ui · **Severity:** blocker · **Effort:** XL · **Phase:** Phase 4 — Interactive DAW UI
> **Depends on:** P09, P19, P20 · **Blocks:** P23, P26, P30, P34

## Objective
Build the production arrangement surface — a real time ruler, one lane per track, audio clips with cached waveform overviews, MIDI clips with note previews, a live draggable playhead, a full tool model, grid snapping, multi-select, zoom/scroll with large-session virtualization, and a minimap — rendered from live `ProjectManifest` data through the interactive shell (P19) and the transport bridge (P20), playing back through the faithful render engine (P09). Every edit (move/trim/split/slip/fade/gain/duplicate/remove, automation breakpoints, tempo/timesig map, markers, folders, take-comping) is authored as an undoable command on the unified history (P03/P07), is fully nondestructive against source media, and is mirrored to the audio graph via validated snapshots. This replaces the canned `StatusPanel` placeholder in `apps/daw/src/main_juce.cpp` and consolidates docs/tasks 028, 029, and the arrangement-side of 032 onto the real engine. This is the production arrangement editor, not an MVP grid.

## Decisions To Make
- **Waveform cache format & residency:** sidecar-backed multi-resolution pyramid vs recompute-on-zoom; bucket precision (`float` min/max/rms is in place — keep, or add `int16` packing for large sessions?); eviction policy and memory cap for many open clips.
- **Snap model surface:** the engine `SnapSettings` exposes `sampleGrid`/`beatGridSamples`/snap-to-clips/markers/transients but not an explicit bar/division enum — decide whether the UI derives bar/division grids from the tempo map locally or we extend `SnapSettings`. Define magnetic-snap pixel threshold and modifier-key override (temporarily disable snap).
- **Split semantics:** confirm `makeSplitClipCommand` produces two independent clips sharing one immutable source (offset-based), and define crossfade-on-adjacent-split behavior. Decide multi-clip batch edits = one coalesced command vs N commands (coalescing policy is P03's; pick the grouping here).
- **Take comping authority:** `ClipTakeLane`/`ClipComp`/`ClipCompSegment` exist in the session layer but comp-segment *rendering* is owned by P33. Decide what the timeline can edit now vs render-gate behind P33; do not duplicate render logic in the UI.
- **Stretch/pitch UI gating:** expose warp/stretch/pitch controls as disabled/"coming soon" until P28, or hide entirely. Pick the capability-flag mechanism so P28 can light them up without UI churn.
- **Virtualization strategy:** lane recycling + clip culling against `visibleRange` vs full retained-mode; target session size (track count × clip count) the minimap and scroll must stay responsive at.

## Compromises And Tradeoffs
- **Accepted:** lean on the already-tested session/command layer (clip commands, `AudioClipEditing`, `Timeline` layout, `Automation`) — the UI is a thin, command-emitting view over verified logic. Waveform/MIDI previews and stretch/pitch are surfaced progressively; warp/stretch/pitch UI ships gated behind P28.
- **Rejected MVP shortcuts:** no direct manifest mutation from the UI (all edits are commands → undoable, MCP-addressable later via P25); no destructive in-place audio writes (source media stays byte-identical — checksum-gated); no fixed single-track view (`buildTimelineLayout` already renders ordered multitrack); no synchronous waveform generation on the message/paint thread; no canned `StatusPanel` text (`main_juce.cpp:112-117`).
- **Risks later tasks must revisit:** (covers_gaps) comp-segment playback fidelity is finalized in **P33** — the timeline must not assume final render shape; stretch/pitch/warp DSP correctness is **P28**; mixer-side automation surfaces overlap with **032/P22** — keep the breakpoint editor reusable across arrangement and mixer; accessibility (keyboard operability, VoiceOver, reduced-motion) of all gestures is deferred to **P30** but gesture code must expose AX-friendly hooks now to avoid a rewrite; RT-deadline/determinism verification of edit→snapshot→render is consolidated in **P26**.

## Acceptance Gates
- `cmake --build build --target lamusica_daw && ctest --test-dir build --output-on-failure` passes; new timeline gesture/layout tests are registered and green.
- **Real data render:** launching the DAW shows tracks/clips/markers from the live `ProjectManifest` (not `StatusPanel` text); audio clips show waveforms, MIDI clips show note previews. Verified by a layout snapshot test asserting `buildTimelineLayout` output drives the painted geometry.
- **Playhead/seek:** during P20 playback the playhead advances smoothly; clicking/dragging the ruler seeks the engine (`AudioEngine::seekSamples`, `AudioEngine.hpp:106`) and the displayed bar/beat matches `samplesToBarBeat` (`AudioEngine.hpp:113`).
- **Command-backed & snapped:** move/trim/split/fade each produce the corresponding `ICommand`, land on the snap grid for the selected resolution (`snapSample`, `Timeline.cpp:232`), and undo/redo restores prior state exactly; multi-select delete is a single undoable step.
- **Nondestructiveness:** a test edits a clip (split + trim + fade + gain + reverse), saves, reopens, and asserts the source asset file checksum is byte-identical before and after; only the manifest/sidecar changed.
- **Click-free fades/crossfades:** render a faded/crossfaded clip via `renderClipRegion`/`renderLinearCrossfade` and assert no inter-sample discontinuity above threshold at fade boundaries (no zipper/click).
- **Automation/tempo/markers/comps:** edit each, render/export, and assert the output reflects the edit (e.g. automation curve changes the gain envelope per P18), then assert persistence (P08 round-trip) and undo.
- **Virtualization:** on a large fixture (hundreds of tracks/clips) scroll+zoom stays responsive and only on-screen clips are materialized (assert culling count against `visibleRange`); waveform generation runs off the message thread and progressively refines.
- **Layout integrity:** screenshots at laptop (1280×800, current `setSize` in `main_juce.cpp:66`) and a larger desktop size show no text overlap or clipped labels in headers/ruler/inspector.

## Notes For Agents
- **Realtime safety (hard rules):** the timeline/UI thread never touches the audio callback directly. Edits mutate the manifest via commands on the message thread, then a validated graph snapshot is published to the audio thread through the lock-free `RealtimeCommandQueue` (`AudioEngine.hpp:74`; impl `AudioEngine.cpp:146`). No allocation/locks/file-IO/JSON/logging on the render path — waveform generation, overview caching, and disk-IO all run on background workers and feed the audio side only via RT-safe queues/snapshots (P05).
- **Reuse the engine layer; do not reimplement in JUCE.** Layout: `buildTimelineLayout` (`Timeline.cpp:122`), `orderedTimelineTracks` (`:276`), `clipsOnTrack`, `snapSample` (`:232`), `zoomTimelineAroundSample` (`:98`), `setTimelinePlayhead` (`:85`), `setTimelineSelection`/`selectionReferencesExistingItems`. Edits: the clip command classes in `Command.hpp:65-223` and `makeSplitClipCommand` (`Command.hpp:1062`). Audio edit math: `AudioClipEditing.hpp`/`.cpp`. Automation: `Automation.hpp`. These are unit-tested already — the UI is the missing layer.
- **Playhead binding** comes from P20's transport bridge; consume the engine position rather than running a UI timer free of the engine. `ApplicationSession` already tracks `playheadSample`/`transportPlaying` (`ApplicationSession.hpp:48-50`).
- **Waveform pipeline:** `WaveformOverview`/`WaveformBucket` and the async `MediaAnalysisJob` (`Assets.hpp:45-77`) already exist — wire the UI to request/observe jobs and persist results to the P08 sidecar cache; invalidate via `WaveformOverview::valid` on edit/source change.
- **Capability gating:** wrap warp/stretch/pitch and final comp rendering behind feature flags (DSP = P28, comp render = P33) so this task ships without their backends.
- **Gotcha:** do not delete the legacy `apps/daw/src/main.mm` (1991 lines) or `main.cpp` as part of this task unless P19/P20 already retired them — confirm which shell entrypoint is active in `apps/daw/CMakeLists.txt` first; this task targets the JUCE `main_juce.cpp` shell.
- **Maps to existing tasks:** implements **028** (timeline render/tools/snapping/virtualization/minimap), **029** (audio clip editing UI + waveform cache + take/comp foundation), and the automation/tempo/marker/comping deliverables shared with **032** — all consolidated onto the real engine (P09) instead of the placeholder shell. Mixer channel-strip/routing portions of 032 stay with P22; comp-segment render stays with P33.
