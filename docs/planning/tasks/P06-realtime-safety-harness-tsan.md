# P06 Real realtime-safety enforcement harness and TSan CI job (replace the tautological audit)

> **Area:** realtime-perf-security · **Severity:** blocker · **Effort:** L · **Phase:** Phase 1 — Realtime Engine Foundation
> **Depends on:** P05 · **Blocks:** P26

## Objective
Replace the fabricated realtime-safety "assurance" with real instrumentation that observes what the audio callback actually does. Today `auditRealtimeGraphCallback` hardcodes its operations list and string-matches a denylist, so `realtimeCallbackSafe` is structurally always true and a bootstrap test asserts this as "proof". The production outcome is: an audit/debug-build trap that arms around the live `renderGraphBlock` callback (global `operator new`/`delete` trap, mutex-lock trap, file-IO/syscall detection), records the genuinely OBSERVED operations, flips an atomic violation flag from the audio thread, and a dedicated ThreadSanitizer CI job that exercises the SPSC queue and the cross-thread command->snapshot->graph path as a required check. A deliberately bad callback must fail; the real one must pass with observed-operation provenance.

## Dependencies
- `P05 — Validated audio-graph snapshot, precompiled noexcept render path, and true lock-free SPSC command queue` (the trap arms around P05's noexcept render path; the TSan job exercises P05's atomic SPSC `RealtimeCommandQueue`).

  - a mutex-lock trap (wrapper or interposition over the lock acquisition used on the audio path) that records a `lock` violation when armed;
  - file-IO/syscall detection (a thin `open`/`read`/`write`/`fopen` interposition or a guarded wrapper) recording `file_io`;

## Decisions To Make
- Trap mechanism: global `operator new`/`delete` override vs a sanitizer-allocator hook vs malloc interposition. Decide whether the trap is always-recording (sets a flag) or aborts in-callback; recommended: record-only on the RT thread (no abort/log/throw on the audio thread), evaluate the flag on the non-RT side after the block.
- Syscall/file-IO detection scope: full `open`/`read`/`write`/`mmap` interposition vs a narrow set of wrappers the engine is known to (not) call. Decide acceptable coverage vs portability (macOS-only `DYLD_INTERPOSE` vs portable weak-symbol overrides).
- Lock-trap strategy: a project `RtMutex` wrapper used everywhere on the audio path vs interposing `pthread_mutex_lock`. Decide whether third-party/system locks inside `renderGraphBlock` are in scope (they should not be reached today; P11/P12 plugin hosting will reintroduce risk).
- TSan scope and runtime: which targets get `-fsanitize=thread` (unit + new threaded test, not the JUCE app), TSan suppressions file policy, and CI time budget (TSan is ~5-10x slower). Decide whether TSan runs on every push or only PRs/main.
- How `renderGraphBlock`'s current `throw std::runtime_error` on malformed graphs (e.g. `AudioEngine.cpp:392,398,403,408,452`) interacts with the trap: throwing is not RT-safe, but it fires only on misconfiguration, not steady state. Decide whether the audit treats a throwing path as a violation or as out-of-scope (recommended: P05's validated-snapshot precondition makes these unreachable on the RT path; the trap records but the bad-callback test targets alloc/lock/IO specifically).
- Whether `RealtimeSafetyReport`'s `noLogging`/`noJsonParsing`/`noMcpWork` flags (header lines 40-48) get real detectors now or remain policy-only string checks deferred to P24/P25 (MCP) and the logging task. Recommended: implement alloc/lock/file_io detectors for real; keep log/json/mcp as documented policy until those subsystems touch the RT path.

## Compromises And Tradeoffs
- Accepted: the trap is audit/debug-build only; release builds carry no `operator new` override or interposition (zero RT overhead). The "safety" guarantee is enforced in CI, not at runtime in shipping binaries.
- Accepted: record-only (flag) semantics on the audio thread instead of aborting, because abort/log/throw are themselves RT-unsafe; verdict is computed off the RT thread.
- Rejected MVP shortcut: keeping the hardcoded operations list "for now". That is exactly the false-green this task exists to delete (`Performance.cpp:257-258`, `bootstrap_tests.cpp:2171`). No partial retention.
- Rejected: asserting safety purely from a static denylist of strings without executing the callback under instrumentation.
- Risk later tasks must revisit: P11/P12/P13 (plugin hosting) and P15/P16/P18 will add nodes that can allocate/lock/do IO inside the render path; the trap and the bad-callback corpus must be extended to cover plugin-node `process` calls and parameter delivery. P09 (instrument/synth node) and P28 (time-stretch DSP) likewise. P26 consumes this harness as a required gate, so its coverage must be broad enough to catch regressions introduced by those tasks.
- Risk: the trap depends on P05 having actually made `RealtimeCommandQueue` lock-free atomic; the current `size_` counter (`AudioEngine.cpp:146-184`) is shared non-atomic state and would be flagged by TSan — that is the desired signal, and P05 must land first.

## Notes For Agents

The trap itself must be allocation-free when armed (use a `thread_local bool armed` and `std::atomic` flags with `relaxed`/`acquire-release` ordering; never `std::cout`/`fprintf` from inside an armed override). Compute and report the verdict only after `RealtimeTrapScope` disarms.

The threaded test must exercise true single-producer/single-consumer concurrency so TSan validates the atomics' memory ordering; do not add locks to "fix" a TSan report — fix the ordering.
