# P17 Live latency-compensated recording with input monitoring, punch/count-in, and take handling

> **Area:** audio-engine · **Severity:** blocker · **Effort:** L · **Phase:** Phase 3 — Live Engine I/O (record/monitor) & MIDI ingestion
> **Depends on:** P04, P10, P14 (engine-latency-reporting half only — see Dependencies) · **Blocks:** P22, P26, P33

## Objective
Make LaMusica capture **real hardware input** through the live audio callback and place it on the timeline sample-accurately — not the current offline graph re-render that fabricates a take by replaying the existing project (`libs/session/src/ApplicationSession.cpp:1504-1520`). Frames arriving on the device input callback are pushed across a realtime-safe queue to an off-thread streaming WAV writer (P10), the take is latency-compensated using measured input round-trip latency plus reported engine output latency (P14), armed-track input is monitored audibly with auto/in/off modes, and punch-in/out plus pre-roll/count-in operate on the live stream. The outcome is a production tracking workflow: arm, count in, record a multi-minute take with no dropouts or unbounded memory, and have it land in the arrangement at the right sample.

P17 may proceed in parallel with the freeze half of P14; it must not be serialized behind freeze.

## Deliverables
- **Live input capture path (new).** `libs/audio/include/lamusica/audio/InputCaptureQueue.hpp` + `src/InputCaptureQueue.cpp`: a bounded, lock-free SPSC ring of interleaved float blocks (mirror the existing `RealtimeCommandQueue` pattern, `AudioEngine.hpp:74-89`) sized for capture latency (e.g. ≥ several device blocks). `push` runs on the audio callback (no alloc/lock/log), `pop`/drain runs on a writer worker thread. Add an overrun counter (lock-free flag) for the UI/tests.
- **Callback-side capture in the device host (change P04 code).** In `AudioDeviceHost::audioDeviceIOCallbackWithContext`, when recording is armed, copy the device input channels into the capture queue immediately (before render mutates buffers). The callback must remain allocation/lock/IO-free.
- **Streaming writer worker (new).** `libs/audio/include/lamusica/audio/RecordingWriter.hpp` + `src/RecordingWriter.cpp`: a thread that drains `InputCaptureQueue` into P10's streaming WAV writer (`.tmp` → atomic rename on commit, reusing the recovery contract in `Recording.cpp:109-138`). Replaces `RecordingSession::appendInterleaved` accumulating into `captured_.interleavedSamples` (`Recording.cpp:196-208`); refactor `RecordingSession` to own the queue+writer instead of one heap vector, preserving its commit/discard/recovery API (`Recording.hpp:100-122`).
- **Latency-compensated take placement (wire existing, dead code).** `measureRecordingLatency` (`libs/audio/src/Recording.cpp:55-75`) and `latencyAlignedStart` (`Recording.cpp:35-38`) exist but `measuredInputLatencySamples` is hardcoded to `0` at the only real caller (`ApplicationSession.cpp:1518`). Implement an actual round-trip measurement pass (emit a reference impulse to output, capture it back via the same queue, run `measureRecordingLatency`) and feed the result into `RecordingOptions.measuredInputLatencySamples` (`Recording.hpp:19`). Add `AudioEngine::engineLatencySamples()` (from P14) into the offset so `committed.timelineStartSample = armedStart − inputRoundTrip − engineOutputLatency`. Persist `measuredInputLatencySamples` on the take (already on `CommittedRecording`, `Recording.hpp:27`).
- **Software input monitoring with auto/in/off modes (new behavior).** `inputMonitoringEnabled` is currently pure metadata threaded through `RecordingWorkflowOptions`/`RecordingPlan`/`FirstTrackRecordingOptions` (`Recording.hpp:51,71`) with no audio effect. Add a monitoring mode enum (`Off`/`In`/`Auto`) and route the armed track's captured input through that track's insert chain to the master output inside the live render block — `Auto` monitors only while transport is stopped or punched-out (i.e. when the playback of the track itself is not present), `In` always monitors, `Off` mutes. Must avoid input→output feedback (do not monitor a track whose source is the same device output bus).
- **Punch + pre-roll/count-in on the live stream (change).** `makeRecordingPlan` (`Recording.cpp:77-107`) already computes `captureStartSample`, `punchInSample/punchOutSample`, `clipStartSample`, lead-in. Drive these against the **live** transport sample position in the callback: only commit captured frames inside the punch window to the timeline clip (capture may start earlier for pre-roll/look-back), and count-in plays the metronome (`AudioEngine::renderMetronome`, `AudioEngine.hpp:119`) without writing to the take.
- **Session/command integration (change).** Replace the offline re-render in `ApplicationSession::recordFirstTrackTake` (`ApplicationSession.cpp:1479-1520`) and generalize beyond "first track": add a command-authored arm/record API keyed on a track id (coordinate with P07's authoring API and P03 command history). Take handling already exists (`addTake`/`activateTake`/`TakeLane`, `Recording.cpp:140-185`); land the committed take into a take lane and an arrangement clip (`manifest.clips`, as `ApplicationSession.cpp:1529-1537` does today).
- **Tests (new).** `tests/integration/live_recording_tests.cpp` (loopback tone recorded back and read out of the committed WAV; click-against-metronome sample-accurate alignment within tolerance; multi-minute take with bounded memory + zero callback-thread allocation + overrun counter == 0; monitoring audible while armed / muted when Off / no feedback). Register it in `tests/CMakeLists.txt` (today only `unit/bootstrap_tests.cpp` is registered). Keep/extend the existing `RecordingSession` unit tests (`tests/unit/bootstrap_tests.cpp:5760-5835`) against the refactored streaming writer.

## Decisions To Make
- **Latency measurement source of truth.** Use the engine's measured input round-trip (impulse loopback) at session start, the device-reported input+output latency (`juce::AudioIODevice::getInputLatencyInSamples()` / `getOutputLatencyInSamples()`), or both with the device value as a fallback when loopback isn't possible (no monitor/aggregate route)? Decide tolerance (e.g. ≤ ±1 sample for synthetic loopback, ≤ a small block for hardware) and how stale measurements are invalidated on device change (P04 device swap).
- **Monitoring path: software vs. direct (low-latency) monitoring.** Software monitoring through the track chain adds round-trip latency (audible on the performer's ears); hardware/direct monitoring on the interface has zero added latency but bypasses the track chain. Decide whether P17 ships software-only (simplest, consistent), exposes a "direct monitoring (zero-latency, no effects)" toggle that simply mutes the software monitor, or both — and what `Auto` means precisely relative to punch state.
- **Record file format and bit depth.** `commit` currently writes PCM16 (`Recording.cpp:215`, `writePcm16Wav`), but `WavFile.hpp:18` already offers `writePcm24Wav`. Production tracking should default to 24-bit (or 32-bit float) to preserve headroom — decide the default and whether it is user-configurable, and align the streaming writer (P10) accordingly.
- **Capture queue sizing & overrun policy.** How many blocks deep is the SPSC ring (latency vs. resilience under writer-thread scheduling jitter), and on overrun do we drop-and-flag, or count and continue? A take with a non-zero overrun count must be surfaced, not silently corrupted.
- **Channel count & input selection.** Mono vs. stereo capture per armed track, which device input channel(s) map to a track, and how this interacts with P04's stereo-master assumption. Define the per-track input routing data even if the UI (P22) wires it later.
- **Pre-roll capture vs. punch commit semantics.** Whether pre-roll audio is retained on disk (look-back / "capture-before-arm") or discarded; this affects whether the writer streams continuously while armed or only inside the punch window.

## Compromises And Tradeoffs
- **Accepted:** Software monitoring carries the device round-trip latency unless direct monitoring is chosen; we document this rather than attempt a sub-block monitoring hack. Capture is per-armed-track stereo/mono; full multi-track simultaneous arm-and-record across many inputs may be staged (the queue/writer design must not preclude it — size for N tracks).
- **Rejected MVP shortcuts:** No keeping the offline graph re-render "for now" — `recordFirstTrackTake` re-rendering the existing project (`ApplicationSession.cpp:1504-1511`) is the exact gap this task closes and must be deleted, not wrapped. No accumulating the whole take in one `std::vector<float>` (`Recording.cpp:204-207`) — multi-minute takes must stream. No leaving `measuredInputLatencySamples` hardcoded to `0` (`ApplicationSession.cpp:1518`) — alignment is a Definition-of-Complete acceptance gate.
- **Risks later tasks must revisit:** Recording UI (arm buttons, input meters, monitor toggles, take lanes) is **P22** — P17 ships the engine/session substance and must expose lock-free status (overrun flag, monitoring state, take list) for it. Comp/take-segment rendering in the GraphCompiler is **P33** (depends on P17) — P17 must persist take lanes in a shape P33 can render. Project-SR vs device-SR mismatch (P04/P29) affects recorded-sample placement: a take captured at device SR into a project authored at another SR will misalign until P29 — document the interim constraint here. Determinism/RT verification of the capture path is folded into **P26**.

## Acceptance Gates
- `cmake --build build && ctest --test-dir build` passes, including the new `live_recording_tests` and the refactored `RecordingSession` unit tests.
- **Loopback capture:** arm + record with a synthetic loopback (engine output fed back into the capture queue, or a virtual/aggregate device gated behind an env var as P04 does for `audio_device_loopback_tests`) records the actual input; reading the committed WAV (`readPcm16Wav`/24) back yields the emitted tone, not a re-render of the project graph.
- **Latency alignment:** a click emitted at known sample N against the metronome, captured and committed, lands within tolerance of N after `measuredInputLatencySamples + engineLatencySamples()` compensation; assert `committed.timelineStartSample` equals the latency-aligned value (mirror the existing `timelineStartSample == 896` assertion at `bootstrap_tests.cpp:5773`, now driven by a real measurement rather than a literal).
- **Monitoring:** with mode `In`, the armed track's input is present in the master output block while armed; with `Off`, it is absent; assert no input→output feedback path is created (output level bounded, no runaway).
- **Punch/count-in on live stream:** count-in plays the metronome and writes nothing to the take; only frames inside `[punchInSample, punchOutSample)` reach the committed clip; pre-roll behaves per the chosen retain/discard decision.
- **Long take / RT-safety:** a multi-minute (≥ several minutes) capture commits with no dropouts, the capture-queue overrun counter is `0`, peak resident memory stays under a bounded ceiling (assert against a budget, not unbounded growth), and a callback-thread allocation check (TSan/no-alloc harness from P06) shows zero allocations in `push`/capture.
- **Crash recovery:** an interrupted take leaves a recoverable `.tmp` (`hasInterruptedRecording`/`recoverInterruptedRecording`, `Recording.cpp:109-138`) and recovers to a valid WAV.

## Notes For Agents
- **Realtime constraints (hard):** the input-capture copy runs on `audioDeviceIOCallbackWithContext` — no allocation, no locks, no file I/O, no logging, no JSON. Only a lock-free SPSC push. All disk writing happens on the writer worker thread. Model the queue on `RealtimeCommandQueue` (`AudioEngine.hpp:74-89`, fixed-capacity array, `push`/`pop`/`full`/`empty` all `noexcept`), but for variable-size interleaved blocks use a fixed-stride block ring sized at `maxBlockSize * channels`.
- **Where the dead/stub code is, precisely:** `latencyAlignedStart` (`libs/audio/src/Recording.cpp:35-38`) and `measureRecordingLatency` (`:55-75`) are implemented and unit-tested but never fed a non-zero measurement from a live path; the only production caller passes `0` (`libs/session/src/ApplicationSession.cpp:1518`). `inputMonitoringEnabled` is plumbed through `RecordingWorkflowOptions`→`makeRecordingPlan`→`RecordingPlan` (`Recording.hpp:51,71`; `Recording.cpp:106`) with **no** audio effect. The offline re-render to fabricate a take is `ApplicationSession.cpp:1504-1520` (`compileProjectAudioGraph` + `audio::renderGraph` into `captured`).
- **Engine latency surface (from P14):** P14 adds `[[nodiscard]] std::int64_t engineLatencySamples() const noexcept;` to `AudioEngine.hpp` and `totalEngineLatencySamples(graph)` in `LatencyCompensation.hpp`, and wires total latency into the `TransportState` reporting surface (`AudioEngine.hpp:40`). Consume that; do not recompute PDC here. If P14 hasn't landed the reporting half, add just that surface (small, mechanical) without blocking on freeze.
- **Streaming writer surface (from P10):** P10 delivers the incremental streaming-to-disk WAV writer; `RecordingWriter` should call into it rather than buffering. Today `commit` writes the whole `captured_` at once (`Recording.cpp:210-227`) and the reader slurps the entire data chunk (`WavFile.cpp:202-203`).
- **Maps to existing tasks docs:** This is the live-recording substance behind **`docs/tasks/009-audio-import-recording-export.md`** ("Recording pipeline with input monitoring, pre-roll, count-in, punch in/out, and take management", `009:14`) and **`docs/tasks/033-browser-assets-import-record-export-ux.md`** — both assume capture works; P17 makes it real. The recording **UI** in those docs moves to **P22**. It does not touch the freeze/bounce mandate (P14) or comp rendering (P33).
- **Coordinate, don't duplicate:** take handling (`addTake`/`activateTake`/`TakeLane`, `Recording.cpp:140-185`), recovery (`hasInterruptedRecording`/`recoverInterruptedRecording`), and the commit/atomic-rename contract already exist and are tested — refactor around them rather than reinventing. Generalize `recordFirstTrackTake`/`FirstTrackRecordingOptions` (`ApplicationSession.hpp:146-152`) into a track-id-keyed API consistent with the P07 multitrack authoring layer.
