# P16 Live MIDI input/output ingestion and routing to instrument nodes

> **Area:** audio-engine · **Severity:** high · **Effort:** L · **Phase:** Phase 4 — Live Performance & Recording
> **Depends on:** P04, P12 · **Blocks:** P22, P23

## Objective
Make LaMusica playable: a hardware or virtual MIDI keyboard produces audio in real time through hosted instrument nodes, and external gear can be driven from the transport. Enumerate and open CoreMIDI input/output devices, parse and time-stamp incoming events **off the audio thread**, deliver them to the device callback through a realtime-safe queue, and route them to the correct instrument (P12 `PluginInstrument` / P09 `Instrument`) node with sample-accurate offsets inside each render block. Add MIDI output and MIDI clock send/receive where required, capture live input into transport-relative recorded notes for commit, and persist the device configuration to a chosen store. This replaces the engine's precomputed-sine note handling (`AudioEngine.cpp:468-496`) with live, event-driven instrument rendering.

## Dependencies
- `P04 — Live CoreAudio device I/O on a realtime thread driving the engine callback` (provides the RT audio thread/callback, the JUCE host shim, and the device-config persistence seam this task plugs MIDI into).
- `P12 — Plugin graph node + RT-safe parameter/bypass delivery in the render path` (provides the `PluginInstrument` node and its preallocated per-instance `juce::MidiBuffer` scratch; P16 is what actually fills that buffer from a live source).

## Deliverables
- **New** `libs/audio/include/lamusica/audio/MidiDeviceHost.hpp` + `src/MidiDeviceHost.cpp` (in the P04 host target, *not* the JUCE-free `lamusica::audio` core): owns `juce::MidiInput`/`juce::MidiOutput`, enumerates devices via `juce::MidiInput::getAvailableDevices()` / `juce::MidiOutput::getAvailableDevices()`, opens enabled inputs, and implements `juce::MidiInputCallback::handleIncomingMidiMessage`. The callback runs on JUCE's MIDI thread (not the audio callback): it parses the `juce::MidiMessage`, converts to a POD event, time-stamps it against the audio clock, and `push`es it to the RT queue. No allocation/parsing/JSON/logging happens on the audio callback; parsing happens here, on the MIDI thread.
- **New** `libs/audio/include/lamusica/audio/MidiEventQueue.hpp` + `.cpp`: a bounded **lock-free SPSC** ring of POD MIDI events (`{kind, channel, data1, data2, sampleTimestamp, targetInstanceHandle}`), modeled on the existing `RealtimeCommandQueue` (`AudioEngine.hpp:74-89`) **but using P05's true-atomic SPSC, not the shared `size_` counter** (the current queue is single-threaded and not safe across the new MIDI→audio thread boundary). Single writer = MIDI thread; single reader = audio callback. Overflow policy = drop-oldest-or-newest (decide below), never block.
- **Change** `libs/audio/src/AudioEngine.cpp::renderGraphBlock` (`:387-547`): after `processRealtimeCommands()` (`:389`), drain the `MidiEventQueue` for events whose timestamp falls in `[transport.samplePosition, transport.samplePosition + frames)`, compute each event's intra-block sample offset, and append it (allocation-free) to the destination instrument node's preallocated `juce::MidiBuffer` at that offset before the node's `processBlock`. Events past the block end stay queued; events before block start (late) are clamped to offset 0 with an under-run flag. Replace the precomputed `noteSequence` sine summation (`:468-496`) as the *live* source while keeping the baked P09 schedule for offline/sequenced playback — the two merge into one `MidiBuffer` per instrument per block.
- **Change** `libs/audio/include/lamusica/audio/AudioGraph.hpp`: add a stable input-routing field to instrument-bearing `GraphNode`s (e.g. `std::int32_t liveMidiInputBus{-1};`) so the snapshot resolves which queued live events feed which node off-thread; the callback only compares integers. Do **not** store device ids or strings read on the callback.
- **MIDI output / clock:** in `MidiDeviceHost`, send queued output messages and transport-derived MIDI clock (24 PPQN) / Start/Stop/Continue via `juce::MidiOutput::sendBlockOfMessages` on a timer or the audio-block boundary — never from inside the audio callback. A `MidiClockMode::Receive` path syncs the transport tempo from incoming clock off-thread through the existing `RealtimeCommand::SetTempo` (`AudioEngine.hpp:62`) command queue.
- **Change/extend persistence for `MidiDeviceConfiguration`** (`libs/session/include/lamusica/session/Midi.hpp:164-169`, helpers already in `libs/session/src/Midi.cpp:34-115`): wire a serializer to the chosen store. Today `ApplicationPreferences` (`ApplicationSession.hpp:262-271`) holds only `enabledMidiInputIds` (a flat id list, in-memory only — it is **never written to disk** anywhere in `libs/session/src`), and has no output routes or clock mode. P16 must persist the full config (inputs + `outputRoutes` + `clock`) by either (a) replacing/augmenting `enabledMidiInputIds` with an embedded `MidiDeviceConfiguration` in preferences and standing up the missing preferences serializer, or (b) splitting machine-scoped device identity into preferences and project-scoped routing (which track → which port/channel) into the project manifest + `project-v1.schema.json`. Decide and document (see Decisions).
- **New** session bridge for recording: feed live events into `RecordedMidiNote` (`Midi.hpp:110-117`) pairs and commit via the existing `commitMidiRecording(MidiClipData&, const MidiRecordingCommit&)` (`Midi.cpp:237`) so captured input lands in a clip with correct transport-relative timing (note-on/note-off sample positions relative to record-start). Reuse `MidiRecordingMode` Overdub/Replace (`Midi.hpp:105-126`).
- **New** tests in `tests/unit/` and `tests/integration/` (P02 framework, registered in `tests/CMakeLists.txt`): (1) a virtual/injected MIDI source → hosted instrument node produces non-silent audio with the note landing at the expected intra-block sample offset; (2) a recorded-MIDI timing test asserting captured note-on/off samples equal the transport positions at which events were injected; (3) an RT-safety test draining the queue inside `renderGraphBlock` under the **P06** allocation/lock trap (TSan-clean); (4) a queue-overflow test (writer outruns reader → bounded, no UB, drop policy honored); (5) a `MidiDeviceConfiguration` round-trip (serialize → deserialize → equal) with the documented scope.

## Decisions To Make
- **Persistence scope: preferences vs. project (correction-relevant).** Device *identity/enablement* and clock mode are machine-scoped (preferences); track→port output *routing* is arguably project-scoped (a song wants its synth track on a specific external port). Decide the split, then either grow `ApplicationPreferences` (and finally build the absent preferences serializer) or add a `midiDevices` block to `ProjectManifest` + `docs/schemas/project-v1.schema.json`. Document which fields live where so device-not-present on another machine degrades gracefully (route ignored, not error).
- **Live MIDI timestamp → sample mapping.** JUCE delivers `MidiMessage::getTimeStamp()` in seconds from the input callback, asynchronously to the audio clock. Decide the conversion: snapshot the audio playhead + a monotonic clock at each callback and interpolate, vs. simply stamping at "current playhead + buffer latency." Affects how tight live timing feels and how recorded timing aligns to transport.
- **Queue overflow policy.** Bounded SPSC must define behavior when the MIDI thread outruns the audio thread (e.g. a dense controller sweep): drop-oldest, drop-newest, or coalesce CC of the same controller. Note-offs must never be dropped silently (stuck-note risk) — decide a protected path or a forced all-notes-off.
- **Routing model.** All enabled inputs → focused/armed track only, vs. per-input channel-to-track routing matrix. v1 likely arms one track and routes all enabled inputs to it; the full matrix is UI work (P22). Define the data so P22 can drive it.
- **MIDI clock obligation.** Decide whether clock send/receive is in-scope for v1 or gated behind a flag — the scope says "where required." At minimum support Internal; Send/Receive (`MidiClockMode`, `Midi.hpp:152-156`) may ship behind config but must round-trip in the serializer.
- **Stuck-note / panic handling.** On stop, device removal, or input disable, decide the all-notes-off / sustain-flush strategy delivered to instrument nodes (a synthesized note-off burst into each instrument's `MidiBuffer`), since live note-offs can be lost on disconnect.

## Compromises And Tradeoffs
- **Accepted:** v1 arms a single track and routes all enabled inputs to it; a full per-input/per-channel routing matrix and its UI are P22/P23. The data model must already support the matrix so the UI is additive.
- **Accepted:** MIDI parsing, MPE expansion, and SysEx handling are coarse in v1 (note/CC/pitchbend/aftertouch/program — the set already modeled in `MidiPlaybackEvent`, `Midi.hpp:87-103`); full MPE/SysEx is a later refinement.
- **Rejected MVP shortcut:** parsing `juce::MidiMessage`, allocating, or touching the `MidiDeviceConfiguration`/strings on the audio callback. All parsing and routing-resolution happen on the MIDI thread / in the off-thread snapshot build; the callback reads only POD events and integer handles.
- **Rejected MVP shortcut:** synthesizing live notes by mutating the precomputed `noteSequence`/`frequencyHz` sine path (`GraphCompiler.cpp:179-189`, `AudioEngine.cpp:468-496`). Live input drives the P12 instrument `MidiBuffer`, not the fixture oscillator.
- **Risks later tasks must revisit:** latency-compensated input monitoring, punch/count-in, and take handling are **P17** — P16 ships raw capture timing only; the recorded-timing offset for monitoring latency must be a documented seam P17 fills. PDC (P14) shifts instrument output and may desync live monitoring. Sample-rate changes (P29) invalidate sample-timestamped queued events; the snapshot must flush the queue on SR change. Automation of incoming CC → parameters is **P18**.

## Acceptance Gates
- `cmake --build --preset debug --target lamusica_daw lamusica_unit_tests && ctest --preset debug --output-on-failure` passes, including the new MIDI tests.
- Virtual-keyboard test: injecting a note-on at a known sample offset and a note-off later produces non-silent audio from a hosted instrument node (assert peak > 0 via `peakAbsoluteSample`, `libs/audio/include/lamusica/audio/Bounce.hpp`), and the note's first non-zero sample is at the expected intra-block offset (sample-accuracy, not block-quantized).
- Recorded-MIDI timing test: events injected at transport positions T0..Tn are captured as `RecordedMidiNote` whose `noteOnSample`/`noteOffSample` (`Midi.hpp:110-117`) equal those positions (within the documented mapping tolerance) and `commitMidiRecording` (`Midi.cpp:237`) lands them in a `MidiClipData` clip.
- RT trap (P06): draining the `MidiEventQueue` and filling `MidiBuffer`s inside `renderGraphBlock` reports zero allocations/locks/file-IO/logging/JSON on the callback thread and is TSan-clean across the MIDI-writer / audio-reader boundary; the callback stays `noexcept` and never throws on an empty/overflowed queue.
- Overflow test: a MIDI writer pushing faster than the reader drains stays bounded, exhibits the chosen drop policy, never corrupts the ring, and never drops a protected note-off (no stuck note after the burst).
- MIDI clock test: in `Send` mode the transport emits 24 PPQN + Start/Stop/Continue at the correct tempo (assert message count/spacing for N beats); in `Receive` mode incoming clock updates `TransportState::tempoBpm` via the command queue, off the callback.
- `MidiDeviceConfiguration` round-trips through the chosen store with the documented scope; a route to an absent device on load is ignored gracefully (no throw), verified by a deserialize-with-missing-device test.

## Notes For Agents
- **Realtime law:** inside the audio callback (P04 `audioDeviceIOCallbackWithContext`) you may only `pop` POD events from the SPSC `MidiEventQueue`, compare integer handles, and write into preallocated `juce::MidiBuffer` scratch owned by the P12 `PluginInstrument` node — no `new`/`malloc`, no mutex, no `juce::MidiMessage` parsing, no string compare, no logging. The current `RealtimeCommandQueue` (`AudioEngine.hpp:74-89`) has a plain `size_` and is **single-thread only**; do not reuse it across threads — build on P05's true SPSC.
- **Thread topology:** JUCE's `MidiInputCallback::handleIncomingMidiMessage` runs on a dedicated high-priority MIDI thread, *not* the audio callback. That is the correct place to parse, time-stamp, and resolve routing → push to the queue. Output/clock send happens on a timer or block boundary, never inside the audio callback.
- **Replace the sine note path:** the precomputed-sine handling P16 supersedes is `AudioEngine.cpp:468-496` (polyphonic `noteSequence` summation) and its source in `GraphCompiler.cpp` — `makeClipNode` sets `GraphNodeKind::Sine` + `starterBassGraphNotes` (`:79`, `:179-189`) using `midiPitchFrequency` (`:66-67`). P09 already moves the baked synth path to `GraphNodeKind::Instrument`; P16 adds the *live* feed into the same instrument node alongside the baked schedule.
- **Persistence reality:** `MidiDeviceConfiguration` (`Midi.hpp:164-169`) and its mutators (`mergeMidiDevice`/`setMidiInputEnabled`/`setMidiOutputRoute`/`setMidiClockOptions`, `Midi.cpp:34-103`) exist but are wired to *nothing* — no serializer references them, and `ApplicationPreferences.enabledMidiInputIds` (`ApplicationSession.hpp:264`) is in-memory only with no JSON writer. Standing up the persistence path is genuinely new work, not a rename.
- **CMake:** the device host needs `juce_audio_devices` (CoreMIDI is in that module) — already added by P04 to the host target's `target_link_libraries`; add the new `MidiDeviceHost`/`MidiEventQueue` sources there and to `tests/CMakeLists.txt`.
- **Maps to existing task docs:** this is the live-MIDI engine substance behind `docs/tasks/010-midi-core.md` (routing/devices/recording/event processing) and the engine precondition for `docs/tasks/030-midi-piano-roll-drums-patterns-ui.md`, whose MIDI device UI and live recording assume exactly this layer exists. No 026-036 task owns realtime live-MIDI ingestion; P16 fills that gap and feeds the recording/mixer UIs (P22) and the drum-machine/step-sequencer/clip-launcher performance surfaces (P23).
