## Decisions To Make
- **Own node kind vs. embed a `juce::AudioProcessorGraph`.** A first-party plugin node kind keeps the P05 snapshot/schedule contract and bounded buffers under our control; an embedded `AudioProcessorGraph` reuses JUCE's routing/PDC but allocates/locks on its own terms and fights our snapshot-swap discipline. Recommend the first-party node kind; revisit only if PDC (P14) becomes intractable.
- **Parameter delivery: SPSC command ring vs. atomic mirror.** A ring preserves edit ordering and coalesces cleanly but is bounded (overflow policy = drop-oldest vs. fail-and-retry off-thread); an atomic-float mirror is allocation-free and never overflows but loses intermediate values (acceptable for knobs, lossy for fast automation â€” which P18 will own). Decide per-edit-source; likely mirror for continuous automation, ring for discrete UI/MCP edits.
- **Bypass semantics at block boundary.** Hard switch (passthrough copy) vs. short equal-power crossfade to avoid clicks on toggling a plugin with tails. Decide the default and whether tail-flush on bypass is in-scope here or deferred to P14 freeze/PDC.
- **Instrument MIDI source.** P12 must render an instrument from *some* MIDI. Decide whether to drive it from the snapshot's baked note schedule (P09 content, deterministic, offline-testable) only and defer live MIDI ingestion entirely to P16, so this task is fully testable offline without a live device.
- **Format priority and licensing.** AudioUnit (`PluginFormat::AudioUnit`) is first-party on macOS and runtime-available (`PluginHostEnvironment::audioUnitRuntimeAvailable`, `Plugin.hpp:82`); VST3 requires the Steinberg SDK and license acceptance (`vst3SdkAvailable`/`vst3LicenseAccepted`, `Plugin.hpp:83-84`). Decide whether P12 hosts AU-only first and gates VST3 behind the P11 license flag. Note JUCE's GPL/commercial split â€” confirm the project's JUCE license posture before shipping VST3 hosting.
- **In-process vs. out-of-process processing.** P11 scans out-of-process (`outOfProcessHostingAvailable`, `Plugin.hpp:85`). Decide whether render-time hosting is in-process for v1 (lower latency, a plugin crash takes the audio thread) with crash isolation deferred to P13's error model, or out-of-process from the start (shared-memory audio bridge, higher latency).

## Compromises And Tradeoffs
- **Accepted:** in-process render hosting for v1; a misbehaving plugin can stall/crash the callback. The graceful plugin error model and editor/state isolation are explicitly P13; PDC and freeze are P14. Document the exposure.
- **Accepted:** instrument MIDI comes from the baked snapshot note schedule (deterministic, offline-testable). Live MIDI input is P16; live automation ramping is P18. P12 ships the delivery *mechanism* and a coarse application path only.
- **Rejected MVP shortcut:** calling `processBlock` directly off the authoring `AudioGraph` or holding `juce::AudioProcessor*`/`std::string` plugin ids on the callback. All live pointers and parameter-index resolution are baked into the P05 snapshot off-thread; the callback touches only integers, raw borrowed pointers, and preallocated buffers.
- **Rejected MVP shortcut:** mutating plugin parameters or chain order directly from the UI/MCP thread. Every change crosses the lock-free mailbox or the snapshot swap.
- **Risks later tasks must revisit:** snapshot must carry/own plugin *state* across swaps so reorder/bypass does not reset a plugin (full opaque state persistence is P13 â€” P12 must at minimum not drop state on swap); PDC latency reporting from `processBlock` is ignored here and must be honored in P14; sample-rate changes require re-`prepareToPlay` (interacts with P29); send/bus channel-strip processing around inserts is P15.

## Acceptance Gates
- `cmake --build --preset <default> && ctest --preset <default>` passes, including the new plugin-render and RT-safety tests.
- Instrument node renders non-silent audio from a baked MIDI note: offline render of a graph with one instrument node has peak > 0 (assert via `peakAbsoluteSample`, `libs/audio/include/lamusica/audio/Bounce.hpp:46`); a silence baseline stays 0.
- An effect insert measurably alters output vs. the same graph with that insert `bypassed=true` (assert a defined per-sample / RMS delta above threshold), and reversing the order of two inserts produces a different rendered buffer.
- RT trap (P06) instrumented run of the plugin-hosting `renderGraphBlock` reports zero allocations/locks/file-IO/logging/JSON on the callback and is TSan-clean; the callback signature is `noexcept` and never throws on a malformed or unresolved-plugin graph (degrades to silence per P05).
- Parameter latency: a `PluginParameterMailbox` push for a known `pluginIdentifier::parameterId` is reflected in the rendered output within the agreed bounded number of blocks; no lock is taken on the callback to apply it (verified under TSan and the trap).
- Mid-playback glitch test: toggling bypass and add/remove/move of an insert while transport is "playing" reaches the graph only via snapshot swap, produces no NaN/Inf and no discontinuity beyond the agreed bypass-crossfade tolerance.
- Determinism: two offline renders of the identical project + plugin chain are bit-identical (or within documented denormal tolerance) given a deterministic instance and baked MIDI.
- `resolveParameterIndex` round-trips every `DiscoveredPluginParameter::automationAddress` from `discoverPluginParameters` (`libs/session/src/Plugin.cpp:354`) back to a valid live parameter index for a hosted instance; an unknown address returns `nullopt`, never an out-of-range index.

## Notes For Agents
- **Realtime law:** the callback may only read the active `GraphSnapshot` (P05 `activeSnapshot_.load(acquire)`), borrowed `juce::AudioProcessor*`, preallocated `AudioBuffer`/`MidiBuffer` scratch, and atomic param/bypass state. No `new`/`malloc`, no mutex, no `std::string` compare, no JSON, no logging. JUCE `processBlock` itself must run with `JUCE_DISABLE_*` denormal handling and FTZ set on the audio thread by P04.
