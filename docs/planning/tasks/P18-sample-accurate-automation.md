# P18 Sample-accurate automation application and ramping in the render path

> **Area:** audio-engine · **Severity:** medium · **Effort:** M · **Phase:** Phase 4 — Mixer, MIDI I/O, recording, automation on the audio path
> **Depends on:** P05, P12 · **Blocks:** P26

## Objective
Make automation audible and artifact-free in the production render path. Today gain/pan/connection gains and plugin parameters are applied as block-constant scalars (`engineNodeOutputGain`/`engineNodeChannelGain` per block in `AudioEngine.cpp:519-543`, `nodeOutputGain`/`nodeChannelGain` per block in `AudioGraph.cpp:289-300`), so any moving lane zippers, and `GraphCompiler.cpp::compileProjectAudioGraph` never reads `ProjectManifest::automation` — automation lanes are simply dropped from rendered output. P18 bakes per-lane automation into the P05 `GraphSnapshot` off-thread and applies it at the correct sample offset with per-sample (or bounded sub-block) ramping for volume, pan, and plugin parameters (via the P12 parameter mailbox), with no allocation/locks/throw on the `noexcept` callback, deterministically across buffer sizes. A volume ramp must produce a measurable amplitude envelope and a plugin-parameter lane must measurably change output.

## Dependencies
- `P05 — Validated audio-graph snapshot, precompiled noexcept render path, and true lock-free SPSC command queue` — supplies the off-thread `buildSnapshot`/`publishSnapshot` spine and the single `noexcept` snapshot-consuming kernel this task extends; per-frame ramp state and baked automation segments live in (or beside) the snapshot.
- `P12 — Plugin graph node + RT-safe parameter/bypass delivery in the render path` — supplies the plugin render node, `resolveParameterIndex(instance, stableAddress)`, and the lock-free `PluginParameterMailbox`/atomic-mirror through which automated plugin-parameter values reach `processBlock`.

## Deliverables
- **New** `libs/audio/include/lamusica/audio/AutomationSchedule.hpp` + `libs/audio/src/AutomationSchedule.cpp`: a POD, snapshot-resident automation record per automated target. It holds, per node and per channel where relevant, a sorted array of breakpoints `{int64 samplePosition, float value, curveToNext}` clamped to the block-render range plus the boundary values, and a precomputed start value + per-sample (or per-segment) slope so the callback evaluates a ramp with integer cursor advance and a single fused multiply-add — no `std::map`/`std::set`/`std::string`, no per-block `evaluateAutomation` call. Targets: node `gain`, node `pan`, `GraphConnection::gain` (send levels), and plugin-parameter normalized values keyed by the resolved P12 parameter index.
- **Change** `libs/session/src/GraphCompiler.cpp::compileProjectAudioGraph` (lines 220-328): sample `manifest.automation` (`ProjectManifest.hpp:138`, lanes of type `AutomationLane`/`AutomationLaneData`, `Automation.hpp:48-56`). For each lane, resolve `targetKind`/`targetId`/`parameterId` (`AutomationTargetKind::{Mixer,Plugin,Instrument,Clip}`) to the graph node + parameter it drives (mixer volume/pan → track/channel node gain/pan at lines 231-263; plugin → P12 `pluginInstanceHandle` + resolved param index; clip → clip node gain). Emit the baked breakpoint arrays into the new schedule structure carried on the snapshot. This is off-thread, may allocate, and reuses `session::evaluateAutomation` interpolation semantics (`Automation.cpp:40-84`: Step holds, Linear interpolates) when precomputing slopes — but does NOT call it per frame on the callback.
- **Change** `libs/audio/src/GraphSnapshot.cpp::buildSnapshot` (created by P05): attach the per-target `AutomationSchedule` arrays to the snapshot POD and initialize each node's ramp cursor to the value at the snapshot's render start sample so the very first block is already on-curve (no first-block jump).
- **Change** the unified snapshot-consuming render kernel — `libs/audio/src/AudioEngine.cpp::renderGraphBlock` (currently `:387-548`, scalar source/connection/output loops at `:463-545`) and the offline path `libs/audio/src/AudioGraph.cpp::renderGraph` (`:205-301`, must remain bit-identical per P05's render-parity gate): replace the block-constant `node.gain`/`node.pan`/`connection.gain` reads with a per-frame value pulled from the lane's ramp cursor at `absoluteSample = transport_.samplePosition + frame` (live) / `startSample + frame` (offline). The cursor advances by `frames` per block; when no lane targets a parameter, fall through to the static value at zero added cost. Plugin-parameter lanes push the per-block (or per-sub-block boundary) value into the P12 mailbox/mirror rather than into the gain math.
- **Decision-gated sub-block stride:** apply volume/pan ramps per-sample inside the existing `frame` loops (cheap, already O(frames·channels)); apply plugin-parameter automation at a bounded sub-block stride (e.g. control-rate every 32–64 samples) because most plugins only read parameters at `processBlock` granularity — see Decisions.
- **New** tests in `tests/unit/` (P02 framework, modeled on the existing graph-compile/render fixtures in `tests/unit/bootstrap_tests.cpp:177-332`): (1) a `0 dB → −12 dB` linear volume lane over N samples yields a strictly descending per-window RMS profile, and removing the lane yields a flat profile (use `peakAbsoluteSample`, `Bounce.hpp:46`, plus a windowed-RMS helper); (2) a fast volume sweep produces no zipper — assert no inter-sample step exceeds the smooth-ramp delta, or a spectral/null check vs. a reference ramp; (3) a plugin-parameter lane (e.g. a gain/cutoff param on a P12 test instance) measurably changes rendered output vs. the same project with the lane removed/`Off`; (4) determinism: identical render at `frames` = 64 vs. 512 vs. 1024 is bit-identical (or within documented denormal tolerance); (5) automation events land at the correct sample offset, not snapped to block boundaries (a breakpoint at sample 100 with `frames=64` takes effect at 100, not 128).

## Decisions To Make
- **Ramp granularity per target.** Per-sample ramping for volume/pan vs. control-rate sub-block stride for plugin parameters. Per-sample for gain/pan is essentially free in the existing loops; plugin params almost always read at block granularity and pushing per-sample through the P12 mailbox would flood it — settle a default sub-block stride (32/64 samples) and whether it is configurable per `EngineConfig`.
- **Plugin-parameter delivery channel.** P12 offers a coalescing SPSC ring vs. an atomic-float mirror (`P12-plugin-graph-node-rt-params.md` "Parameter delivery" decision). Continuous automation loses intermediate values through a mirror but never overflows; a ring preserves ordering but is bounded. Choose mirror-for-automation / ring-for-discrete-edits and define the back-pressure contract when a fast sweep outpaces the stride.
- **Curve types honored at render time.** The manifest only models `AutomationCurve::{Step, Linear}` (`Automation.hpp:15-18`). Decide whether P18 ships Step+Linear only (matching the data model) and explicitly defers exponential/S-curve/Bézier to the UI-curve task (P21/P22), or pre-bakes higher-order curves into dense linear segments at compile time.
- **Pan automation law.** The current pan math is a simple linear taper (`nodeChannelGain`, `AudioGraph.cpp:33-45` / `engineNodeChannelGain`). Decide whether automated pan keeps that law or moves to equal-power; either way the ramp must be evaluated in the chosen domain to stay click-free across the center.
- **`Trim`/`Latch`/`Touch`/`Write` modes vs. render.** `effectiveAutomationValue` already defines `Trim = current + lane`, `Off = passthrough` (`Automation.cpp:118-127`). Decide that P18 renders `Read`/`Trim`/`Off` deterministically and that `Write`/`Touch`/`Latch` capture (live writing) stays a UI/transport concern (P20/P21), not render-path scope.
- **Snapshot cost cap.** Baked breakpoint arrays grow the snapshot POD. Decide a per-lane breakpoint cap (or dense-segment budget) consistent with P05's fixed-capacity philosophy so `buildSnapshot` stays bounded and the swap stays cheap.

## Compromises And Tradeoffs
- **Accepted:** all curve sampling, slope precompute, and target→node resolution happen off-thread in `GraphCompiler`/`buildSnapshot`; the callback only advances an integer cursor and does FMA. We accept a one-block publish latency for re-baking when lanes are edited (consistent with the P05 snapshot-swap model).
- **Accepted:** plugin-parameter automation is delivered at a control-rate sub-block stride, not per-sample, because hosted plugins read parameters at `processBlock` granularity; per-sample param automation is out of scope and would require per-sample `processBlock` calls.
- **Rejected MVP shortcut:** calling `session::evaluateAutomation`/`evaluateAutomationBlock` (`Automation.cpp:40-94`, which allocates a `std::vector` per block at line 88) inside the callback, or snapping automation to block boundaries. Both violate the RT contract / the sample-accuracy requirement and are explicitly forbidden.
- **Rejected MVP shortcut:** applying automation only in the offline `renderGraph` and not the live `renderGraphBlock` (or vice versa) — P05's render-parity gate requires the unified kernel, so both paths must read the same baked schedule.
- **Risks later tasks must revisit:** P28 (time-stretch/warp) changes the sample-position↔source-time mapping, so automation evaluated against absolute samples must be re-derived against warped time; P29 (project sample-rate) must re-bake breakpoint sample positions on SR change; P14 (PDC) shifts node output in time and the automation cursor must align to the post-PDC sample position; P21/P22 curve editing drives the lanes this task consumes (`relation_to_existing`: this is the audio-path half of deferred 016/032's automation). P15's send/channel-strip gains become automatable through the same `GraphConnection::gain` baking.

## Acceptance Gates
- `cmake --build build --target lamusica-tests && ctest --test-dir build --output-on-failure` passes, including the five new automation tests.
- **Descending envelope:** a project with a `0 dB → −12 dB` linear volume lane renders a windowed-RMS profile that decreases monotonically from ~1.0× to ~0.251× (−12 dB) within tolerance; setting the lane `mode = Off` (or removing it) renders a flat profile — proves the lane is actually sampled into output.
- **No zipper:** a fast volume sweep (e.g. full-scale swing over a few ms) shows no per-sample discontinuity larger than the smooth single-sample ramp delta; a spectral/null comparison against an analytic reference ramp shows no broadband step artifact above threshold.
- **Sample-accurate offset:** a breakpoint at sample 100 rendered with `frames = 64` takes effect at sample 100 (mid-block), not at 64 or 128 — asserted by inspecting the exact sample the value changes.
- **Plugin-param lane audible:** a P12 plugin-parameter lane (resolved via `stableParameterAddress`) measurably changes the rendered buffer vs. the same project with the lane `Off`; the value reaches `processBlock` within the agreed bounded sub-block stride / block count.
- **Determinism across buffer sizes:** the same project rendered at `frames` ∈ {64, 256, 512, 1024} is bit-identical (or within documented denormal tolerance), and live `renderGraphBlock` matches offline `renderGraph` (P05 parity).
- **RT contract (P06):** the automated `renderGraphBlock` runs under the P06 allocation/lock/file-IO/logging trap with zero violations and TSan-clean; the callback stays `noexcept` and never throws on a malformed or unresolved automation target (degrades to the static/`defaultValue`, never crashes).

## Notes For Agents
- **Realtime law:** the callback may only read the active `GraphSnapshot` (P05 `activeSnapshot_.load(acquire)`), advance integer ramp cursors, and push plugin-param values into the P12 mailbox/mirror. No `new`/`malloc`, no mutex, no `std::string` compare, no `evaluateAutomation` call, no JSON, no logging on-thread. All curve evaluation/slope precompute is off-thread in `GraphCompiler`/`buildSnapshot`.
- **Code-grounded pointers:** block-constant gain/pan to replace — live `AudioEngine.cpp:519-543` (`engineNodeOutputGain`/`engineNodeChannelGain`/`connection.gain`), offline `AudioGraph.cpp:289-300` (`nodeOutputGain`/`nodeChannelGain`) and the per-frame loops at `AudioEngine.cpp:463-533` / `AudioGraph.cpp:222-300`. Automation is dropped today: `GraphCompiler.cpp::compileProjectAudioGraph` (`:220-328`) never reads `manifest.automation`. Interpolation semantics to reuse off-thread: `Automation.cpp:40-84` (Step/Linear), `effectiveAutomationValue` `:118-127` (Read/Trim/Off). Lane data: `Automation.hpp:36-56` (`AutomationPoint`/`AutomationRegion`/`AutomationLaneData`), manifest field `ProjectManifest.hpp:138`. Plugin param address scheme to consume: `session::stableParameterAddress` (`Plugin.cpp:347`), resolved via P12 `resolveParameterIndex`.
- **Gotcha:** `evaluateAutomationBlock` (`Automation.cpp:86-94`) allocates a `std::vector<float>` per call — never reachable from the callback. Use it only off-thread for slope baking. The first block must start on-curve: initialize the cursor to `evaluateAutomation(lane, renderStartSample)` in `buildSnapshot`, not to `defaultValue`.
- **Gotcha:** mixer mute/solo already zeroes node gain via `channelGain`/`trackMixGain` (`GraphCompiler.cpp:141-157`); automated volume must multiply on top of that mute/solo gain, not bypass it.
