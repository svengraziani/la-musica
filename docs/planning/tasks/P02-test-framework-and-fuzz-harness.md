# P02 Migrate test suite to a real framework with per-case isolation, JUnit output, and a corrupt-project fuzz harness

> **Area:** tests-qa · **Severity:** high · **Effort:** M · **Phase:** Phase 0 — Foundations & Test Infrastructure
> **Depends on:** P01 — Fix on-disk format integrity: schema conformance, lossless/locale-independent floats, JSON escaping, durable atomic save · **Blocks:** P26

## Objective
Replace the monolithic, fail-fast test harness with a real registered-case test framework so that every later subsystem (live audio, plugin hosting, MCP, UI) lands with isolated, parallel, machine-readable, credibly-measured tests. Concretely: each test is a discrete CTest case with its own timeout that reports independently (one failure no longer aborts the run), CI consumes JUnit XML, a libFuzzer/seeded-mutation harness proves the project loader rejects every corrupt input cleanly under ASan, and a coverage build with a per-subsystem baseline gate blocks regressions. This is production test infrastructure, not a smoke suite — it is intentionally scheduled before feature work so the rest of the roadmap inherits it.

## Dependencies
- P01 — Fix on-disk format integrity: schema conformance, lossless/locale-independent floats, JSON escaping, durable atomic save. (Needed so the loader/round-trip tests and fuzz harness assert against the corrected, schema-conformant on-disk format rather than codifying current bugs.)

## Decisions To Make
- **Framework: GoogleTest vs Catch2.** GoogleTest has mature death-tests/`gtest_discover_tests`/JUnit and matches the C++23 toolchain; Catch2 is header-light and single-translation-unit friendly. Pick one and standardize — it becomes the harness for all of P03–P34.
- **Vendoring vs lock-doc change.** `cmake/CheckDependencyLock.cmake` (lines 21-27) bans `FetchContent`/`CPM`/`ExternalProject`/`vcpkg` in build files and requires JUCE pin text in `docs/developer/dependencies.md`. Decide: vendor the framework at a pinned commit under a `LAMUSICA_<FW>_PATH` (consistent with JUCE) and update the lock doc + check to record the new pin, or relax the lock policy. Do not silently bypass the gate.
- **Fuzzing scope and "clean rejection" contract.** Define exactly what "clean" means: `parseProjectManifest` may only throw `std::runtime_error` (its existing failure mode — e.g. lines 84, 114, 138, 248, 303, 323) or return a valid manifest; never crash/abort/hang/UB and never leave partial on-disk state. Add a missing recursion-depth guard to `JsonParser` (see Notes) so depth-bomb inputs are rejected, not stack-overflowed.
- **CI fuzzing model.** Bounded seeded-corpus replay every CI run (deterministic, fast) plus optional longer libFuzzer runs (scheduled/nightly), or libFuzzer in-PR with a wall-clock budget. Decide iteration count and time budget so CI stays green and fast.
- **Coverage tool + baseline policy.** `llvm-cov` (Apple Clang native) for line/region coverage; set initial per-subsystem baselines from the first measured run and choose ratchet semantics (fail on any drop vs drop > N%). Decide which libs are "core" gated (`libs/audio`, `libs/session`, `libs/commands`, `libs/mcp_bridge`).
- **Test isolation contract.** Cases that touch the filesystem (project create/open/save) must use unique per-case temp dirs (`std::filesystem::temp_directory_path()` + test name) so `ctest -j` has no shared-state collisions; decide the helper API and enforce it.

## Compromises And Tradeoffs
- **Accepted:** the large CLI/daw-smoke integration matrix in `tests/CMakeLists.txt` (lines 13-561, with elaborate `PASS_REGULAR_EXPRESSION`/`DEPENDS` chains) is retained verbatim in this task; rewriting those into the framework is out of scope and would balloon effort. They keep `DEPENDS` serialization and are not part of the parallel unit pool.
- **Accepted:** seeded-corpus replay is the always-on CI gate; true coverage-guided libFuzzer may be gated to a nightly/scheduled job to keep PR CI fast and deterministic.
- **Rejected (MVP shortcut):** keeping the `require()/std::exit(1)` harness and merely splitting it into multiple `main()`s. That preserves the core defect — first failure still hides all later results — and was explicitly the gap this task closes.
- **Rejected:** asserting the fuzz harness against current (pre-P01) loader behavior. Must depend on P01 so tests encode the corrected format, not today's float/locale/escaping bugs.
- **Risk later tasks must revisit:** coverage baselines set now will be challenged by P04–P25 introducing hard-to-unit-test realtime/device/plugin/UI code; P06 (RT-safety harness/TSan) and P26 (determinism/RT-deadline suite) extend this infra rather than re-found it, so the framework choice and JUnit/coverage conventions must be stable. The fuzz harness should be extended to the binary sidecar formats (P08) and MCP/JSON-RPC parser (P24) when those land.

## Acceptance Gates
- `cmake --preset debug && cmake --build --preset debug && ctest --preset debug` shows individually-named framework cases (not a single `lamusica_unit_tests`), runnable with `ctest -j8` and filterable (`ctest -R project_manifest`).
- **Isolation proof:** a deliberately-failing assertion in one case does NOT abort the run — `ctest` reports that one case as failed and ALL other cases still execute and report (the explicit defect of the old `std::exit(1)` harness). Verify by temporarily flipping one assertion and confirming the failed/passed counts.
- **Fuzz under ASan:** `cmake --preset asan && cmake --build --preset asan && <fuzz-target> tests/fuzz/corpus` runs thousands of mutated/truncated/binary/duplicate-key/depth-bomb inputs and exits 0 with zero ASan reports, zero crashes/aborts, and no hang (per-input timeout). Each input is asserted valid-or-`std::runtime_error`, never partial state on disk.
- **Depth-bomb regression:** a deeply-nested array/object input that today would recurse `parseValue`→`parseArray`→`parseValue` unbounded is rejected with a thrown error (not a stack overflow) after the depth guard is added.
- **JUnit:** `ctest --output-junit <file>` produces parseable XML uploaded as a CI artifact; CI step consumes it.
- **Coverage gate:** `cmake -DLAMUSICA_ENABLE_COVERAGE=ON ...` + run + `llvm-cov` produces a per-subsystem report; the CI gate FAILS when `libs/audio`/`libs/session`/`libs/commands`/`libs/mcp_bridge` coverage drops below `tests/coverage-baseline.json`.
- **Lock gate stays green:** `cmake -P cmake/CheckDependencyLock.cmake` passes — i.e. the framework was vendored/pinned, not pulled via a banned `FetchContent`/`CPM`/`ExternalProject`/`vcpkg` call.
- `ctest --preset asan` and the existing `lamusica_cli benchmark-smoke` / `validate examples` CI steps (`ci.yml:18-19,37-42`) continue to pass unchanged.

## Notes For Agents
- **The exact defect:** `tests/unit/bootstrap_tests.cpp:50-55` — `require()` calls `std::exit(1)` on the first false condition, so any single failure masks the remaining ~898 assertions. `tests/fixtures/mcp_fixture_client.cpp:11-16` repeats the pattern. Eliminating fail-fast is the central goal; do not preserve any `std::exit` assertion path.
- **Fuzz targets are real:** `parseProjectManifest` is at `libs/session/src/ProjectManifest.cpp:799`; it delegates to `JsonParser{json}.parse()` (line 800) then `asObject` (line 299), `requireUint32` (line 391), `rejectUnknownTopLevelFields` (line 308). `ProjectDocument::open` (`libs/session/src/ProjectDocument.cpp:39-48`) slurps `project.json` and calls `parseProjectManifest`. Fuzz BOTH the in-memory string path and the on-disk path (write mutated bytes to a temp project dir, call `open`).
- **Missing depth guard (confirmed gotcha):** `JsonParser::parseValue` (`ProjectManifest.cpp:111`) recurses into `parseObject` (line 142, `object.emplace(key, parseValue())` at 155) and `parseArray` (line 164, `array.push_back(parseValue())` at 173) with NO recursion-depth limit. A depth-bomb (`[[[[...]]]]`, hundreds of thousands deep) will stack-overflow — that is a crash, which the fuzz gate must catch. Add a bounded depth counter to `JsonParser` so such input throws `std::runtime_error` instead. Coordinate this with P01's loader work to avoid conflicting edits.
- **No allocation/realtime concern here** — this is offline tooling/test code; standard heap/exceptions are fine. The RT constraints apply to the subsystems this infra later validates (P05/P06/P26), not to the harness itself.
- **Determinism:** seed the PRNG mutator from a fixed seed (and allow override via env/arg) so corpus replay is reproducible in CI; log the seed on any failure for triage.
- **Isolation under `-j`:** loader/round-trip cases create real project directories — give each case a unique temp dir; never write into `${PROJECT_SOURCE_DIR}` (the old `WORKING_DIRECTORY` at `tests/CMakeLists.txt:11`) from parallel cases.
- **Mapping to existing docs/tasks 026–036:** this task implements the test-infrastructure prerequisites that `docs/tasks/035-performance-stress-realtime-verification.md` assumed would come AFTER feature tasks 028–034 ("Dependencies: Tasks 028 through 034"). P02 inverts that ordering — it extracts framework/fuzz/coverage so they precede feature work. It cross-cuts 035 but does not replace it; 035's realtime/stress verification is taken up by P06 and P26, which build on the harness P02 establishes here.
