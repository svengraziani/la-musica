# P31 Localization/internationalization scaffold and a Spanish (es) localization

> **Area:** daw-ui · **Severity:** medium · **Effort:** M · **Phase:** UI & Interaction — Application Shell (cross-cutting, lands with P19)
> **Depends on:** P19 · **Blocks:** none

## Objective
Introduce internationalization into the DAW UI *before* P20-P23 hardcode hundreds of English literals, so every user-facing string (menu titles, command names, dialog text, tooltips, error/diagnostic messages) is authored through a translation layer rather than retrofitted. The product is named "LaMusica" (Spanish for "the music"), implying a Spanish-speaking audience, so ship an English base plus a complete Spanish (`es`) translation, a contributor workflow for adding locales, and a test that fails the build on any untranslated/missing key. Display formatting (decimal separators, number grouping) becomes locale-aware while on-disk project serialization stays byte-stable under the C locale — no regression of P01. This is the production i18n spine, not a token string-table demo.

## Dependencies
- P19 — Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding

## Deliverables
- **New i18n module `apps/daw/src/i18n/` (new dir).**
  - `Localization.hpp` + `Localization.cpp`: a thin facade over `juce::LocalisedStrings` that loads the active locale's table at startup and installs it via `juce::LocalisedStrings::setCurrentMappings(...)`, exposing the JUCE `TRANS("...")` macro as the single string-lookup entry point. Resolve the active locale from (1) an explicit app preference, (2) `juce::SystemStats::getDisplayLanguage()` / the system region, falling back to `en`.
  - `StringTables.hpp`: a compile-time-registered list of bundled locales (`en`, `es`) and their `.txt` resource keys, so the missing-key test and the loader share one source of truth.
- **String tables `apps/daw/resources/i18n/en.txt` and `es.txt` (new dir + files).** JUCE `.txt` translation format (`"source" = "translation"` pairs with `language:` / `countries:` headers). `en.txt` is the canonical base (identity mappings, the key inventory); `es.txt` is the full Spanish translation. Embed via `juce_add_binary_data` (or `BinaryData`) and reference from `apps/daw/CMakeLists.txt`.
- **Route the shell through `TRANS` — change every P19 source.** P19 creates `apps/daw/src/ui/{TransportRegion,BrowserRegion,TimelineRegion,InspectorRegion,MixerRegion}.cpp`, `apps/daw/src/MainMenuModel.cpp`, `apps/daw/src/ShellCommands.cpp`, and `apps/daw/src/DocumentController.cpp`. Every literal those files would otherwise emit — the ten menu titles (LaMusica/App, File, Edit, View, Transport, Audio, MIDI, Tools, Window, Help), `getCommandInfo` command names/categories/descriptions, `FileChooser` titles, the dirty-close save prompt, recovery prompt, and error text — must be wrapped in `TRANS(...)`. The current placeholder literals in `apps/daw/src/main_juce.cpp` (`"Transport"`, `"Browser"`, `"Timeline / Piano Roll"`, `"Inspector"`, `"Mixer"`, `"Stopped"`, `"Loop on/off"`, etc. across `main_juce.cpp:25-55,88-134`) are deleted by P19; P31 ensures their P19 replacements never reintroduce bare literals.
- **Locale-aware display formatting helper `apps/daw/src/i18n/NumberFormat.hpp` + `.cpp`.** Wrap the BPM / dB / peak / time-signature display formatting that today is bare `juce::String{status.tempoBpm, 0}` and `juce::String{status.lastMixExportPeak, 2}` (`main_juce.cpp:95-96,130`) in a locale-aware formatter (decimal separator, optional grouping) for DISPLAY only. This formatter must never be on the serialization path: `libs/session/src/ProjectManifest.cpp` float emit points (`bpm` `:663`, clip `gainDb` `:708`, `volumeDb`/`pan` `:735`, automation `:758`/`:774`) stay C-locale `std::to_chars` per P01.
- **Info.plist localizations (file to change, coordinate with P27).** Add `CFBundleLocalizations` (`en`, `es`) to `apps/daw/Info.plist.in` (currently only `CFBundleDevelopmentRegion en` at `Info.plist.in:6`, no localizations array). If localized bundle metadata (app display name, usage strings `NSMicrophoneUsageDescription`/`NSAppleEventsUsageDescription`) is wanted, add `<lang>.lproj/InfoPlist.strings` resources. **Note:** P27 owns wiring the curated plist into the build via `JUCE_CUSTOM_PLIST` (today the JUCE-synthesized plist ships); P31 supplies the `CFBundleLocalizations`/`.lproj` content and a coordination note so P27 packages it.
- **Missing/untranslated-key test `tests/unit/i18n_tests.cpp` (new, registered in `tests/CMakeLists.txt` near the existing `lamusica_daw_app_session_*` targets, lines 19-48).** Parse both string tables; assert (a) every key present in `en.txt` exists in `es.txt`, (b) no `es` value is byte-identical to its English source unless explicitly allow-listed (catches forgotten translations), (c) no `es` value is empty, (d) optionally, a source-scan that greps the shell sources for string literals passed to JUCE display APIs not wrapped in `TRANS` (allow-list known exceptions). The grep-style enforcement is the "no user-facing literal bypasses the translation layer" gate.
- **Contributor workflow doc `docs/developer/localization.md` (new).** Document: the `.txt` table format, how to add a new locale (drop `<lang>.txt`, register it in `StringTables.hpp`, add to `CFBundleLocalizations`, run the test), how `TRANS` keys are named/extracted, and the C-locale-serialization-vs-locale-display invariant. Cross-link from `README.md` and `docs/developer/build-and-test.md`.

## Decisions To Make
- **Locale selection precedence and override:** system display language vs. an explicit in-app preference (stored in `ApplicationPreferences`) vs. an env/launch override for testing. Recommend: preference wins, else system, else `en`; expose a hidden test hook so `i18n_tests` can force a locale without a window.
- **String key strategy:** English-source-as-key (`TRANS("Save As…")`, JUCE-idiomatic, zero indirection) vs. symbolic keys (`TRANS("menu.file.saveAs")`, decoupled from English copy). Recommend English-source keys to match JUCE conventions and keep `en.txt` an identity table, accepting that English copy edits churn the key. Settle before P20-P23 author strings.
- **Translation completeness policy:** hard-fail the build on any missing/untranslated `es` key (strict, blocks merges of new UI until translated) vs. warn-only with a tracked allow-list. Recommend strict for bundled locales, allow-list for in-progress community locales.
- **Scope of "user-facing":** which strings are localized vs. left in English — e.g. internal command *ids* (`edit.undo`) and MCP/JSON-RPC payloads (P24) stay English/stable; only *display* names/categories/tooltips localize. Diagnostic/error strings surfaced in dialogs localize; log lines do not. Draw this line explicitly so it does not drift.
- **Plurals, ordering, and RTL:** JUCE `.txt` tables do not natively handle plural forms or interpolation ordering; decide whether to ship a minimal plural/placeholder convention now (e.g. `{0}`-style with `juce::String::replace`) or defer. RTL is out of scope for `en`/`es` but the layout should not hardcode left-justification assumptions that block it later.
- **Display number/date formatting backend:** `juce`-only locale formatting vs. ICU/`std::format` with a `std::locale`. Recommend a small in-house formatter (decimal separator + grouping) to avoid an ICU dependency, keeping it strictly off the serialization path.

## Compromises And Tradeoffs
- **Accepted:** ship only `en` + `es` at launch (covers_gaps: "no localization/i18n; all strings hardcoded English; no support for the implied Spanish-speaking audience"). Additional locales are a documented, table-only contribution — no code change — which is exactly the acceptance bar.
- **Accepted:** the missing-key test catches *absent* and *suspiciously-identical* translations, not *wrong* ones; semantic translation quality relies on human review of `es.txt`. The grep-for-unwrapped-literals gate is heuristic (allow-listed exceptions) and cannot prove 100% coverage, but it makes a bare literal a CI failure by default.
- **Rejected MVP shortcut:** retrofitting i18n after P20-P23 ship hundreds of English literals (relation_to_existing: P31 lands alongside P19 so subsequent UI authors through the layer). Rejected: a single combined table or a runtime English-only build with "TODO translate" — both defeat the contributor workflow and the audience intent.
- **Rejected:** localizing the on-disk format or MCP payloads. Project files and JSON-RPC stay C-locale / English-keyed (P01, P24) — localizing them would break byte-stability and tool interop.
- **Risk later tasks must revisit:** P20-P23 must author *all* new strings through `TRANS` and add keys to both tables, or the strict test breaks their build (intended). P27 must actually package the `.lproj`/`CFBundleLocalizations` content into the signed bundle (it owns the plist wiring; this task only authors the content). P29 (sample-rate) and P28 (warp) UI strings inherit the same `TRANS` requirement. P30 (accessibility) accessible names should also flow through `TRANS` — coordinate the shared key inventory.

## Acceptance Gates
- `cmake --build --preset debug && ctest --preset debug --output-on-failure` is green including the new `tests/unit/i18n_tests.cpp`; `lamusica_daw` still builds, links, and embeds both string tables.
- Launching the app under `es` (preference or forced system locale) shows Spanish menu titles, command names, dialog text, tooltips, and error messages; launching under `en` shows English. Verify the ten menu titles and the dirty-close save prompt in both.
- The missing-key test fails if a key is removed from `es.txt` or left untranslated: temporarily delete one `es` mapping and confirm `ctest` goes red; restore and confirm green.
- The unwrapped-literal gate fails on a bare display literal: add a `juce::String` display literal not wrapped in `TRANS` to a shell source and confirm the test/CI flags it (with the allow-list mechanism documented).
- **No P01 regression:** serializing a fixed manifest under `LC_ALL=C` and `LC_ALL=es_ES.UTF-8` (and `de_DE.UTF-8`) produces byte-identical output (re-run P01's locale-stability assertion); BPM/dB written to disk use `.`-decimal `std::to_chars` regardless of UI locale, while the *displayed* BPM under `es_ES` shows a `,` decimal separator via the display formatter.
- **Add-a-locale smoke:** following `docs/developer/localization.md`, add a stub third locale (`fr.txt` copied from `en.txt` and translated minimally), register it, and confirm it loads and passes the key test with no code change beyond `StringTables.hpp` registration — proving "adding a new locale requires only a string table + a documented step."
- `apps/daw/Info.plist.in` contains `CFBundleLocalizations` with `en` and `es`; a note in `docs/release/` or the P27 task records that P27 must ship the `.lproj` content in the signed bundle.

## Notes For Agents
- **Realtime safety:** i18n is pure UI/message-thread work. `TRANS`/`LocalisedStrings` lookups, table loading, and display formatting must never run on the audio callback (P04/P05 RT thread) — they allocate and touch JUCE strings. The shell already reads only the published `ApplicationSessionStatus` snapshot (P19 contract, `ApplicationSession.hpp`); P31 only changes how its *display strings* are produced, not the data path.
- **Land with P19, not after:** P19 creates the shell sources (`apps/daw/src/ui/*`, `MainMenuModel`, `ShellCommands`, `DocumentController`) and the ten-menu/command-manager spine (see `docs/planning/tasks/P19-interactive-daw-shell.md`). P31 must wrap those sources' literals as they are written. If P31 lands immediately after P19, do a single sweep converting P19's literals and lock in the strict test so P20-P23 cannot regress.
- **Current baseline to replace:** `apps/daw/src/main_juce.cpp` is today a frozen `StatusPanel` window with all strings hardcoded (`main_juce.cpp:25-55` panel titles; `main_juce.cpp:88-134` `refresh()` builds every label, including bare number formatting `juce::String{status.tempoBpm, 0}` `:95` and `juce::String{status.lastMixExportPeak, 2}` `:130`). P19 deletes this; do not wrap the old code — wrap its P19 replacement.
- **C-locale serialization invariant (P01):** the file writer in `libs/session/src/ProjectManifest.cpp` uses `std::ostringstream`/`std::to_chars` for floats (`:573-789`, float points at `:663,:708,:735,:758,:774`) and P01 pins these to the C locale for byte-stability. Do **not** call `std::locale::global` or `imbue` a user locale process-wide — that would corrupt serialization. Keep all locale-sensitive formatting in `apps/daw/src/i18n/NumberFormat` and only on display values.
- **JUCE APIs:** `juce::LocalisedStrings`, `TRANS`, and `setCurrentMappings` live in `juce_core`/`juce_gui_basics`, pulled in transitively by the already-linked `juce::juce_gui_extra` (`apps/daw/CMakeLists.txt:9`) — no new link dependency. Binary-embed tables with `juce_add_binary_data` and add the generated target to `lamusica_daw` sources in `apps/daw/CMakeLists.txt:7`.
- **Tests run headless (no display):** drive the locale loader and table-consistency checks directly in `i18n_tests.cpp` without constructing a window (mirror the headless `ApplicationSession`-driving pattern P19's `shell_tests.cpp` uses). For the live-menu-string check, assert `TRANS` lookup results against the loaded table rather than instantiating a `MenuBarModel`/peer.
- **Mapping to existing task docs:** no `docs/tasks/026-036` task addresses localization (relation_to_existing). P31 is a new cross-cutting concern that layers onto the P19 shell (which implements `docs/tasks/027-production-daw-shell.md`) and feeds the downstream editor UIs that `docs/tasks/028-033` sketch (now P20-P23) — those must author through this layer. Its `.lproj`/`CFBundleLocalizations` packaging is consumed by P27 (`docs/planning/tasks/P27-release-signing-notarization.md`), which owns the `JUCE_CUSTOM_PLIST` wiring and the `dev.lamusica.app`/`dev.lamusica.daw` bundle-id reconciliation.
