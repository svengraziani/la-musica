# P20 Functional transport bar and UI-to-engine playback/monitoring bridge

> **Area:** daw-ui · **Severity:** blocker · **Effort:** L · **Phase:** Phase 4 — Interactive DAW UI
> **Depends on:** P04, P05, P19 · **Blocks:** P21, P22

## Objective
Turn the placeholder status shell into a real, interactive transport surface wired to a live audio device. Build a production transport bar — play/stop/record-arm/loop/metronome toggles whose visual state mirrors the actual engine `TransportState`, BPM and time-signature editors, dual bars:beats:ticks + absolute-timecode position readouts, and a clickable/draggable position — and stand up the app-level `AudioDeviceManager`/`AudioIODeviceCallback` so pressing Play produces audible output through the selected CoreAudio device and advances a live playhead. Every UI→audio action crosses the thread boundary through the P05 SPSC `RealtimeCommandQueue` and the callback does zero alloc/lock/IO/logging. BPM and time-signature edits update the engine and persist into the project arrangement.

## Dependencies
- `P04 — Live CoreAudio device I/O on a realtime thread driving the engine callback` — supplies the `AudioDeviceManager`/`AudioIODeviceCallback` host and RT-priority thread that P20's transport actually drives; P20 owns the app-level wiring of that host into the JUCE shell.
- `P05 — Validated audio-graph snapshot, precompiled noexcept render path, and true lock-free SPSC command queue` — supplies the genuine lock-free SPSC `RealtimeCommandQueue` and the `noexcept` snapshot-consuming render path; P20 is the producer side that enqueues transport commands and must obey the back-pressure contract P05 defines.
- `P19 — Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding` — supplies the `ApplicationCommandManager`, native menu structure, document lifecycle, and the live status-binding/refresh timer the transport bar plugs into; P20 replaces the `StatusPanel` transport pane with a real component bound to that data flow.

## Decisions To Make
- **Producer-thread topology (P05 invariant).** P05's queue is single-producer/single-consumer by construction; if MCP (P25) ever enqueues transport commands too, that breaks. Decide now that the message thread is the *sole* producer and any MCP transport control is marshalled onto it, so P20 does not silently create a second writer.
- **Back-pressure / overflow policy.** Define what happens when `enqueueCommand` returns `false` (queue full): transport commands must never be silently lost — decide between message-thread retry, coalescing successive Seek/SetTempo, or surfacing a UI error. Coordinate with P05's overflow contract.
- **Position-edit coalescing during drag.** A live drag fires many Seek requests; decide whether the bridge coalesces to the latest pending Seek before enqueue (recommended) vs. flooding the 64-slot ring.
- **BPM/timesig persistence shape.** Decide whether the editable tempo/timesig is a single global arrangement value (today `arrangement.tempoBpm`/`timeSignatureNumerator` per `ApplicationSession.cpp:2201-2202`) or the start of a future tempo/meter map; P20 ships the single-value path but must not bake an assumption that blocks a later map (P21/P23).
- **Timecode/PPQ display policy.** Ticks resolution (engine `pulsesPerQuarter`), absolute-time frame-rate basis (wall-clock `HH:MM:SS.mmm` vs. SMPTE), and rounding for the readout — must match the engine conversions, not reimplement them.
- **Device readiness UX.** What the transport shows when no audio device is available (P04 fail-safe): disabled Play with a clear "no output device" affordance vs. silent no-op. Decide the message-thread reflection of P04's device-error state.

## Compromises And Tradeoffs
- **Accepted:** P20 ships transport, position, and tempo/meter control only — no clip/region editing on the ruler, no automation lanes, no waveform timeline (that is P21), and no mixer/metering surface (P22). The position strip is a single playhead lane, not the arrangement.
- **Accepted:** A single global tempo/time-signature, not a tempo/meter map. Loop region is set via the existing `setLoop`/`setLoopRegion` plumbing; graphical loop-handle editing on the ruler is deferred to P21.
- **Rejected MVP shortcuts:** No "render to WAV then play with a system player" — Play drives the live device callback. No reading/writing `engine_.transport()` from the UI thread to fake state; status flows one way (engine→status→UI) and intent flows the other (UI→queue→callback). No keeping the hardcoded `"Stopped"` string (`main_juce.cpp:95`) — toggle state is bound to `status_.transportPlaying`/`transportRecording`.
- **Risks later tasks must revisit:** the message-thread-as-sole-producer rule must be re-validated when P25 lands the live MCP→DAW bridge; the `SetTimeSignature` command and tempo-persistence path must extend cleanly into P21/P23's tempo/meter map; the position strip's pixel↔sample mapping is the seed of P21's timeline ruler and must be factored so it is reusable rather than transport-private.

## Notes For Agents
- Realtime/safety: the device callback is the consumer; the message thread is the **sole** producer. Build `RealtimeCommand` structs (`AudioEngine.hpp:66-72`) on the message thread and `enqueueCommand` (`AudioEngine.hpp:100`, impl `AudioEngine.cpp:199-201`); the callback calls `processRealtimeCommands` (`AudioEngine.cpp:203-207`) then renders. Never touch `engine_.transport()` from the UI to *write* — read-only via `ApplicationSession::status()` on the P19 refresh tick.
- The engine already provides musical/absolute conversions — `samplesToPpq`/`ppqToSamples` (`AudioEngine.cpp:290-300`), `samplesToBarBeat`/`barBeatToSamples` (`AudioEngine.cpp:302-326`). Reuse them; do not reimplement tick math in the UI. `BarBeatPosition` is `{bar, beat, ppqOffset}` (`AudioEngine.hpp:34-38`).
- Status fields to bind already exist on `ApplicationSessionStatus`: `transportPlaying`, `transportRecording`, `loopEnabled`, `playheadSample`, `tempoBpm`, `timeSignatureNumerator/Denominator`, `renderFrames`, `loopStartSample/loopEndSample` (`ApplicationSession.hpp:44-54`). The current shell already reads many of these (`main_juce.cpp:95-103`) but renders them as static text — replace that with bound widgets.
