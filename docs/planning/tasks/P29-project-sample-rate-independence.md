# P29 Project sample-rate field, asset resampling on import, and SR-mismatch reconciliation

> **Area:** session-model · **Severity:** blocker · **Effort:** M · **Phase:** Phase 8 — Portability & cross-cutting hardening
> **Depends on:** P01, P04, P08 · **Blocks:** none

## Objective
Make LaMusica projects portable across machines and audio interfaces. Today the manifest stores no project sample rate and every position is sample-based, so a project authored at 48 kHz opened on a 44.1 kHz device plays every clip, fade, marker, tempo event, and automation point at the wrong time and pitch. Add a persisted, schema-validated `projectSampleRate`; resample (or record-and-convert) imported audio assets so they play at correct pitch/tempo regardless of their native rate; and implement a documented, deterministic device-vs-project SR reconciliation policy at the CoreAudio device boundary. This is full production behavior, not a single-rate assumption.

## Dependencies
- `P01 — Fix on-disk format integrity: schema conformance, lossless/locale-independent floats, JSON escaping, durable atomic save` — the SR field must serialize losslessly and validate against the bumped schema.
- `P04 — Live CoreAudio device I/O on a realtime thread driving the engine callback` — supplies the live device rate that may differ from the project rate; this task makes that boundary correct instead of assuming `project SR == device SR`.
- `P08 — Versioned binary sidecar format for MIDI, automation, plugin state, and waveform/analysis caches, with migration registry and autosave/recovery` — provides the chained migration registry used to default legacy projects, and the analysis/waveform caches that must record the rate they were computed at.

## Decisions To Make
- **Reconciliation strategy:** drive the engine at the project rate + resample once at the device boundary (cleaner determinism for offline render parity), vs. convert all content to the device rate (lower live latency, but offline≠live unless render also uses device rate). Pick one and document it as the deterministic policy.
- **Import policy:** resample-on-import to a single canonical project rate (simpler render path, immutable assets, but lossy/irreversible and re-resamples if project rate later changes) vs. store source rate + convert at render time (preserves originals, supports later project-rate changes, but every render pays conversion cost). Decide whether the policy is per-project or per-asset.
- **Canonical default + legacy default:** confirm `48000.0` as both the new-project default (matches `EngineConfig`/`AudioDevice` defaults at AudioEngine.hpp:15,23) and the deterministic value assigned to migrated v0/v1 files. Document why 48 k and how a user changes it.
- **Resampler quality / algorithm:** which `StretchQuality` tier (Warp.hpp) maps to SR conversion, and whether SR conversion uses a dedicated polyphase/sinc path distinct from time-stretch. Affects determinism, CPU, and any third-party DSP licensing.
- **Changing project rate post-creation:** is `projectSampleRate` editable after assets are imported? If yes, define re-resample/cache-invalidation flow; if no, lock it at project creation and document.
- **Whole-number vs fractional device rates:** WAV header stores `std::uint32_t` (WavFile.cpp:118,195). Decide policy for fractional/unusual device rates and pull-up/pull-down (e.g. 47.952 k) — at minimum reject or round deterministically.

## Compromises And Tradeoffs
- **Accepted:** an initial single canonical project rate (e.g. 48 k) with the device-boundary resampler is acceptable for first production; per-project arbitrary rates can land later without a schema break (the field already carries the value).
- **Accepted:** resample-on-import is lossy and irreversible; documented as a tradeoff. If store-source-rate is chosen instead, accept a per-render conversion cost.
- **Rejected MVP shortcut:** keeping the current `config_.sampleRate = device_.sampleRate` assumption (AudioEngine.cpp:192) and the GraphCompiler dropping `wav.sampleRate` (GraphCompiler.cpp:206–211). That is the precise defect causing cross-device pitch/time drift and must not survive.
- **Rejected:** silently truncating fractional device rates into the `uint32_t` WAV header without a documented rounding/rejection policy.
- **Risks later tasks must revisit:** (P28) warp/stretch and SR conversion must compose without double-resampling artifacts; (P10) WAV round-trip must remain integer-exact for the header rate; (P17/P21) recording and clip-edit UIs must surface the project rate and warn on mismatched-rate imports; (P08) cache version/keys must invalidate analysis & waveform caches when the rate they were computed at changes.

## Acceptance Gates
- Build + test green: `cmake --build build --target lamusica_tests && ctest --test-dir build --output-on-failure`.
- **Persistence/migration:** a v2 project with `projectSampleRate` saves, reloads, and validates against the bumped schema; a legacy v0/v1 fixture migrates deterministically to `48000.0` and round-trips byte-stably (re-save is idempotent). `parseProjectManifest` rejects `projectSampleRate <= 0` / non-finite.
- **Cross-device timing/pitch:** a fixture project authored at 48 k, rendered/played as if on a 44.1 k device, produces output whose total duration in seconds and dominant pitch match the 48 k reference within tolerance — no progressive drift across a long clip (verify with a sustained sine: frequency and clip end-time match, not just the first block).
- **Mixed-rate import:** importing a 44.1 k WAV into a 48 k project yields a clip that plays at correct pitch and tempo; assert `AssetRecord.sourceSampleRate == 44100` and that the rendered length matches the project-rate expectation.
- **Header round-trip (P10 tie):** `writePcm16Wav` → `readPcm16Wav` preserves the integer header rate exactly (WavFile.cpp:118 ↔ 195) for 44100, 48000, 96000.
- **Cache invalidation:** changing project rate (or importing at a different rate) invalidates analysis/waveform caches computed at the prior rate (P08 cache version), and recomputed caches record the correct rate.
- **Determinism / RT:** offline render of the same project at a fixed device rate is bit-reproducible across runs; the device-boundary resampler performs no allocation/locks/logging on the audio callback (verify under P06's RT-safety harness / TSan job).

## Notes For Agents
- **Realtime constraints:** any resampler invoked on the audio thread (device-boundary or render-time Sample-node conversion) must be preallocated, `noexcept`, lock-free, and allocation-free — author it inside the validated snapshot/SPSC spine from P05, not as a callback-time allocation. Import-time resampling runs off the audio thread and may allocate.
- **Exact defect to fix:** `GraphCompiler.cpp:206–211` copies `wav.audio.{channels,frames,interleavedSamples}` into the Sample node and throws away `wav.sampleRate`; `AudioEngine.cpp:192` (`config_.sampleRate = device_.sampleRate`) collapses project and device rate. These two lines are the root cause of cross-device pitch/time drift.
- **Field locations:** add the SR field to `ProjectManifest` (ProjectManifest.hpp ends at line 140); `currentProjectSchemaVersion` to bump is at ProjectManifest.hpp:13; migration ladder is ProjectManifest.cpp:947–964; serialize near line 654; parse near line 815; validate at line 966. Schema const to bump is project-v1.schema.json:23.
- **Reuse, don't reinvent:** `WavAudioData` already exposes `sampleRate` (WavFile.hpp), `AssetAnalysis` already has `sampleRate` (Assets.hpp:36), and P28's `Warp.hpp` provides the stretch/pitch/resample DSP (`pitchShiftRatio`, `stretchRatio`, `StretchQuality`). P29 plumbs and decides policy; it must not duplicate DSP kernels.
- **Migration via P08:** register the v1→v2 step in P08's chained migration registry; do not extend the ad-hoc `if (schemaVersion == 0)` branch. Migrating must remain deterministic and idempotent for VCS-friendliness (mandate from tasks 005/036).
- **Mapping to existing `docs/tasks/026–036`:** no task in 026–036 owns project sample rate — this hardens the deterministic VCS-friendly format mandate (005/036) for portability and complements the device-rate handling assumed by the production-shell/transport tasks (027) and the browser/import/record/export UX (033). It is orthogonal to the UI tasks (028–032). P29 supersedes none of them but is a prerequisite for correct cross-machine playback before any of those ship.
