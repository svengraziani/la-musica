# P22 Mixer, plugin-management, and recording UIs

> **Area:** daw-ui · **Severity:** blocker · **Effort:** XL · **Phase:** Phase 3 — Production UI front-ends on the real engine
> **Depends on:** P11, P13, P15, P16, P17, P20 · **Blocks:** P23, P30, P34

## Objective
Build the production mixing, plugin-hosting, and recording front-ends on top of the now-real engine: a full mixer with one command-backed channel strip per track/bus (dB fader, pan, mute/solo/record-arm, input-monitor, phase invert, inserts, sends, fader groups, routing matrix, and live peak/RMS/LUFS meters), a plugin chooser/insert-rack/editor-window manager driven by the real scan cache, and recording controls that capture live input into nondestructive takes at the playhead. Every user gesture is authored as a command (undoable, persisted) and mirrored to the audio graph via validated snapshots; meters are read on a UI timer from the P15 lock-free taps. This replaces the hardcoded `StatusPanel` placeholder in `apps/daw/src/main_juce.cpp` with three interactive surfaces — no canned strings, no fixed-length recordings, no offline-only metering.

## Dependencies
- P11 — JUCE plugin host module + crash-isolated out-of-process scanner with persistent cache
- P13 — Plugin opaque state persistence, editor windows, and graceful plugin error model
- P15 — Real bus/send/channel-strip processing and live metering taps
- P16 — Live MIDI input/output ingestion and routing to instrument nodes
- P17 — Live latency-compensated recording with input monitoring, punch/count-in, and take handling
- P20 — Functional transport bar and UI-to-engine playback/monitoring bridge

## Decisions To Make
- **Plugin format priority and licensing:** AUv3/AudioUnit first (no SDK license friction; `PluginHostEnvironment::vst3LicenseAccepted` is `false` by default, `Plugin.hpp:80-86`) versus enabling VST3. Decide whether the chooser shows VST3 entries as unavailable-with-reason (`PluginFormatSupport.reason`, `Plugin.hpp:88-92`) and whether bundling any GPL/AGPL plugin conflicts with LaMusica's license — keep third-party hosting opt-in, ship nothing copyleft in the app bundle.
- **Editor-window threading and process model:** in-process `AudioProcessorEditor` vs out-of-process (`outOfProcessHostingAvailable`, `Plugin.hpp:85`). Decide whether editor windows are modeless top-level `DocumentWindow`s or docked, and how `pinned`/geometry persistence (`PluginEditorWindow.pinned/x/y/width/height`) round-trips across sessions.
- **Meter ballistics and LUFS scope:** refresh rate, peak-hold/clip-hold decay, RMS window, and whether LUFS is momentary/short-term/integrated. Settle the UI-timer cadence vs. tap ring-buffer depth so no reading is dropped or torn under load.
- **Recording length model:** confirm the live path is stop/punch-bounded with no fixed cap. Decide the take naming/numbering scheme (current impl auto-numbers `recorded-take-N`, `ApplicationSession.cpp:1493-1497`) and whether comp candidates are stored as muted clips/`TakeLane` (`Recording.hpp:85-88`) or a new comp model — coordinate with P33 (comp-segment rendering).
- **Routing-matrix UX for illegal edges:** disable-and-explain vs. allow-then-validate-then-reject. Decide how feedback (`wouldCreateFeedback`) and sidechain routes (`SidechainRoute`, `Mixer.hpp:52-57`) are presented, and whether sends are pre/post-fader togglable inline (`Send.preFader`, `Mixer.hpp:29`).
- **Command granularity & coalescing:** whether continuous fader/pan drags coalesce into one undo entry (drag-begin/drag-end) per the P03 history, and which mixer ops map to which command names.

## Compromises And Tradeoffs
- **Accepted:** first iteration may render group/return/master strips with a reduced control set, and the routing matrix may cap visible channels with virtualization rather than rendering thousands of cells. LUFS may ship momentary-only initially.
- **Accepted:** meters read a snapshot of the latest tap value on a timer; sub-frame meter accuracy is intentionally not a goal (visual feedback, not measurement-grade unless LUFS gating is later certified).
- **Rejected MVP shortcuts:** no hardcoded strip strings (removes `main_juce.cpp:126-133`); no fixed-length capture (removes the canned-frames recording path); no synchronous/blocking plugin scan on the message thread; no metering computed on the audio callback; no direct manifest mutation from UI — every edit goes through a command and a validated snapshot.
- **Risks later tasks must revisit:** comp/take selection rendering is deferred to P33 (this task only persists candidates and audition/mute); sample-accurate automation of plugin params is P18 (this task exposes param controls but not lane editing — that is P21/P23); accessibility of all new controls is P30; localization of new strings is P31. PDC/freeze interactions (P14) must be re-checked once inserts add latency — the mixer must not assume zero-latency inserts.

## Acceptance Gates
- Build and full suite green: `cmake --build build --target lamusica_daw lamusica_unit_tests && ctest --test-dir build --output-on-failure`.
- **Mixer is real:** launching `lamusica_daw` shows one strip per project track/bus generated from the real `MixerState`/`firstTrackTrackMix()` — assert no occurrence of the literal `"Generated Drums -> Master"` in the UI output. Moving a fader/pan, toggling mute/solo: (a) changes the rendered mix (verify via `auditionCurrentMixBlock` / a bounce diff), (b) persists to disk and survives reopen, (c) undoes/redoes via the command history.
- **Meters animate from the audio thread:** with playback running, meter components update from the P15 lock-free tap on the UI timer; a test asserts the drain path performs no heap allocation and reflects a known-amplitude signal (`measureInterleaved` reference, `libs/session/src/Mixer.cpp:603`). With realtime device I/O, a TSan run (P06) over the tap read/write shows no data race.
- **Plugin workflows:** the chooser lists scanned plugins asynchronously without blocking the message thread; insert/remove/reorder/bypass are undoable and audibly affect playback; opening then closing N editor windows leaves no leaked editors (instance count returns to baseline); insert-chain state persists across reopen; a blacklisted plugin appears in the recovery panel with its reason and can be re-allowed via `allowPluginRescan` and rescanned.
- **Recording:** arming a track and recording captures live input into a take at the playhead whose length equals the stop point or the punch range (NOT a fixed 48000-frame block); count-in offsets the capture by `countInSamples`; input monitoring toggles audibly; recorded takes appear in the take list, can be auditioned and muted, and persist nondestructively. A test drives `prepareFirstTrackRecording(...)` and asserts `recordFrames`/`timelineStartSample`/`punchInSample`/`punchOutSample`/`prerollStartSample` match the planned count-in+punch (`ApplicationSession.cpp:1444-1475`).
- **Determinism:** identical user gesture sequences produce identical persisted manifests and identical offline bounces (byte-for-byte), confirming command-authored + snapshot-mirrored edits.

## Notes For Agents
- **Realtime/safety constraints:** the audio callback (P05 render path, `AudioEngine::renderGraphBlock`, `libs/audio/src/AudioEngine.cpp:387`) must never see UI work. Mixer/plugin/recording gestures cross to audio only as commands through the `RealtimeCommandQueue` (`libs/audio/include/lamusica/audio/AudioEngine.hpp:74-89`) or a P05 graph-snapshot swap — never by mutating shared state under a lock. Meters flow the other way through P15's lock-free taps, drained on a `juce::Timer`; do no allocation/logging/JSON in that drain.
- **Command-backed only:** UI must not call `document_->mutableManifest()` directly. Route every edit through `ApplicationSession` command methods (the pattern at `recordFirstTrackTake` `ApplicationSession.cpp:1479` with `rememberUndoSnapshot()` `:1522` and `document_->save()` `:1539`). Continuous controls should begin/end an undo transaction so a drag is one entry (P03 coalescing).
- **Generation, not literals:** strips come from the real project. Replace `main_juce.cpp:126-133` mixer text and `:119-124` inspector text; drive `MixerView` from `firstTrackTrackMix()` / `MixerState::channels`. The placeholder window structure (`MainComponent`, `StatusPanel`, `main_juce.cpp:25-143`) is what you are dismantling.
- **Plugin scan is async:** the chooser reads the persistent `PluginScanCache` and must not run the out-of-process scanner on the message thread; show progress and merge results via `mergeScanResult` (`Plugin.hpp:142`). Honor `PluginScanPolicy` blacklist-on-crash/timeout (`Plugin.hpp:63-67`).
- **Editor lifetime:** P13 owns the `AudioProcessorEditor`; the UI window is a managed wrapper. Tie window close to `closePluginEditor` (`Plugin.hpp:188`) and ensure teardown on insert removal/track close/app quit. Persist via `serializePluginEditorState`/`parsePluginEditorState` (`Plugin.hpp:190-191`).
- **Recording length:** the current fixed-`frames` API (`ApplicationSession.cpp:1420-1424`, `1465-1471`) is the MVP shortcut to remove. With live capture (P17 `RecordingSession::appendInterleaved`, `libs/audio/include/lamusica/audio/Recording.hpp:110`) length is determined by stop/punch. Reuse `makeRecordingPlan`/`RecordingPlan` (`Recording.hpp:61-93`) and `measureRecordingLatency` (`Recording.hpp:90`, impl `Recording.cpp:55`) for latency-compensated capture; persist takes via `TakeLane`/`addTake`/`activateTake` (`Recording.hpp:85-98`).
- **Routing validation:** always run `validateRouting`/`hasRoutingCycle` and consult `RoutingMatrixCell` flags before issuing a route command; the GraphCompiler consumes `MixerState` (`compileProjectAudioGraph`, `GraphCompiler.hpp:25`) and `prepareMixerGraphUpdate` returns a `validationError`/`ready` flag (`GraphCompiler.hpp:19-23`) — surface that to the user, do not silently drop edits.
- **Maps to existing tasks docs:** this implements **docs/tasks/031** (plugin hosting UI), **docs/tasks/032** (mixer/routing/automation UI — mixer+routing+metering portion; automation-lane editing stays with P21), and the recording-UX portion of **docs/tasks/033** (browser/import/record/export UX — import/browser/export portions belong to P21/P23). The engine substrate it sits on is delivered by P11–P17; this is the UI layer only.
