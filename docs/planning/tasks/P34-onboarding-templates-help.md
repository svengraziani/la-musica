# P34 In-app onboarding, project templates/chooser, and contextual help

> **Area:** daw-ui · **Severity:** low · **Effort:** M · **Phase:** Phase 7 — Verification, performance, and release (UX completeness alongside accessibility/localization)
> **Depends on:** P19, P21, P22, P23 · **Blocks:** none

## Objective
Give a from-scratch user a real first-run experience inside the app, not just the external docs P27 ships: a welcome/new-project flow with selectable starter templates (empty, basic multitrack, drum+synth, podcast/voice) authored entirely through the generic P07 authoring API, a project chooser/recents surface on launch, a Help menu that reaches the bundled user manual, contextual tooltips, and an optional skippable guided first-session tour. Every template must produce a valid, renderable, independently-saveable bundle, and all onboarding content must be localizable (P31) and accessible (P30). This is the in-app onboarding surface no other task owns — the product-grade replacement for "create a hardwired FirstTrack starter and freeze a status panel."

## Dependencies
- P19 — Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding
- P21 — Arrangement timeline with clip/automation editing, plus high-level audio clip editing UI
- P22 — Mixer, plugin-management, and recording UIs
- P23 — Drum machine, step sequencer, clip-launcher, browser/inspector, and export UIs

## Decisions To Make
- Template set and scope: lock the four templates above vs. adding more (mastering, beat-making); decide whether `drum-synth`/`podcast-voice` ship bundled instrument/plugin references (which would couple P34 to P11-P13 plugin availability and to bundled-plugin licensing) or stay plugin-free using only built-in instrument nodes (P09) to keep P34 dependency-light and license-clean.
- Bundled sample/media policy: whether any template ships audio assets (raises AGPL/CC-BY/licensing and bundle-size questions, ties to P27 SBOM/provenance) or all templates are media-free; default to media-free.
- Launch policy: always show the welcome window vs. "reopen last project" preference vs. respecting a system "reopen documents on quit" path; reconcile with P19's New = no-temp-file guarantee and crash-recovery prompt ordering (recovery vs. chooser precedence).
- Tour storage & re-trigger: which `ApplicationPreferences` field records "tour seen", and whether Help > Restart Guided Tour resets it; coordinate the preference field name with P19's preferences surface.
- Help content delivery: render `docs/user-manual.md` in-app (bundle + lightweight markdown view) vs. open the external/online manual via `juce::URL`; decide for offline-first since the manual is a repo doc.
- Localization completeness gate: whether a missing `es` onboarding key is a hard test failure (consistent with P31's untranslated-key test) or a soft fallback to base English.

## Compromises And Tradeoffs
- Accepted: templates are authored at runtime via the P07 API rather than shipped as pre-serialized `.lamusica` bundles — slower to instantiate but guarantees they always conform to the current schema/validation and never bit-rot against `ProjectManifest` changes. Rejected the shortcut of checking in frozen sample bundles.
- Accepted: media-free, plugin-light templates so P34 does not gate on plugin availability or drag in bundled-plugin licensing; richer "starter content" is a later, owned task.
- Rejected MVP shortcut: keeping the hardwired single canned project (`createFirstTrackProject`) as the launch experience, or leaving the leaking `uniqueUntitledProjectPath()` temp bundle (`main_juce.cpp:12-23`). P34 closes the "no in-app onboarding/templates/chooser for a from-scratch user" gap directly.
- Risk to revisit: the `FirstTrack*` API and `StarterProject.hpp` helpers remain in `libs/session` after P19 demotes them; P34 must not reintroduce them in the product path, and a later cleanup task should delete the demoted surface once nothing references it. Template builders depend on P09's instrument node and P07 id-stability/validation invariants being load-bearing — schema changes upstream require re-validating every template.

## Notes For Agents
- Realtime/safety: this is entirely non-realtime UI/control-thread code. Template builders mutate the session through the P07 command path on the message thread; nothing here touches the audio callback. The only RT contract is the same as P07's — every template must yield a manifest `validateProjectManifest` accepts so P05/P09 can build a validated snapshot from it; never instantiate a template that fails validation.
- Author templates strictly through the generic API: `createProject` (`ApplicationSession.cpp:594`) then the P07 methods (`addTrack`/`createAudioClip`/`createMidiClip`/`addPlugin`/`addSend`/`addAutomationLane`/`addMarker`/`setTempo`/`setTimeSignature`). Use `TrackType` from `ProjectManifest.hpp:38-45`. Do not call any `FirstTrack`/`makeFirstTrack*` symbol (`StarterProject.hpp:49-58`); those are the demoted bootstrap P19 removes from the product menu.
- Recents come from `ApplicationSession::recentProjects()` (`ApplicationSession.cpp:582`), maintained by `rememberRecentProject` (capped at 10, `ApplicationSession.cpp:2220-2226`). The chooser is a *view* over this; do not duplicate the recents store.
- Headless test pattern: drive everything through `ApplicationSession` without constructing a window/peer, following the existing `bootstrap_tests.cpp` style (command/route assertions at `bootstrap_tests.cpp:2542-2586`, `2891+`) that P19 reuses for `tests/unit/shell_tests.cpp`. Avoid `juce::Component` construction that needs a display in CI.
- JUCE wiring (`juce::TooltipWindow`, `juce::URL`, dialog windows, accessibility) lives in `juce_gui_basics`, already linked transitively via `juce::juce_gui_extra` (`apps/daw/CMakeLists.txt:9`); no new link deps. `Info.plist.in` already declares `LSMinimumSystemVersion 14.0` — no plist change for the chooser; `CFBundleLocalizations` is P31's concern.
- Mapping to existing `docs/tasks/026-036`: extends `027-production-daw-shell.md` ("Replace Bootstrap Cocoa Shell With DAW Shell") and `036-release-hardening-signing-docs-examples.md` with an *in-app* onboarding surface neither owns — those docs cover the shell and external release docs/examples, but no template chooser, welcome flow, or contextual help. P34 adds that surface on top of the P19 shell and the P21-P23 feature UIs, using the P07 generic authoring API instead of the FirstTrack bootstrap. It does not replace 026 (JUCE pinning) or the engine/MCP tasks.
- Gotcha: P19 already deletes the on-launch temp-bundle leak; P34's job is to ensure the launch lands on the chooser rather than auto-creating any project, and that "New" (P19) and "template" both route through `createProject`/the P07 API with no file written until Save-As (P19's New-no-temp-file guarantee).
