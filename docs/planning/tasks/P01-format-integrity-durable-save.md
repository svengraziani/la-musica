# P01 Fix on-disk format integrity: schema conformance, lossless/locale-independent floats, JSON escaping, durable atomic save

> **Area:** session-model · **Severity:** blocker · **Effort:** M · **Phase:** Phase 0 — Foundation hardening (format & test bedrock)
> **Depends on:** none · **Blocks:** P02, P03, P07, P08, P29

## Objective
Make the existing project serializer trustworthy as the permanent on-disk contract before the model is expanded. Every manifest LaMusica writes today fails its own published JSON Schema (the writer emits fields the schema's `additionalProperties:false` forbids), floats are written with locale-sensitive lossy `ostringstream <<` formatting, JSON string escaping silently drops control characters, and `save()` is not crash-durable. This task delivers a schema-conformant, byte-deterministic, locale-independent, lossless, and durably-atomic project format — the bedrock the versioned sidecar format (P08), generic session API (P07), command history (P03), and SR field (P29) all build on.

## Dependencies
- None (foundation task).

## Deliverables
- **`docs/schemas/project-v1.schema.json` reconciled with the serializer** (file to change). The schema root (lines 7-21, `required`; lines 22-45, `properties`; line 6, `additionalProperties:false`) currently omits `loopEnabled`/`loopStartSample`/`loopEndSample`, which the writer always emits at `libs/session/src/ProjectManifest.cpp:656-658`. The `midiClip` `$def` (lines 132-140) forbids `transposeSemitones`, which the writer emits at `ProjectManifest.cpp:719`. Decide and encode the single source of truth: either add these properties to the schema or stop emitting them. Also reconcile `trackMix`: it is a schema *property* (line 41) but absent from `required` (lines 7-21), while the serializer *always* emits it (`ProjectManifest.cpp:732`) — make presence consistent (add to `required`, or document it as optional and keep the parser's `optionalArray` path at `ProjectManifest.cpp:887`).
- **`std::to_chars` shortest round-trip float writer** (new helper, file to change). Replace `ostringstream <<` of `double`/`float` for `bpm` (`ProjectManifest.cpp:663`), `gainDb` (`:708`), `volumeDb`/`pan` (`:735`), `defaultValue` (`:758`), automation point `value` (`:774`), and the duplicate emitters in `libs/session/src/Plugin.cpp` (send `gainDb` and parameter values around `:154`,`:174`) and `libs/session/src/Mixer.cpp` (send `gainDb` at `:649`). Use `std::to_chars(buf, buf+n, x)` (already `#include <charconv>` at `ProjectManifest.cpp:5`) under the C locale; emit a JSON-legal token (ensure a `.0` or exponent so integers-as-doubles are unambiguous, and reject `NaN`/`Inf` at write time per RFC 8259).
- **RFC 8259-complete `escapeJson`** (files to change). The three duplicate two-character escapers — `ProjectManifest.cpp:468`, `Plugin.cpp:12`, `Mixer.cpp:73` — escape only `"` and `\`. Extend to emit `\b \f \n \r \t` and `\u00XX` for all remaining control chars `< 0x20`. Consolidate into one shared helper (e.g. `libs/session/include/lamusica/session/JsonWrite.hpp` + `libs/session/src/JsonWrite.cpp`) and have all three call sites use it.
- **Escape strings on write in commands** (file to change). `libs/commands/src/Command.cpp` `serialize()` writes raw values straight into JSON (e.g. `AddTrackCommand::serialize` at `:266-269` interpolates `track_.id`/`track_.name`/`metadata_.name` with no escaping; there is currently zero escaping in the file — its only `escape` tokens are the parser's `escaped` flag at `:72-81`/`:166-173`). Route every serialized string field through the shared `escapeJson`.
- **Durable atomic save** (file to change). `libs/session/src/ProjectDocument.cpp` `save()` (`:50-78`) writes a `.tmp` via `std::ofstream`, flushes, then `std::filesystem::rename`. Add: `fsync` (use `fcntl(fd, F_FULLFSYNC)` on macOS, fall back to `fsync` elsewhere) on the temp file's fd *before* `rename`, and `fsync` the parent directory fd *after* `rename`. This requires obtaining a real fd (write via POSIX `open`/`write` or `ofstream` then `open` the path for fsync); keep the existing temp-then-rename atomicity and the cleanup-on-rename-failure path (`:74-76`).
- **CI schema-validation test** (new). A test that serializes empty/starter/populated manifests and validates the output against the published `project-v1.schema.json` with a real JSON-Schema 2020-12 validator, plus a negative test feeding an undeclared top-level field that must be rejected by *both* the validator and `parseProjectManifest` (`rejectUnknownTopLevelFields`, `ProjectManifest.cpp:308-326`).
- **Round-trip / determinism / fault-injection tests** (new, in `tests/unit/` until P02 lands a framework): non-6-sig-digit floats restore bitwise; names with `\n`/`\t`/`0x01` round-trip exactly; byte-identical output under `C` and `de_DE.UTF-8`; an fsync fault-injection/interposition test proving a failed write leaves the prior manifest intact.

## Decisions To Make
- **Single source of truth for `loop*`/`transposeSemitones`/`trackMix`:** add the loop fields and `transposeSemitones` to the schema (preserving today's files and the parser's tolerant reads at `ProjectManifest.cpp:815-817`,`:877`), or strip them from the writer. Recommended: extend the schema (writer + parser already model them; removing them is data loss). Decide `trackMix` required-vs-optional and align serializer/schema/parser.
- **Sample-rate gap:** the manifest has no stored project sample rate (verified: no `sampleRate` in `ProjectManifest.hpp`), yet all positions are in samples. Do not add the field here; reserve the schema slot and additive-migration path so **P29** can introduce it without a breaking bump. Decide now whether to bump `schemaVersion` (currently `1`, `ProjectManifest.hpp:13`) when P29 lands or keep `v1` additive.
- **Float token policy:** shortest round-trip via `std::to_chars` (`std::chars_format::general`) vs fixed precision. Decide the exact canonical form for whole-valued doubles (e.g. always append `.0`) so output is byte-stable and JSON-legal, and how to handle the existing `requireDouble`/`requireInt64` distinction at `ProjectManifest.cpp:378-405`.
- **fsync portability and cost:** `F_FULLFSYNC` is slow but the only true barrier on macOS/APFS. Decide whether to gate it behind a "durable save" path used for explicit Save (vs autosave in P08), and how to fsync a directory portably (POSIX `open(O_RDONLY)` + `fsync`; Windows has no equivalent — document the per-OS contract).
- **Validator dependency & licensing:** pick the JSON-Schema 2020-12 validator for CI (e.g. a vendored header-only C++ validator, or a Python `jsonschema`/`check-jsonschema` step). Confirm its license is compatible with the project and whether it ships in the repo, as a dev-only CI tool, or both.
- **Shared JSON-write module placement:** new `JsonWrite` helper in `libs/session` (consumed by `Plugin.cpp`/`Mixer.cpp`/`ProjectManifest.cpp`) and re-exported to `libs/commands`, vs duplicating. Decide the module boundary so P03/P07/P08/P24 reuse one escaper and one float writer.

## Compromises And Tradeoffs
- **Accepted:** keeping the hand-rolled recursive-descent parser/serializer (`JsonParser`, `ProjectManifest.cpp:97-297`) rather than adopting a JSON library now — minimizes dependency churn for a blocker fix. Risk to revisit: P24 (standards-compliant MCP/JSON-RPC) may want one shared, fully RFC-conformant JSON codec; this task only hardens the write side, not full read-side validation.
- **Accepted:** validating output against the schema in CI (external validator) while the in-process parser remains a tolerant subset reader. The parser already rejects unknown top-level keys (`:308-326`) but is not a full schema validator; full structural validation on read is deferred (P02 fuzz harness + P08 migration registry).
- **Rejected MVP shortcut:** "good enough" `ostringstream` floats and `"`/`\`-only escaping. These are silent corruption sources for VCS-friendly diffs and any non-ASCII/locale environment, and were the explicit covers-gaps for session-model/commands/plugin-hosting.
- **Rejected MVP shortcut:** plain `rename` without fsync. On a crash/power-loss the rename can land with empty file contents on APFS/ext4; the durability gate is non-negotiable for a DAW that owns user work.
- **Risk later tasks must revisit:** the canonical-float form and escape rules become a *format compatibility contract*. P08's migration registry and P02's corrupt-project fuzz harness must treat any future change to these rules as a versioned migration, not an in-place edit.

## Acceptance Gates
- `cmake --preset release && cmake --build --preset release && ctest --preset release --output-on-failure` passes, including the new schema-validation and round-trip tests.
- A new CI job validates empty, starter (`StarterProject`), and a fully-populated manifest against `docs/schemas/project-v1.schema.json` with a real 2020-12 validator and exits non-zero on any violation.
- Negative test: a manifest with an undeclared top-level field is rejected by the external validator AND by `parseProjectManifest` (`rejectUnknownTopLevelFields`).
- `gainDb`, `bpm`, `pan`, `volumeDb`, and an automation `value` chosen to need >6 significant digits (e.g. `0.1f`, `1.0/3.0`) restore **bitwise** after serialize→parse round-trip.
- Serializing the same manifest under `LC_ALL=C` and `LC_ALL=de_DE.UTF-8` produces **byte-identical** output (assert via a test that switches locale or runs the binary twice with different `LC_ALL`).
- A project/track/marker `name` containing `"\n"`, `"\t"`, and `"\x01"` serializes to JSON that the external validator accepts and that round-trips to the exact original bytes.
- Command serialization: a command carrying a string with `"`, `\`, and a control char produces parseable, escaped JSON (no raw control bytes in the output).
- Durability: an fsync fault-injection/interposition test (e.g. `DYLD_INSERT_LIBRARIES` shim or an injected fd wrapper) forcing the temp-file fsync or rename to fail leaves the previously-saved `project.json` byte-identical to before `save()`.
- Determinism: serializing a fixed manifest twice in the same process yields identical bytes (no map-ordering or float-format drift).

## Notes For Agents
- **Realtime safety:** all of this is disk/serialization code that must never run on the audio callback. `save()`/serialize allocate, do file I/O, and (will) call `fsync`/`F_FULLFSYNC` — keep them strictly on the document/UI/MCP thread. Nothing here touches the render path; do not introduce any header that the audio graph includes transitively.
- **Three escapers, one fix:** `ProjectManifest.cpp:468`, `Plugin.cpp:12`, `Mixer.cpp:73` are byte-for-byte duplicates. Consolidate to avoid a future where one is fixed and two rot. `libs/commands/src/Command.cpp` has *no* escaper at all today — every `serialize()` (first at `:266`) interpolates raw strings.
- **Float emit points (verified):** `ProjectManifest.cpp` `bpm` `:663`, clip `gainDb` `:708`, `volumeDb`/`pan` `:735`, automation `defaultValue` `:758`, point `value` `:774`; `Plugin.cpp` ~`:154`/`:174`; `Mixer.cpp` send `gainDb` `:649`. Parser reads via `requireDouble` (`:399-405`) and `static_cast<float>` — ensure `to_chars` output stays within what `from_chars`/`parseNumber` (`:226-251`) accepts.
- **fsync mechanics:** `ProjectDocument::save()` is `const` and `manifest_`/`path_` are members; the fd plumbing must stay within `save()` (`:50-78`). Preserve the existing atomic temp→rename and the rename-failure cleanup (`:74-76`); add temp-fd fsync before `rename` (`:73`) and parent-dir fsync after. Use `<fcntl.h>`/`<unistd.h>` guarded by `__APPLE__` for `F_FULLFSYNC`.
- **Schema/parser asymmetry to fix:** `rejectUnknownTopLevelFields` (`:308-326`) currently *allows* `loopEnabled`/`loopStartSample`/`loopEndSample` for both v0 and v1, so the C++ side and the JSON Schema disagree. Whatever you decide for the schema, the allow-list, serializer, and validator must end up consistent.
- **Tests location:** until **P02** introduces a real test framework with isolation, add cases to the hand-rolled `tests/unit/bootstrap_tests.cpp` harness (existing `add_test(lamusica_unit_tests …)` in `tests/CMakeLists.txt`). The external schema validator is a separate CI/ctest step. P02's corrupt-project fuzz harness will depend on the conformant format this task establishes.
- **Mapping to existing tasks docs:** no direct analogue in `docs/tasks/026-036` — those (026 JUCE pin/app, 027 DAW shell, 028-033 UIs, 034 MCP, 035 perf/RT, 036 release hardening) all *assume* a trustworthy project format that was never actually delivered. 036's release/docs obligations and the VCS-friendly-format mandate explicitly depend on the byte-deterministic, schema-conformant output this task produces. This is a prerequisite hardening that the 005-style serialization work presumed done.
