## Objective
Turn the mixer model into a real audio-path backbone. Implement Bus-node sub-mixing (independent gain/pan/mute/solo + an insert chain that actually processes audio, distinct from plain summing), pre/post-fader sends that tap a gained copy into a bus, and a multichannel (>2) routing matrix â€” replacing today's stereo-only `nodeChannelGain` and the unhandled `Bus` node kind. This is full production mixing infrastructure, not a stereo demo.

## Decisions To Make
- **Pan law**: choose and document the constant-power / -3 dB law (vs -4.5 dB or -6 dB linear), and define the >2-channel placement model (stereo-pair-per-bus vs true surround speaker angles). This is a product-audible decision that P22's mixer UI and later surround work inherit.
- **Insert processing on buses**: do bus inserts run in-place inside `renderGraph` (simpler, but couples DSP graph to plugin host) or as discrete P12 plugin nodes chained ahead of the bus node (cleaner, but more nodes/edges and ordering constraints)? Decide before touching the GraphCompiler.
- **Mixer persistence location**: embed `MixerState` as a `mixer` block in `project.json` (atomic with the manifest, one schema-version bump, must pass P01's schema conformance) vs a separate `mixer.json` sidecar in the bundle (decouples versioning, but adds a second durable-write/recovery path P08 must cover). Pick one and bump `schemaVersion` (`ProjectManifest.hpp:123`) accordingly.
- **LUFS scope**: ship true ITU-R BS.1770 K-weighted, gated LUFS now, or keep the current `20*log10(rms)-0.691` momentary approximation (`Mixer.cpp:617`) and label it "RMS-approx LUFS" until a dedicated loudness task? Decide what "where practical" means and document the limitation honestly.
- **Sidechain delivery**: `SidechainRoute` (`Mixer.hpp:52-57`) names a `targetInsertId`; decide how the sidechain bus signal is delivered to a P12 plugin's sidechain input (extra graph edge into a secondary input bus) vs deferring true sidechain feed to a follow-up while persisting/validating the routes here.
- **Feedback policy**: which loops are hard-rejected vs allowed-with-explicit-latency (e.g. a feedback send through a delay). Confirm whether `hasRoutingCycle`/`validateRouting` (`Mixer.hpp:96-97`) plus `validateGraph`'s cycle check (`AudioGraph.cpp:94-115,155-164`) are sufficient or need a "safe feedback" exception.

## Compromises And Tradeoffs
- **Accepted**: meter taps publish a downsampled per-block summary (one reading per block), not per-sample envelopes â€” adequate for a UI timer and bounded in CPU; finer scope-style metering is out of scope. Pan law fixed at compile time initially (no per-track law selection) until P22 exposes it.
- **Accepted**: LUFS may remain an RMS-derived approximation this task (see Decisions) provided it is documented as such; covers_gaps notes LUFS only "where practical".
- **Rejected MVP shortcuts**: do **not** keep buses as bare summation nodes (the current reality this task exists to fix); do **not** leave `MixerState` test-only and unpersisted (the session-model gap); do **not** retain stereo-only `nodeChannelGain` returning unity for ch>=2; do **not** poll meters by copying graph buffers off the audio thread under a lock.
- **Risks later tasks must revisit**: P22 (mixer UI) depends on the meter-tap API and routing-matrix shape settled here, so the `MeterTapBank` read API and `RoutingMatrixCell` (`Mixer.hpp:86-94`) semantics are a contract. P18 (sample-accurate automation) will write fader/pan/send gains into this same render path â€” the bus branch must accept per-sample gain ramps without re-architecture. P28 (time-stretch) and surround work inherit the pan-law/channel-layout decision. P14 PDC must account for bus insert latency. This task replaces deferred legacy task `015` and is the audio-path foundation `032` assumes.

## Acceptance Gates
- `cmake --build build && ctest --test-dir build --output-on-failure` is green, including the new mixer/metering tests.
- **Bus sub-mix**: a bus with two sources, gain âˆ’6 dB, pan hard-left, mute off renders summed inputs at the expected left-weighted gains; muting the bus produces silence; with another bus soloed, the unsoloed bus is silent while Master/HardwareOutput pass through (per `channelGain` solo logic, `GraphCompiler.cpp:141-149`).
- **Sends measured at the bus**: with a known source and a send at `gainDb = -6`, the destination bus's measured peak equals `source_peak * dbToLinearGain(-6)` within 1e-4 for post-fader, and equals the pre-fader source (independent of the source fader) for `preFader = true`.

- **Metering correctness**: feed a full-scale and a âˆ’20 dBFS sine; meter peak/RMS match analytic values within tolerance; the clip indicator latches on the full-scale signal and holds until reset (`resetMeter`, `Mixer.hpp:102`); master meter is taken post-master-gain.
