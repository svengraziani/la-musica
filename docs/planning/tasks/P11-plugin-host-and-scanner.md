# P11 JUCE plugin host module + crash-isolated out-of-process scanner with persistent cache

> **Area:** plugin-hosting · **Severity:** blocker · **Effort:** XL · **Phase:** Plugin Hosting Foundation
> **Depends on:** P04, P05 · **Blocks:** P12, P22, P27

## Objective
Stand up *real* plugin hosting for the production DAW: link `juce_audio_processors`, introduce a `PluginHost` that turns a `session::PluginDescription` into a live `juce::AudioPluginInstance` with a correct `prepareToPlay`/`processBlock`/`releaseResources` lifecycle, and replace the entire mocked scan layer with an on-disk scanner that enumerates real Audio Units (via `AudioComponentFindNext`) and configured VST3 paths, extracting real parameter lists, latency, and the instrument flag. Probing must run in a **separate, crash-isolated worker process** with versioned IPC and an enforced timeout so a hung or crashing plug-in cannot take the host down, and results must persist in a deterministic on-disk scan cache keyed by identifier + file mtime. This is the runtime substance that 014/031 only scaffolded; no MVP shortcut (in-process scanning, canned candidate lists, hand-set `PluginScanOutcome`) survives this task.

## Dependencies
- `P04 — Live CoreAudio device I/O on a realtime thread driving the engine callback` — supplies the real sample rate / block size that `prepareToPlay` must be called with, and the RT thread context the host will eventually run inside (via P12).
- `P05 — Validated audio-graph snapshot, precompiled noexcept render path, and true lock-free SPSC command queue` — establishes the RT-safe boundary (`GraphSnapshot`, SPSC queue) that P12 will use to deliver plugin instances/parameters to the audio thread; P11 must produce instances and metadata in a form P12 can publish without allocation on the callback.

## Decisions To Make
- **AU-first vs VST3 parity.** Spec gates VST3 on SDK + license. Decide whether VST3 ships in the first release at all, or whether the runtime probe simply reports it unavailable until the SDK pin + recorded acceptance land. AU is the no-license-cost default on macOS.
- **VST3 SDK licensing.** The Steinberg VST3 SDK is dual-licensed (GPLv3 / proprietary). Decide and record acceptance terms before enabling: how "recorded license acceptance" is represented on disk, who can flip it, and whether GPLv3-linked VST3 hosting is compatible with LaMusica's distribution license (`P27` release pipeline must not silently bundle an incompatible SDK).
- **Scan transport / IPC mechanism.** Choose the parent↔worker channel: JUCE `ChildProcess` + stdin/stdout framed messages, a named pipe, or `juce::InterprocessConnection` over a local socket. Decide the kill semantics on timeout (SIGKILL vs cooperative abort) and how a hung child's partial output is discarded deterministically.
- **Cache key & location.** Decide the exact cache key tuple (identifier + mtime + file size + format-plugin-list version + host ABI version) and where the file lives (`~/Library/Application Support/LaMusica/` vs an XDG-style path) so a host upgrade safely invalidates stale entries. Must round-trip deterministically (byte-stable) for diffability and P01-grade integrity.
- **Probe depth.** Decide whether the worker also runs a minimal `prepareToPlay`/`processBlock` smoke pass (catches plug-ins that crash on activation, not just on construction) versus metadata-only probing — the former is safer for users but slower and must still honor the timeout.
- **Blacklist vs quarantine UX.** Spec blacklists on crash/timeout per policy. Decide whether a blacklisted plug-in is permanently hidden or surfaced as "quarantined, rescan to retry" so the P22 plugin-management UI can offer recovery (`allowPluginRescan` already exists, `Plugin.cpp:230`).

## Compromises And Tradeoffs
- **Accepted:** Out-of-process *scanning* only; live *hosting* stays in-process for now. Per-plugin process spawning is slower than an in-process scan but is the only honest way to keep the app launchable past a crashing plug-in (`covers_gaps`: out-of-process crash isolation). Out-of-process *hosting* (sandboxed per-plugin RT processes) is explicitly deferred — note it for P12/P14 as a future hardening, not a regression.
- **Accepted:** Cache is identifier+mtime keyed, not content-hash keyed — cheaper, but a plug-in rewritten in place with a preserved mtime would be missed. Acceptable; flag for P27 hardening if it bites.
- **Rejected MVP shortcuts:** keeping the `PluginHostEnvironment` boolean table as the source of truth (it must become a runtime probe); fabricating `PluginScanCandidate.outcome` in tests/callers (must come from real worker verdicts); in-process scanning "for speed"; an in-memory-only cache (must persist and survive relaunch); a no-op timeout (`timeoutMilliseconds` must *actually* abort a hung probe).
- **Risks later tasks must revisit:** P12 must consume `AudioPluginInstance` and parameter metadata across the P05 RT boundary with zero allocation on the callback — P11 must hand off instances + a flat parameter index, not `std::string`-keyed maps, so P12 can publish them via `GraphSnapshot`. P13 needs the opaque-state path; P11 must not foreclose `getStateInformation`/`setStateInformation`. PDC (P14) consumes the latency this scanner discovers — store it, do not discard it.

## Acceptance Gates
- `cmake --build build --target lamusica_plugin_host lamusica_unit_tests apps/plugin_scan_worker && ctest --test-dir build -R plugin` builds and passes.
- **Real AU lifecycle:** a real system Audio Unit (e.g. `AUDelay`/`AULowpass`) instantiates through `PluginHost`, `prepareToPlay` is provably called before `processBlock`, and processing a non-silent buffer yields output that differs from the input (assert sample difference). No leaks under `-DLAMUSICA_ENABLE_ASAN=ON` (`CMakeLists.txt:19`).
- **Scan populates real cache:** scanning the system AU set produces `PluginScanResult`s with real parameter lists and latency; the on-disk cache file is written deterministically; a second launch reuses it without re-probing (assert no worker spawns when mtimes are unchanged), and touching a plug-in's mtime forces a re-probe.
- **Timeout is real:** a mock that sleeps past `policy.timeoutMilliseconds` is recorded as `PluginScanOutcome::TimedOut` and the parent returns within ~timeout + a bounded margin (not hung).
- **Crash is contained:** a mock that segfaults during probe is recorded as `Crashed`, gets blacklisted when `policy.blacklistOnCrash` is true (`Plugin.cpp:291-304`), and the host process exits cleanly / `report.appLaunchSafe` stays true.
- **Capability probe:** `pluginFormatSupport` reports AU available on macOS from the live `AudioPluginFormatManager`, and VST3 reports unavailable unless the SDK pin is present AND license acceptance is recorded (assert both `vst3_sdk_not_available` and `vst3_license_not_accepted` reason paths, `Plugin.cpp:250-253`).
- **Determinism:** serializing the same scan result twice produces byte-identical cache files (locale-independent floats, per P01).

## Notes For Agents
- **Realtime/safety:** scanning, IPC, cache I/O, and JSON are all off-thread — never touch the audio callback here. `prepareToPlay`/`releaseResources` allocate and must happen off the RT thread; the RT-safe *delivery* of a prepared instance to the audio graph is P12's job via the P05 SPSC queue and `GraphSnapshot` double-buffer (see `P05-audio-graph-snapshot-spsc-queue.md`). `RealtimeCommandQueue` (`AudioEngine.hpp:74-81`) is the existing interim ring; do not push plugin-instance pointers through it until P12 hardens the handoff.
- **Don't rewrite policy:** `scanPluginCandidates` (`Plugin.cpp:258-331`), `blacklistPlugin` (`:219`), `isBlacklisted` (`:201`), `mergeScanResult` (`:205`), `allowPluginRescan` (`:230`) already encode the merge/blacklist policy correctly and are well-tested — the bug is upstream (mocked `outcome`). Feed them real verdicts; reuse `stableParameterAddress`/`discoverPluginParameters` (`Plugin.cpp:347`, `:354`) for automation-address stability so P18 automation keeps working across reloads.
- **Capability table to kill:** `PluginHostEnvironment` (`Plugin.hpp:80-86`) booleans `audioUnitRuntimeAvailable`/`vst3SdkAvailable`/`outOfProcessHostingAvailable` are placeholders consumed by `pluginFormatSupport` (`Plugin.cpp:236-256`). Replace their *source* with a runtime probe; keep the `PluginFormatSupport` output contract.
- **OBJCXX:** AU enumeration (`AudioComponentFindNext`, AudioToolbox/AudioUnit frameworks) is Objective-C++; the root already enables OBJCXX on Apple (`CMakeLists.txt:9-11`). Link `"-framework AudioToolbox" "-framework AudioUnit" "-framework CoreAudioKit"` as needed.
- **SDK pinning** is a build concern folded in here as a hard prerequisite (it was a 036/build item): extend `cmake/CheckDependencyLock.cmake` and document in `docs/developer/dependencies.md`, matching the JUCE pin discipline at `CMakeLists.txt:21-30`.
