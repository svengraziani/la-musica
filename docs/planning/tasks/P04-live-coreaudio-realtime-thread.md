# P04 Live CoreAudio device I/O on a realtime thread driving the engine callback

> **Area:** audio-engine · **Severity:** blocker · **Effort:** L · **Phase:** Phase 1 — Realtime Audio Foundation
> **Depends on:** none · **Blocks:** P05, P10, P11, P16, P17, P20, P29

## Objective
Make the engine produce and capture real audio on hardware. Stand up a `juce::AudioDeviceManager`, enumerate real CoreAudio input/output devices (sample rates, buffer sizes, channel counts), and install an `AudioIODeviceCallback` on a realtime-priority thread that drives the existing engine render path with the device-provided block size. This is the foundation the entire production DAW silently assumes: a loaded project must play audibly through a selected device with a sample-accurate playhead, device open/select must fail safely, and runtime sample-rate/buffer-size changes must reallocate off-thread and swap atomically with zero allocation on the callback.

## Dependencies
None (foundation task). It does, however, share a correctness boundary with **P29 — Project sample-rate field, asset resampling on import, and SR-mismatch reconciliation**: P04 must not silently assume `project-SR == device-SR` (see Decisions and Notes). P29 depends on P04; P04 references the contract P29 will land.

## Deliverables
New device-I/O backend in `libs/audio` (or a thin `libs/audio_host` target if we keep the pure-DSP core JUCE-free — see Decisions):
- New `libs/audio/include/lamusica/audio/AudioDeviceHost.hpp` + `src/AudioDeviceHost.cpp`: owns `juce::AudioDeviceManager`, implements `juce::AudioIODeviceCallback` (`audioDeviceIOCallbackWithContext`, `audioDeviceAboutToStart`, `audioDeviceStopped`, `audioDeviceError`), and bridges the device buffer to `AudioEngine::renderGraphBlock`.
- Replace the hardcoded `AudioDeviceInfo` POD and stub `selectDevice` in `libs/audio/include/lamusica/audio/AudioEngine.hpp:20-27,99` and `libs/audio/src/AudioEngine.cpp:186-197`. Add real enumeration: a `std::vector<AudioDeviceInfo>` populated from `AudioIODeviceType::getDeviceNames()` / `AudioIODevice::getAvailableSampleRates()` / `getAvailableBufferSizes()` / `getInputChannelNames()` / `getOutputChannelNames()`, plus an `AudioDeviceInfo` that carries the *available* lists, not just the single chosen values currently hardcoded to 48000/512/2.
- Off-thread reallocation + atomic swap: extend the buffer-sizing currently done eagerly in the ctor (`AudioEngine.cpp:130-131`) and `selectDevice` (`AudioEngine.cpp:195-196`, both call `realtimeGraphBuffers_.resize/assign`) so block-size/channel changes resize a *new* buffer on the message/worker thread and publish it via an atomic pointer the callback reads; the callback never calls `resize`/`assign`. Keep `renderGraphBlock`'s existing bounded checks (`AudioEngine.cpp:391-404`).
- Master output clamp/soft-limit: a final RT-safe stage applied to the interleaved block before it is copied into the device output channels, replacing/guarding the raw copy. Sanitize NaN/Inf and clamp/soft-limit to [-1, 1]; raise a lock-free "clip occurred" flag for the UI.
- CMake: add `juce_audio_devices` and `juce_audio_utils` (and their transitive `juce_audio_basics`) link in `libs/audio/CMakeLists.txt` (currently links nothing JUCE) and/or the new host target; add the device-host source to the lib. Root `CMakeLists.txt:63` only guards `juce::juce_gui_extra` today — add a guard for `juce::juce_audio_devices`.
- Integration test: new `tests/integration/audio_device_loopback_tests.cpp` registered in `tests/CMakeLists.txt`, exercising the host against a virtual/loopback device (offline `AudioIODeviceType` or a BlackHole/ARM "Aggregate" device gated behind an env var) for a 60s glitch-free run with underrun counting; plus a fast unit-level test that block-size changes trigger no callback-thread allocation.

## Decisions To Make
- **JUCE in the DSP core vs. a host shim.** Linking `juce_audio_devices` into `lamusica::audio` (currently a JUCE-free STATIC lib) couples the pure engine to JUCE. Decide: link directly, or introduce `lamusica::audio_host` that owns the device manager and calls into the JUCE-free `AudioEngine`. Recommendation: a thin host target keeps `renderGraphBlock` testable headless.
- **Project-SR vs. device-SR contract (correction-driven).** The project format is entirely sample-based (`startSample`/`lengthSamples`/`sourceOffsetSamples` at `ProjectManifest.hpp:71`, automation `samplePosition`) and stores **no** project sample rate — grep confirms none in `docs/schemas/project-v1.schema.json` or `ProjectManifest.hpp`. `EngineConfig.sampleRate` defaults to 48000 (`AudioEngine.hpp:15`); opening a 48k-authored project on a 44.1k device would play every clip/automation point at the wrong time and pitch, and `GraphCompiler` reads `wav.audio.interleavedSamples` 1:1 with no resampling (`libs/session/src/Assets.cpp:123`). Decide the interim behavior P04 ships: drive the device at the project's intended rate where supported, else (a) refuse + clear error, or (b) play with a documented "SR mismatch — pitch/timing may be wrong" warning until P29 lands resampling. Do **not** silently run at the wrong rate.
- **Realtime thread priority/affinity policy.** Use JUCE's device-managed RT thread vs. setting an explicit `THREAD_TIME_CONSTRAINT_POLICY` on macOS; decide whether we set workgroup membership (`os_workgroup`) for the audio thread now or defer to P06.
- **Default device selection + persistence.** Which device on first run (system default output), how a missing/removed device falls back, and whether the chosen device id/SR/block lives in preferences. Coordinate the persisted shape with P19/P20 (UI) but define the data here.
- **Channel mapping.** How >2-channel devices map to the stereo master today (take first 2 / down-mix), and whether input channels are captured now or stubbed until P17 recording.
- **Limiter character.** Hard clamp vs. cheap soft-clip (tanh/cubic) for the master safety stage — must be allocation-free and deterministic; a true look-ahead brickwall limiter is out of scope and deferred to P15.

## Compromises And Tradeoffs
- **Accepted:** Master stage is a *safety* clamp/soft-limit, not a mastering limiter; real bus/channel-strip processing and metering taps are P15. Input capture may be wired but unused until P17. Multichannel devices may be reduced to stereo for now.
- **Accepted:** P04 ships a *defined* SR-mismatch policy (error or warn), not resampling. The actual resample-on-import + Sample-node SR conversion is P29; P04 must encode the contract so P29 slots in without re-plumbing the callback.
- **Rejected MVP shortcut:** No "render to a WAV and play it back with a system player," no allocation on the callback "just to ship," no fixed 48k/512 assumption. The hardcoded `AudioDeviceInfo`/`EngineConfig` defaults that exist today (`AudioEngine.hpp:14-27`) are the placeholder this task removes.
- **Risks later tasks must revisit:** P05 replaces today's eager `realtimeGraphBuffers_` sizing and the bounded but non-snapshotted `renderGraphBlock` with a validated precompiled noexcept render path + true SPSC queue — P04's atomic-swap must be compatible with that, not a parallel mechanism. The current `RealtimeCommandQueue` (`AudioEngine.hpp:74-89`) is a single-thread ring with a plain `size_` counter, **not** lock-free SPSC; P04 may use it across the new RT thread only as an interim measure and must flag it for P05 to harden.

## Acceptance Gates
- `cmake --build build/unix-debug --target lamusica_daw lamusica_unit_tests && ctest --test-dir build/unix-debug --output-on-failure` passes, including the new loopback test.
- A loaded project plays audibly through the selected CoreAudio device; the playhead (`TransportState::samplePosition`, advanced in `AudioEngine::advanceTransport`, `AudioEngine.cpp:270-288`) advances by exactly the device block size each callback — assert sample-accuracy against a counted-frame loopback capture.
- `enumerateDevices()` returns the real device list with non-empty available sample-rate/buffer-size/channel sets; selecting a valid device reconfigures SR/block/channels; selecting an invalid one throws/returns a user-visible error and leaves the previous device running (no crash).
- Block-size change at runtime performs the realloc off-thread and atomic swap with **zero** allocation observed on the callback thread (verify via a callback-scoped allocation hook / counter in the test).
- Hot-plug / device-removed (`audioDeviceError`/`audioDeviceStopped`) falls back safely to the default or to silence with a surfaced error, never crashes.
- 60s continuous loopback run: zero underruns/dropouts logged; no NaN/Inf or out-of-range samples reach the device buffer (inject a deliberately over-unity graph and assert the clamp engages and the clip flag is raised).
- SR-mismatch case: opening a project whose intended rate differs from the device rate produces the chosen defined behavior (clean error OR explicit warning), never silent wrong-pitch playback.

## Notes For Agents
- Realtime constraints (hard): inside `audioDeviceIOCallbackWithContext` — no `new`/`malloc`, no locks, no file I/O, no logging, no JSON/MCP work, no exceptions escaping. `renderGraphBlock` currently *throws* on bounded-capacity / cycle / buffer-size violations (`AudioEngine.cpp:392-453`); on the RT thread these must be pre-validated off-thread (the precompiled graph is P05) and the callback must treat any residual error as "output silence + set error flag," not throw across the device boundary.
- Buffer ownership: today the ctor and `selectDevice` mutate `realtimeGraphBuffers_` directly (`AudioEngine.cpp:130-131,195-196`). For runtime changes, allocate the replacement vector on the message thread, store it behind a `std::atomic<Buffers*>` (or double-buffer + flag), and have the callback `load(std::memory_order_acquire)` once per block. Free the old buffer on the non-RT thread after the swap is observed.
- Clamp/sanitize: do it on the interleaved span just before the device copy, using branch-light `std::isfinite` replacement-with-0 then `std::clamp(x, -1.0f, 1.0f)` (or chosen soft-clip). The clip indicator must be a lock-free flag read by the UI/metering, not a log call.
- Channel/format bridging: JUCE hands the callback non-interleaved `float* const*`; the engine render path is interleaved (`renderGraphBlock(std::span<float>, frames)`, `AudioEngine.hpp:120-121`). Convert without per-sample allocation, reusing a preallocated scratch buffer sized to `maxBlockSize * outputChannels` (same sizing math as `sampleCount(...)` used at `AudioEngine.cpp:131`).
- `ApplicationSession` already owns an `AudioEngine` (`libs/session/include/lamusica/session/ApplicationSession.hpp:349`) but only renders offline; the device host should drive *that* engine instance so the live and offline paths stay coherent, or expose a clear seam for P20's transport bridge.
- CMake: `libs/audio/CMakeLists.txt` currently links no JUCE module; JUCE is provided via `LAMUSICA_JUCE_PATH` `add_subdirectory` at root `CMakeLists.txt:34-67`. Add `juce_audio_devices`/`juce_audio_utils` to the (host) target's `target_link_libraries` and a target guard mirroring `CMakeLists.txt:63`.
- Mapping to existing tasks docs: this is the unstated foundation that 027-036 all assume but none own. It is closest to the deferred "audio engine core" (legacy 004) elevated to a blocker, and it directly enables **027** (`docs/tasks/027-production-daw-shell.md`, the transport bridge) and **035** (`docs/tasks/035-performance-stress-realtime-verification.md`, realtime verification). P04 supersedes the implicit "audio just works" assumption baked into 027/035 and feeds P05 (snapshot/SPSC), P10/P17 (recording), P11/P16 (plugin/MIDI I/O), and P20 (transport UI bridge).
