# P30 Accessibility: VoiceOver/AX wiring, keyboard operability, contrast, and reduced-motion across the DAW UI

> **Area:** daw-ui · **Severity:** high · **Effort:** L · **Phase:** Phase 5 — Interactive DAW UI
> **Depends on:** P19, P21, P22, P23 · **Blocks:** none

## Objective
Make the entire LaMusica UI usable by assistive technology and by keyboard alone — a hard requirement for a professional macOS / Mac App Store DAW. Every interactive custom `juce::Component` built by P19–P23 (transport, timeline lanes/clips/ruler, mixer strips/faders/meters, piano roll, drum pads, step grid, clip launcher, plugin chooser/insert rack, browser tree, Inspector, export dialog) must expose a `juce::AccessibilityHandler` with a correct role, accessible name/value/description, and actions, so VoiceOver announces controls and their live state. The full primary workflow set (transport, edit a clip, set a fader, insert a plugin, export) must be completable without a mouse, with a logical focus order, and the app must honor macOS Increase Contrast and Reduce Motion (suppressing continuous playhead/meter animation). The repo currently has zero accessibility and zero keyboard-focus code (`grep -rn AccessibilityHandler|setWantsKeyboardFocus apps libs` → no hits), so this task installs that layer across the existing component tree without changing the engine.

## Dependencies
- P19 — Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding
- P21 — Arrangement timeline with clip/automation editing, plus high-level audio clip editing UI
- P22 — Mixer, plugin-management, and recording UIs
- P23 — Drum machine, step sequencer, clip-launcher, browser/inspector, and export UIs

## Deliverables
- **New** `apps/daw/src/ui/a11y/AccessibleControl.{hpp,cpp}` — shared helpers/mixins for the common patterns: a value-text formatter set (`formatGainDb`, `formatPan`, `formatBarBeat`, `formatMeter`) reused by every fader/pan/meter/position so announcements are consistent. dB/pan formatting mirrors the session value fields already present (`ChannelStrip::volumeDb`/`pan`, `ApplicationSession.hpp:201-202`; `ApplicationSessionStatus::drumClipGainDb`/`bassClipGainDb`, `ApplicationSession.hpp:42-43`). Position text comes from `AudioEngine::samplesToBarBeat` (`libs/audio/include/lamusica/audio/AudioEngine.hpp:113`), not recomputed in the UI.
- **New** `apps/daw/src/ui/a11y/LiveRegion.{hpp,cpp}` — a wrapper over `juce::AccessibilityHandler::notifyAccessibilityEvent` / `postAnnouncement` for transport- and record-state announcements ("Playing", "Stopped", "Recording armed", "Loop on 0–96000"), driven off the ~30 Hz binding timer P19 already runs (`docs/planning/tasks/P19-interactive-daw-shell.md:34`). Coalesce so identical state does not re-announce.
- **New** `apps/daw/src/ui/a11y/MotionPreferences.{hpp,cpp}` + `ContrastPreferences.{hpp,cpp}` — query macOS `NSWorkspace` accessibilityDisplayShouldReduceMotion / accessibilityDisplayShouldIncreaseContrast (and the change notifications), exposing observable flags. These gate the P21 playhead animation and the P22 meter `juce::Timer` repaints, and switch the LookAndFeel palette to a high-contrast variant.
- **Change** the P19 region classes (`TransportRegion`, `BrowserRegion`, `TimelineRegion`, `InspectorRegion`, `MixerRegion` under `apps/daw/src/ui/`, per `docs/planning/tasks/P19-interactive-daw-shell.md:14`): each region and its child interactive components override `createAccessibilityHandler()` returning a handler with the right `juce::AccessibilityRole` and a non-empty name. Regions map to the existing `ApplicationPanel` enum (`libs/session/include/lamusica/session/ApplicationSession.hpp:242-248`) so AX focus and `session_.focusPanel(...)`/`focusedPanel()` (`ApplicationSession.hpp:279,335`) stay in sync.
- **Change** the P21 timeline components (`apps/daw/src/timeline/TimelineComponent`, `TrackLaneComponent`, `ClipComponent`, `TimeRulerComponent`, `MinimapComponent`, per `docs/planning/tasks/P21-arrangement-timeline-clip-editing.md:15-19`): clips get role `button`/`listItem` with name = clip label + bar/beat span; lanes are accessible containers; the playhead seek action is keyboard-reachable (arrow/Home/End) calling the P20 bridge / `AudioEngine::seekSamples` (`AudioEngine.hpp:106`). Gate continuous playhead repaint on `MotionPreferences`.
- **Change** the P22 mixer/plugin/recording components (`ChannelStripComponent`, `MeterComponent`/`MeterPoller`, `RoutingMatrixComponent`, `PluginChooser`, `InsertChainComponent`, `RecordingControls`, per `docs/planning/tasks/P22-mixer-plugin-recording-ui.md:18-26`): faders get role `slider` with `dB` value text + `setValue` action; mute/solo/arm/monitor/phase get role `toggleButton` with on/off value; meters get a read-only value (peak dB, clip flag) and are excluded from the focus order. The meter timer (P22 30–60 Hz) freezes its visual animation under Reduce Motion while still exposing the latest value text.
- **Change** the P23 surfaces (`DrumMachineView` pads, `StepSequencerView` grid, `ClipLauncherView` scene/slot grid, `PianoRollView`, `BrowserView` tree, `InspectorView`, `ExportDialog`, per `docs/planning/tasks/P23-drums-patterns-browser-export-ui.md:18-25`): step cells are toggle buttons with row/column position in the name; the browser is an accessible tree; the modal `ExportDialog` traps focus, has a labeled control for every `ProjectExportOptions`/`StemExportOptions` field (`libs/session/include/lamusica/session/Export.hpp:13-38`), and returns focus on close.
- **New tests** in `tests/unit/bootstrap_tests.cpp` (current `require(...)` harness; migrate to the P02 framework when present) + a headless accessibility-audit driver registered in `tests/CMakeLists.txt`: instantiate the component tree under `juce::ScopedJuceInitialiser_GUI`, recursively walk children, and **fail if any interactive component lacks an `AccessibilityHandler` or has an empty accessible name or unset role**. Add focus-order and keyboard-reachability assertions and a Reduce-Motion-disables-animation assertion.
- **Change** `apps/daw/CMakeLists.txt` — add the new `ui/a11y/*` sources to `lamusica_daw`; add the audit test target to `tests/CMakeLists.txt`. A `--a11y-audit` smoke flag in `apps/daw/src/main.cpp` (mirroring the existing `--app-session-*-smoke` flags, `apps/daw/src/main.cpp:701-742`) lets CI run the headless walk without a display.

## Decisions To Make
- **Audit strictness vs. allowlist:** does the audit treat *every* leaf `Component` as needing a handler, or only those returning `true` from a project `isInteractive()` marker? Decide a single, explicit allowlist for genuinely decorative components (backgrounds, separators) so the test is strict but not noisy.
- **Value-text source of truth:** announce engine-derived values (read from `ApplicationSession::status()` / `samplesToBarBeat`) vs. the widget's own display string. Recommend engine-derived so VoiceOver and the visible label can never diverge; settle dB/pan rounding (e.g. `-6.0 dB`, `Pan L23` / `Center`).
- **Reduce-Motion semantics:** does Reduce Motion *stop* the playhead/meter repaint entirely (jump-update on a slow timer) or merely *reduce* it? Decide the slow-update cadence and confirm it still satisfies "stops continuous animation" while keeping playback position legible.
- **Increase-Contrast strategy:** ship a dedicated high-contrast `LookAndFeel` palette vs. programmatically boosting the existing dark palette (current placeholder uses `0xff202124`/`0xff30363d`, `apps/daw/src/main_juce.cpp:43,139-143`). Decide minimum contrast-ratio target (WCAG AA 4.5:1) and whether focus rings are always drawn or only under Increase Contrast.
- **Live-region verbosity:** which transitions announce (play/stop/record/loop/punch) and how to debounce so rapid transport toggles or per-frame playhead motion do not flood VoiceOver.
- **Mac App Store posture:** whether accessibility conformance is a hard release gate in P27 and whether a VoiceOver manual-pass checklist is added to `docs/release/release-checklist.md`.

## Compromises And Tradeoffs
- **Accepted:** this task adds the AX/keyboard/contrast/motion layer *onto* components owned by P19–P23; it does not redesign those components. If a P19–P23 component is structurally inaccessible (e.g. a single monolithic canvas with no child focusables for clips/steps), this task may require a small refactor of that component, coordinated with its owning task rather than reimplemented here.
- **Accepted:** automated coverage proves *presence* of a handler with name/role and basic keyboard reachability; it cannot prove the announcement *reads well*. A scripted VoiceOver manual pass for the five primary workflows is the qualitative gate and is not fully CI-automatable on macOS.
- **Rejected MVP shortcuts:** no "tab order is good enough" hand-wave — every interactive control must expose a handler; no meter/playhead animation that ignores Reduce Motion; no hardcoded value strings that bypass the shared formatters; no decorative-only inaccessible custom widgets standing in for real controls.
- **Risks later tasks must revisit:** the P23 clip-launcher model is *new* (`libs/session/include/lamusica/session/ClipLauncher.hpp`, per `docs/planning/tasks/P23-drums-patterns-browser-export-ui.md:21`) — its scene/slot grid must be made accessible the moment it lands. P28 (warp/stretch) and P29 (SR reconciliation) add controls/dialogs that must inherit these AX hooks. P31 (localization) will translate the accessible names/value-text these formatters emit — keep all announced strings funneled through one place so they are localizable.

## Acceptance Gates
- **Build + suite green:** `cmake --build build --target lamusica_daw lamusica_unit_tests lamusica_daw_smoke && ctest --test-dir build --output-on-failure`, including the new accessibility-audit, focus-order, and reduced-motion cases.
- **Coverage check fails on regression:** the headless audit (`--a11y-audit`) walks the full component tree and fails if any interactive component lacks an `AccessibilityHandler`, has an empty accessible name, or an unset role. Add a deliberately-unlabeled fixture component in a test and assert the audit catches it.
- **Keyboard-only workflows:** a headless keyboard-driver test completes each primary workflow without synthetic mouse events — start/stop transport, select+trim a clip, change a fader value, insert a plugin via the chooser, open and confirm the export dialog — asserting the resulting commands fire and focus lands logically (regions resolve to the correct `ApplicationPanel`).
- **Value text correct:** assert a fader at -6 dB announces dB value text equal to `formatGainDb` over the strip's `volumeDb`; a meter announces peak dB + clip state; the playhead/ruler announces a bar/beat position equal to `samplesToBarBeat` for the current `playheadSample`.
- **Reduce Motion:** with the Reduce-Motion flag forced on, assert the playhead/meter timer no longer drives continuous repaints (animation tick count stays at the slow cadence) while value text still updates.
- **Increase Contrast:** with the flag forced on, assert the active palette switches to the high-contrast variant and primary control foreground/background pairs meet the chosen contrast ratio (computed in the test).
- **Manual VoiceOver pass:** documented checklist run — VoiceOver announces transport, a mixer strip, a timeline clip, a piano-roll note, and a plugin control with correct role/name/value; each of the five primary workflows is completable by keyboard with VoiceOver on.

## Notes For Agents
- **Realtime/safety:** accessibility is message-thread-only. AX queries, announcements, contrast/motion observers, and value-text formatting must never touch the audio callback (`AudioEngine::renderGraphBlock`, `libs/audio/src/AudioEngine.cpp` render path) — read only the published `ApplicationSessionStatus` snapshot and the P15 lock-free meter taps that P22 already drains on a `juce::Timer`. Do no allocation in the meter/playhead repaint suppression path.
- **JUCE wiring:** override `createAccessibilityHandler()` on each component (returning a `std::unique_ptr<juce::AccessibilityHandler>` with `AccessibilityRole`, `AccessibilityActions`, and a `juce::AccessibilityValueInterface` / `AccessibilityTextInterface` where applicable); set `setAccessible(true)` and `setTitle(...)` on containers. `juce::AccessibilityHandler`, roles, and `KeyPressMappingSet` all live in `juce_gui_basics`, already linked transitively via `juce::juce_gui_extra` (`apps/daw/CMakeLists.txt:9`). P19 already installs the `KeyPressMappingSet` and `setWantsKeyboardFocus(true)` regions (`docs/planning/tasks/P19-interactive-daw-shell.md:14,19`) — extend, do not duplicate.
- **Focus contract:** the focus traversal must walk the five P19 regions and stay consistent with `ApplicationPanel` (`ApplicationSession.hpp:242-248`) and `focusPanel`/`focusedPanel` (`ApplicationSession.hpp:279,335`; impl `ApplicationSession.cpp:1960`) so view commands resolve against the live focus. Modal surfaces (`ExportDialog`, plugin editor windows) must trap and restore focus.
- **What you are building on, not replacing:** the current `apps/daw/src/main_juce.cpp` is still the `StatusPanel` placeholder grid (`main_juce.cpp:25-55` for `StatusPanel`, `:57-144` for `MainComponent`) with no accessibility and read-only `juce::Label`s — that is exactly what P19–P23 dismantle into real components, and this task makes those real components accessible. Do not wire accessibility onto the placeholder `Label`s.
- **Maps to existing tasks docs:** no doc in `docs/tasks/026-036` owns accessibility — `docs/tasks/027-production-daw-shell.md:19` only mentions "Keyboard focus and shortcut routing across panels" in passing, and the mixer/timeline UI docs (`032`, `028`/`029`) say nothing about AX/VoiceOver/contrast/reduced-motion. P30 is the cross-cutting accessibility task that hardens the entire 027–033 UI line; it adds the layer those tasks omitted rather than replacing any of them.
