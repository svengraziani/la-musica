# P25 Live MCP-to-running-DAW bridge, OS daemon lifecycle, sandbox confinement, and addressable undo registry

> **Area:** mcp · **Severity:** blocker · **Effort:** XL · **Phase:** AI Delegation & Operability — Live MCP Integration
> **Depends on:** P05, P19, P24 · **Blocks:** P27

## Objective
Turn the headless MCP logic engine into a real AI-delegation loop that operates the *running* DAW: an MCP `apply` mutates the same authoritative `ProjectDocument`/`ProjectManifest` the UI is bound to (visible without reload) and mirrors a validated snapshot to the audio engine over the lock-free path from P05 — with zero allocation/lock/IO/JSON/MCP work on the audio callback during an edit, and invalid commands never reaching the graph. Replace the in-memory daemon fiction with a real launchd-supervised process (install/launch/stop/health probing the live PID/service) confined by an OS-level sandbox that holds even if the protocol denylist is removed. Add an addressable, durable undo-id registry so an agent can undo a *specific* past operation across daemon restart, and an MCP-aware UI surfacing live reconciliation, confirmation-token prompts, and per-action agent-activity/audit affordances gated by `allowMcpProjectMutation`.

## Dependencies
- P05 — Validated audio-graph snapshot, precompiled noexcept render path, and true lock-free SPSC command queue
- P19 — Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding
- P24 — Standards-compliant MCP/JSON-RPC server with edit/render/orchestration tools and correct JSON

## Deliverables
- **Shared-session bridge (new).** `libs/mcp_bridge/src/LiveSessionBridge.cpp` + `include/lamusica/mcp_bridge/LiveSessionBridge.hpp`: a single object owning a reference to the app's authoritative `session::ProjectDocument`/`ProjectManifest` and its `commands::CommandHistory` (`libs/commands/include/lamusica/commands/Command.hpp:1010`). MCP `applyCommand` (currently in `libs/mcp_bridge/src/EditTools.cpp:406`) routes through this bridge instead of a private manifest copy, so the same `CommandHistory::execute` the UI uses is the single mutation entry point. After each applied command, recompile the graph (`session::compileGraph` in `libs/session/src/GraphCompiler.cpp`) and publish a validated `AudioGraph` snapshot to the engine via the P05 snapshot-swap API — never by mutating `AudioEngine::renderGraphBlock`'s `const AudioGraph&` argument (`libs/audio/src/AudioEngine.cpp:387`) on the caller side. The bridge marshals UI/MCP edits onto the message thread (JUCE) so both share one undo timeline.
- **Addressable durable undo-id registry (new).** `libs/mcp_bridge/src/UndoRegistry.cpp` + header. Extend `commands::CommandHistory` (`Command.hpp:1010-1024`, today a bare `std::vector<CommandPtr>` undo/redo pair with last-only `undo()`/`redo()`) with stable, externally-addressable ids keyed off `CommandMetadata.auditId` (`Command.hpp:24-28`). Add `undoById(auditId)` / `redoById(auditId)` semantics (defining how non-tip undo interacts with the stack — see Decisions) and persist the registry to a sidecar next to `project.json` (`ProjectDocument::manifestFileName`, `libs/session/include/lamusica/session/ProjectDocument.hpp:12`) using serialized commands (`ICommand::serialize()` + `commandFromSerialized`, `Command.hpp:1032`), so an attached project's registry survives daemon restart. Replaces the `"undo"`/`"redo"` stub auditIds in `EditTools.cpp:457-487`.
- **Real launchd lifecycle (rewrite).** Replace the in-memory state machine in `libs/mcp_bridge/src/DaemonSession.cpp:58-76` (`install`/`launch`/`stop` just flip `lifecycleStatus_`). New `libs/mcp_bridge/src/LaunchdService.cpp` + header: `--install` writes a `com.lamusica.mcpd.plist` LaunchAgent to `~/Library/LaunchAgents/` and `launchctl bootstrap`/`load`s it; `--launch` starts the supervised process; `--stop` `launchctl bootout`/`unload`s it; `--health` probes the live service via `launchctl print`/PID liveness rather than returning an enum-derived string (`DaemonSession::health()`, `DaemonSession.cpp:24-32`, consumed by `apps/mcpd/src/main.cpp:37-44`). Wire `KeepAlive` restart behavior in the plist. Ship the plist template under `apps/mcpd/resources/`.
- **OS-level sandbox (new).** `apps/mcpd/resources/lamusica-mcpd.sb` seatbelt profile and/or `mcpd.entitlements` (App Sandbox) applied at daemon launch: deny `process-exec`/`process-fork`, jail `file-read*`/`file-write*` to the attached project root + asset/cache roots only. This is defense-in-depth *behind* the existing string denylist `classifyForbiddenProtocolRequest` (`libs/mcp_bridge/src/Protocol.cpp:388-405`), which is bypassable and stays only as a fast first-line filter.
- **MCP-aware UI (change).** Extend the P19 shell — today `apps/daw/src/main_juce.cpp` hardwires `createUntitledProject()` (`main_juce.cpp:80-86`) into static `StatusPanel` labels with no MCP awareness. Add: live reconciliation of external edits into the bound document/view, confirmation-token prompts driving `EditApplyOptions.confirmationToken` (`EditTools.cpp:431`, `commandNameRequiresConfirmation` `EditTools.cpp:31-35`), and an agent-activity/audit view rendering `DaemonSession::daemonLog()`/`readAuditLog()`/`deniedProtocolLog()` (`DaemonSession.hpp:65-67`) with a per-action undo affordance bound to the undo-id registry. A new `allowMcpProjectMutation` preference gates whether the bridge accepts mutating commands at all.
- **Tests (new).** `tests/mcp_live_bridge_test.cpp`, `tests/daemon_lifecycle_test.cpp`, `tests/mcp_sandbox_negative_test.cpp` (under `tests/CMakeLists.txt`), plus an e2e fixture extending `tests/fixtures/mcp_fixture_client.cpp` covering create→load→MCP edit→play→export.

## Decisions To Make
- **Same-process vs shared-memory bridge.** Whether `lamusica-mcpd` and the DAW are one process (daemon embedded in the app, simplest single-`ProjectDocument` ownership) or two processes sharing the document via IPC/shared memory. The spec says "in-process/shared-session"; pick the topology and define who owns the authoritative `CommandHistory` and on which thread it executes.
- **Non-tip undo semantics.** What `undoById` of a non-tip operation means: a true selective/inverse-patch undo (hard, risks divergence), vs. "roll back to that point" (sequential), vs. only the tip is durably addressable. Define interaction with redo and with interleaved UI edits on one timeline.
- **Sandbox mechanism: App Sandbox entitlements vs seatbelt (`.sb`).** App Sandbox integrates with notarization/hardened runtime (P27) but is heavier to scope; seatbelt (`sandbox_init`) is finer-grained but quasi-deprecated. Decide which (or both) and how the FS jail enumerates project/asset/cache roots when the attached project changes.
- **launchd registration scope.** Per-user LaunchAgent (no privilege, recommended) vs system LaunchDaemon. Decide `KeepAlive` policy (always vs crash-only) and how `--health` distinguishes "installed-not-running", "running-idle", "running-attached", and "crashed-pending-restart" against the four-state `DaemonLifecycleStatus` (`DaemonSession.hpp:12-17`).
- **Durable registry format and trust.** Whether the undo-id sidecar reuses the P08 binary sidecar format or a JSON sidecar by `project.json`; how it is reconciled if the project file changed on disk between daemon sessions (stale-registry detection).
- **Mutation consent surface.** Default value of `allowMcpProjectMutation` (default-deny recommended) and whether confirmation tokens are per-op single-use or per-session, given `confirmationTokenForMetadata` derives a deterministic token today (`EditTools.cpp:37-39`).

## Compromises And Tradeoffs
- **Accepted:** initial bridge may serialize all MCP edits onto the JUCE message thread (no concurrent MCP/UI mutation) to guarantee one consistent undo timeline; throughput is secondary to correctness. The string denylist (`Protocol.cpp:388-405`) is kept as a cheap first filter even though the OS sandbox is now authoritative.
- **Accepted:** non-tip undo may ship as "roll back to point" rather than true inverse-patch selective undo if the latter cannot be made consistent under interleaved edits within this task.
- **Rejected MVP shortcuts:** keeping the in-memory lifecycle fiction (`DaemonSession.cpp:58-76`) or an enum-derived `--health`; an MCP-only mutation path that bypasses the DAW's `CommandHistory` validator (034 already forbids this); leaving the sandbox as the bypassable denylist with no OS confinement; requiring a project reload to see MCP edits.
- **Risks later tasks must revisit:** P27 (release hardening) must reconcile the daemon's sandbox entitlements with notarization/hardened-runtime and codesign the LaunchAgent; the P05 snapshot-swap contract must be honored exactly (a snapshot that fails `validateGraph` in `libs/audio/src/AudioGraph.cpp` must be dropped before publish, never partially applied); the durable registry interacts with P08 autosave/recovery and must not double-apply on crash recovery.

## Acceptance Gates
- `cmake --build build && ctest --test-dir build -R 'mcp_live_bridge|daemon_lifecycle|mcp_sandbox'` passes.
- **Live mutation:** with the DAW running and a project attached with `Capability::Edit`, an MCP `apply` mutates the authoritative manifest and the bound UI reflects it without reload; a snapshot is published to the engine; the same edit appears in the UI undo history (one timeline). An invalid command (failing `ICommand::validate`) returns `validationOk:false`, is **not** applied, and **no** snapshot reaches the graph (assert engine graph unchanged).
- **Interleaving:** a UI edit followed by an MCP edit followed by a UI undo unwinds in correct order with no divergence between manifest, registry, and published snapshot.
- **Addressable durable undo:** apply three distinct ops with known auditIds; `undoById(auditId#2)` produces the defined result; restart the daemon (process exit + relaunch) with the same attached project and confirm the registry reloads and the same undo-id remains addressable.
- **Lifecycle:** `lamusica-mcpd --install` registers a launchd service (verifiable via `launchctl print gui/$UID/com.lamusica.mcpd`); `--health` reflects the real OS service state and PID liveness, not a static enum; `--stop` unregisters it; KeepAlive restart-on-crash is documented and demonstrable.
- **Sandbox negative tests (must hold with the protocol denylist removed):** the daemon cannot spawn a subprocess (`fork`/`exec` denied at the OS layer); cannot read/write outside the project/asset/cache jail; `..`, absolute out-of-jail, and symlink-escape paths are all denied by the kernel, not by string matching.
- **Consent + RT:** mutation is rejected when `allowMcpProjectMutation` is false; destructive ops (`commandNameRequiresConfirmation`, `EditTools.cpp:31-35`) require a matching `confirmationToken` (`EditTools.cpp:431`); a realtime-safety/TSan check (P06 harness) confirms no allocation/lock/IO/JSON occurs on the audio callback during a live MCP edit.

## Notes For Agents
- **RT contract:** the audio callback only ever consumes the P05 lock-free SPSC queue and a published snapshot. Today `RealtimeCommandQueue` uses plain `std::size_t` indices with no atomics (`libs/audio/include/lamusica/audio/AudioEngine.hpp:74-89`, `AudioEngine.cpp:146-184`) and there is no graph-snapshot push API — both are delivered by P05; do not regress them. Graph recompile + `validateGraph` must run on the bridge thread; only the validated, swappable snapshot pointer crosses to the audio thread.
- **Single authoritative document:** the bug to kill is `applyCommand` operating on a manifest the UI does not see and `main_juce.cpp` having no daemon reference at all. The bridge must hold *the* `ProjectDocument` the shell from P19 opened, and MCP edits flow through that same `CommandHistory`.
- **Undo-id grounding:** `CommandMetadata{commandId, auditId, name}` (`Command.hpp:24-28`) is already plumbed through every `EditToolResult` (`EditTools.cpp:692-705`); reuse `auditId` as the registry key rather than inventing a parallel id. Persist via existing `serialize()`/`commandFromSerialized` (`Command.hpp:1032`).
- **Do not hand-roll JSON in new code.** Existing `escapeJson` (`EditTools.cpp:11-21`) and `splitWords` line grammar (`Protocol.cpp:15-23`) are stubs replaced by the P24 JSON-RPC server; emit bridge/lifecycle responses through P24's serializer, not these.
- **Sandbox is defense-in-depth.** Keep `classifyForbiddenProtocolRequest` (`Protocol.cpp:388-405`) but treat it as advisory; the acceptance bar is that confinement holds with that function deleted.
- **Maps to existing task docs:** completes **034 Production MCP App Integration** (`docs/tasks/034-production-mcp-app-integration.md` — which today does not even acknowledge the in-memory lifecycle fiction or the missing live bridge, and whose "UI undo history includes MCP edits" gate this task actually makes true) and the daemon-foundation half of **019 MCP Daemon Foundation** (`docs/tasks/019-mcp-daemon-foundation.md` — its `install/launch/stop/health` deliverable becomes a real launchd service here). Honors 034's "DAW command validator remains authoritative" / "no MCP-only mutation paths" directive.
