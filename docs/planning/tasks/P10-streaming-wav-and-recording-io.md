# P10 Streaming WAV read/write, incremental recording-to-disk, and accurate sample-rate header handling

> **Area:** audio-engine · **Severity:** medium · **Effort:** M · **Phase:** Engine correctness & I/O foundations (capture/render memory budgets)
> **Depends on:** P04, P09 · **Blocks:** P17, P28

## Objective
Make audio file I/O scale to real-world media instead of toy assets. Today the WAV reader slurps the entire `data` chunk into a `std::vector<char>` (`libs/audio/src/WavFile.cpp:202-203`) before decoding, `RecordingSession` grows the whole take in one heap vector before commit (`libs/audio/src/Recording.cpp:204-207`), and the CLI/offline render allocates one full-arrangement buffer (`libs/audio/src/Bounce.cpp:115-122`, `libs/session/src/ApplicationSession.cpp:1505-1511`). Deliver a streaming/chunked (or memory-mapped) WAV read path for playback/import, incremental streaming-to-disk capture (feeding P17), block-wise offline render under a fixed memory budget, and exact sample-rate/byte-rate header round-trip with a documented non-integer-rate policy. The outcome is a production engine that captures multi-minute takes and bounces long arrangements within a bounded, measured memory ceiling.

## Dependencies
- `P04 — Live CoreAudio device I/O on a realtime thread driving the engine callback` (the realtime capture source whose blocks the streaming writer must drain off-thread).
- `P09 — Faithful GraphCompiler: render real user audio/MIDI content and add an instrument/synth graph node kind` (Sample-node loading at `libs/session/src/GraphCompiler.cpp:206-210` is the read path that must stream rather than copy full files into `node.samples`).

## Decisions To Make
- **mmap vs chunked `pread`.** Memory-mapped read is simplest for random-access Sample nodes but risks SIGBUS on truncated/removed files and complicates the no-blocking-on-audio-thread rule; chunked buffered reads with a small page cache are more portable and fail gracefully. Pick one policy for read; capture writes are sequential so a buffered streaming writer is fine.
- **Non-integer sample-rate policy.** WAV `fmt ` stores `nSamplesPerSec`/`nAvgBytesPerSec` as integer `uint32`. Decide: (a) reject non-integer rates at the I/O boundary; (b) round to nearest integer and record the true rate in the sidecar/manifest (P08/P29); or (c) round and warn. Document the chosen policy and where the authoritative rate lives.
- **Block-wise normalize/dither strategy.** Whole-buffer `normalizePeak` needs the global peak (`Bounce.cpp:68-82`). Choose: two-pass render (pass 1 measures peak, pass 2 writes), single-pass with no-normalize fast path, or a deferred-gain pass over the written file. Triangular dither (`Bounce.cpp:99-110`) must keep its deterministic PRNG seed `0x4C4D5343` and produce byte-identical output regardless of block size — verify the seed is not reset per block.
- **Sample-node memory ownership.** Whether streamed sample data is shared across graph recompiles (P09 recompiles per export, `GraphCompiler.cpp:220`) via a cached reader pool, or re-opened each compile. A pool avoids re-reading large assets on every bounce/stem export (`Export.cpp:121-138` loops compile per stem).
- **Block-size / memory budget default.** Pick the default `blockFrames` and document the resulting peak memory (block buffer × channels × float + writer buffer + per-node graph buffers from `AudioGraph.cpp:217-220`, which are currently allocated full-length and must also be block-sized to actually bound memory).

## Compromises And Tradeoffs
- **Accepted:** keeping `readPcm16Wav` as an eager convenience wrapper for non-hot-path callers (MCP analysis at `libs/mcp_bridge/src/RenderTools.cpp:344,394,416`, manifest reads at `ApplicationSession.cpp:404,769`) rather than converting every call site in this task; only the playback/import/render hot paths must stream. Those eager callers remain a documented risk for very large MCP-driven imports.
- **Accepted:** PCM16/24-only support is unchanged (`WavFile.cpp:220-222`); float/WAV64/AIFF and >4GB `data` chunks are out of scope (RF64 is a later concern, flagged for the release/stress work).
- **Rejected MVP shortcut:** "just `reserve()` a bigger vector" or capping take length — this defeats the bounded-memory goal P17 depends on. The accumulator must be removed, not enlarged.
- **Rejected shortcut:** leaving `renderGraph`'s full-length per-node buffers (`AudioGraph.cpp:217-220`) untouched — block-wise bounce only bounds memory if the underlying graph render is also block-able; if P09 has not yet made `renderGraph` block-friendly, this task must add a block-iterating wrapper rather than claim a false memory bound.
- **Risk for later tasks:** P28 (time-stretch/warp) and P33 (comp/take rendering) will read sample data with non-1:1 input/output mapping; the streaming read API must expose seek + arbitrary-range reads, not just sequential, or those tasks will be forced back to full-buffer loads.

## Acceptance Gates
- `cmake --build build && ctest --test-dir build --output-on-failure` passes, including the new cases below.
- **Bounded capture:** a test recording of ≥ several minutes at 48k stereo completes and commits with measured peak RSS/heap for the capture buffer bounded to ~block size, not proportional to take length (assert allocator high-water or RSS delta below a fixed ceiling).
- **Bounded import/read:** streaming-reading a large (multi-hundred-MB-equivalent) WAV for playback/import stays under the same fixed ceiling; the eager `readPcm16Wav` path is not used on the streamed import.
- **Exact SR round-trip:** write+read at 44100, 48000, 88200, 96000 returns the identical `sampleRate` and a `byteRate == round(sampleRate) * channels * bytesPerSample`; a fractional rate is handled exactly per the documented policy (rejected or rounded-with-record), asserted by test.
- **Byte-identical block-wise render:** for a fixed reference graph, `bounceGraphToWav` with `blockFrames = N` produces a byte-for-byte identical WAV to the previous single-block render across at least two different block sizes and the full-length case (covers the dither-determinism edge case).
- **Recovery:** after simulating an interrupted recording (orphan `.tmp`), `hasInterruptedRecording` + `recoverInterruptedRecording(Recover)` (`Recording.cpp:109-138`) yields a `CommittedRecording` with the correct `frames`/`channels` reconstructed from the streamed file, not from an in-memory take.
- **Edge cases:** zero-length take commit is rejected as before; odd `data` size padding (`WavFile.cpp:212-214`) still handled; truncated file mid-stream surfaces the existing "Unexpected end of WAV" error rather than silently producing short audio.

## Notes For Agents
- **Realtime safety:** the capture path runs from the P04 audio callback. Do NOT call `std::ofstream`, `std::filesystem`, allocate, or back-patch headers on the audio thread. The callback must push interleaved blocks into a lock-free/bounded RT-safe queue (the P05 SPSC mechanism) drained by a dedicated writer thread that owns the `WavStreamWriter`. `RecordingSession::appendInterleaved` (`Recording.cpp:196`) becomes the writer-thread consumer, not a callback-time function — adjust its contract and the `ApplicationSession` offline-render usage (`ApplicationSession.cpp:1519`) accordingly (offline render is not on the audio thread and may stream synchronously).
- **Read path on the audio thread:** Sample-node playback reads file data during `renderGraph` (`AudioGraph.cpp:258-264`). For live playback this read must be served from a preloaded/page-cached region prepared off-thread, or from an mmap region that is guaranteed resident; never issue a blocking disk read inside the render. For offline bounce (not RT) a synchronous streamed read is acceptable.
- **Header back-patching:** open the writer, reserve the 44-byte canonical PCM header, stream frames, then `seekp` to fix `RIFF` size (`WavFile.cpp:112`) and `data` size (`WavFile.cpp:123`) on commit. Verify the rename-after-commit atomicity (`Recording.cpp:215-219`) still holds after the seek/flush.
- **Determinism:** the dither PRNG state (`Bounce.cpp:99`) must be seeded once per render, not per block, or block-wise output diverges from single-block — this is the most likely byte-identity failure.
- **Per-node buffer sizing:** `renderGraph` pre-allocates `totalSamples` per node (`AudioGraph.cpp:217-220`); a block-wise bounce that still calls it with full `frames` does not bound graph memory. Either render in blocks via repeated `renderGraph` calls with sliding `startSample` (already supported by its `startSample`/`frames` signature, `AudioGraph.cpp:205`) or coordinate with P09 to add a stateful block renderer.
- **Existing tasks mapping:** per the plan, this concern is owned by no `026-036` doc; it supports `docs/tasks/009-audio-import-recording-export.md` and `033-browser-assets-import-record-export-ux.md` (recording/import/export) and `035-performance-stress-realtime-verification.md` (stress/memory budgets). Treat those as the legacy descriptions this task supersedes for streaming I/O; do not duplicate their UX scope here.
