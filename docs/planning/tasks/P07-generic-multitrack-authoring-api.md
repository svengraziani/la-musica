# P07 Generic command-authored multitrack session API (add/remove/rename tracks, audio+MIDI clips, sends, automation, markers) with dirty-tracking and Save-As

> **Area:** session-model · **Severity:** blocker · **Effort:** XL · **Phase:** Foundations — Session & Authoring Core
> **Depends on:** P01, P03 · **Blocks:** P08, P09, P19, P24

## Objective
Replace the FirstTrack-only `ApplicationSession` surface with a generic, validated, command-authored multitrack API over `ProjectManifest` that covers all six `TrackType` values, audio clips, MIDI clip references, plugin references, sends/routing, automation lanes, markers, and tempo/time-signature edits. Every mutation routes through the unified `commands::CommandHistory` (P03), validates via `validateProjectManifest`, and either commits atomically or rejects with no partial mutation. Ship a genuine dirty/clean state machine (dirty on mutation, clear only on save, reflecting undo/redo) and a `saveProjectAs(path)` that copies the manifest plus `Assets/` and sidecars into a new independent bundle. This is the prerequisite for any from-scratch project and for the CLI/UI/MCP authoring surfaces — not an MVP, a production authoring core.

## Dependencies
- P01 — Fix on-disk format integrity: schema conformance, lossless/locale-independent floats, JSON escaping, durable atomic save
- P03 — Unify all edit-domain commands under one polymorphic ICommand history with serialization, coalescing, bounded depth, and graceful error handling

## Deliverables
- Change `libs/session/include/lamusica/session/ApplicationSession.hpp` and `libs/session/src/ApplicationSession.cpp`: add generic authoring methods — `addTrack`, `removeTrack`, `renameTrack`, `createAudioClip`, `createMidiClip`, `addPlugin`, `addSend`, `addAutomationLane`, `addMarker`, `setTempo`, `setTimeSignature`. Each takes caller-supplied or session-generated ids, constructs the corresponding `commands::CommandPtr`, and executes it through a single owned `commands::CommandHistory` against `document_->mutableManifest()`.
- Replace the whole-manifest snapshot stacks (`undoSnapshots_`/`redoSnapshots_`, `rememberUndoSnapshot`, `restoreEditSnapshot` at `ApplicationSession.cpp:2250-2284`) with delegation to `commands::CommandHistory::{execute,undo,redo}` (declared `libs/commands/include/lamusica/commands/Command.hpp:1010-1024`). Wire `undoLastEdit`/`redoLastEdit` (`ApplicationSession.hpp:329-330`) and `status().canUndo/canRedo/undoDepth/redoDepth` to the history's per-command stacks.
- New command kinds + factories in `libs/commands/include/lamusica/commands/Command.hpp` and `libs/commands/src/Command.cpp`, registered in `registeredProjectCommandNames()` / `commandFromSerialized` (Command.hpp:1031-1035): `RemoveTrackCommand` (cascade), `CreateMidiClipCommand` (manifest-level `MidiClipReference`, distinct from the `MidiClipStore`-based note commands), `AddPluginReferenceCommand` (manifest `PluginReference`), `AddAutomationLaneCommand` (manifest `AutomationLane`, distinct from the `AutomationLaneStore`-based `AddAutomationPointCommand` at Command.hpp:708), and `AddTimeSignatureEventCommand`. Reuse existing factories where present: `makeAddTrackCommand`, `makeAddClipCommand`, `makeAddMarkerCommand`, `makeAddRoutingConnectionCommand` (sends), `makeSetTrackNameCommand`, `makeAddTempoEventCommand` (Command.cpp:3810-3935).
- `removeTrack` cascade: a `RemoveTrackCommand` (or a `TransactionCommand`, Command.hpp:992, composing removals) that deletes the track and every dependent entity — clips on the track and their `midiClips` references, `trackMix`, `plugins`, automation lanes whose `targetId` resolves to the track or its plugins, and routing whose source/destination is the track — capturing all removed state for exact undo. Must reject (no mutation) if removing a `Master`/`Group`/`Return` would leave dangling routing that cannot be cascaded; the post-apply manifest must pass `validateProjectManifest` (`libs/session/src/ProjectManifest.cpp:966`).
- Decouple from save-on-every-edit: delete the `document_->save()` + forced `status_.dirty = false` epilogue repeated across every setter (e.g. `ApplicationSession.cpp:646-648`, `680-682`, `706-708`, and the rest of the `status_.dirty = false` sites at lines 648..1900). Mutations set `dirty = true`; only `saveProject()`/`saveProjectAs()` and undo-to-clean clear it.
- `saveProjectAs(std::filesystem::path)` on `ApplicationSession`: serialize the current manifest into a fresh bundle directory, copy `Assets/` (assets resolve as `projectRoot / asset.relativePath`, base dir `"Assets"` — `ProjectManifest.hpp:34`, `libs/session/src/Assets.cpp:168`, `244`) and any sidecars, then switch `document_->path_` to the new bundle and add it to recents. Reuse `ProjectDocument::create`'s "already exists" guard (`libs/session/src/ProjectDocument.cpp:31`) and atomic-rename save (`ProjectDocument.cpp:50-78`). Reject nested-inside-self and occupied targets.
- Make `createProject` (`ApplicationSession.cpp:594`, backed by `ProjectDocument::createEmpty`, `ProjectDocument.cpp:21`) the canonical from-scratch entry; demote `createFirstTrackProject` (`ApplicationSession.cpp:602`) to an opt-in template, not the default the CLI hardwires.
- New integration test in `tests/unit/` (registered via `tests/CMakeLists.txt`): empty project -> add 2 Audio + 1 MIDI + 1 Group + 1 Return + 1 Master track, audio clips + a MIDI clip reference, one plugin, one send (routing), one automation lane, one marker, a tempo and time-signature edit; assert `validateProjectManifest` passes throughout, then save, reopen, and assert structural equality of the reloaded `ProjectManifest`. Add negative tests for each rejected illegal op and a `removeTrack` cascade test.

## Decisions To Make
- Id-generation policy: caller-supplied ids (deterministic, MCP/test-friendly) vs session-minted ids. Recommendation: accept optional caller ids, mint a collision-checked stable id otherwise; required for the "arbitrary ids round-trip" acceptance gate.
- Cascade-vs-reject contract for `removeTrack` on structural tracks (`Master`, `Group`, `Return`): which removals auto-cascade routing and which are hard-rejected. Settle whether a project must always retain exactly one `Master`.
- Whether `addSend` is modeled purely as a `RoutingConnection` (current manifest shape, `ProjectManifest.hpp:86`) or gains a send-level (gain/pre-post) record; the manifest has no send gain field today — decide if P07 extends the schema (touches P01/P08) or defers.
- Dirty-tracking granularity: a single document-level dirty flag vs reflecting "clean at last-saved history position" (so undoing back to the saved state clears dirty). Recommend the latter; record the saved history index.
- `saveProjectAs` semantics: deep copy of all `Assets/` and sidecars vs reference/relink of external media; and whether autosave/recovery sidecars are carried over (coordinate with P08).
- Transactionality mechanism: per-op bespoke commands vs composing `TransactionCommand` (Command.hpp:992) for multi-entity ops like cascade-remove. Recommend `TransactionCommand` for cascades to get all-or-nothing apply/undo for free.

## Compromises And Tradeoffs
- Accepted: P07 authors the validated session model and command log only; it does not mirror edits into the live audio graph — that snapshot/compile path is P05/P09. The manifest remains the single source of truth and tests assert structural round-trip, not audio output.
- Accepted: sends stay as plain `RoutingConnection` rows unless the schema-extension decision above says otherwise, to avoid forcing a P01 schema bump inside P07.
- Rejected MVP shortcut: keeping the whole-manifest snapshot undo and the save-on-every-field-edit pattern. They make N edits cause N disk writes and cannot express per-command undo-ids the UI (P19) and MCP undo registry (P24/P25) require — this task explicitly removes them.
- Rejected: leaving `create-project` wired to `makeFirstTrackStarterManifest` (covers_gaps: "create-project produces an unrenderable empty project" / "editing verbs hard-gated to the canned starter"). The empty-project path becomes first-class here; the CLI verb rewrite lands in the P26 CLI productionization pass but P07 must not regress it.
- Risk later tasks must revisit: P08 will add binary sidecars for MIDI/automation/plugin state — `createMidiClip`/`addAutomationLane`/`addPlugin` only persist *references* now (`dataId`, lane regions inline, plugin `identifier`); P08 owns the opaque payloads. P09's `GraphCompiler` will consume these manifest entities, so id stability and validation invariants set here are load-bearing downstream.

## Acceptance Gates
- `cmake --build build --target lamusica_tests && ctest --test-dir build --output-on-failure` is green, including the new authoring integration test and negative tests.
- From-scratch round-trip: build an empty project with arbitrary user-supplied ids, author the full multitrack scene, `saveProject`, `openProject` the same path, and assert the reloaded `ProjectManifest` is structurally equal field-for-field (tracks, clips, midiClips, routing, trackMix, plugins, automation, markers, tempo/timesig). Re-serializing must be byte-identical given P01's lossless float and JSON-escape guarantees.
- Atomicity: `createAudioClip`/`createMidiClip` on a missing track id, duplicate track/clip/plugin/automation/marker ids, and dangling routing source/destination each throw and leave the manifest byte-identical to its pre-call serialization (assert via `serializeProjectManifest` snapshot before/after).
- Cascade: after `removeTrack` on a track carrying clips, a MIDI reference, a plugin, an automation lane targeting it, mix state, and routing, `validateProjectManifest` passes and no entity references the removed track id; `undoLastEdit` restores the exact pre-removal manifest.
- Dirty state machine: `status().dirty` is false on open, true after any mutation, false after `saveProject`/`saveProjectAs`; undoing back to the last-saved history position returns `dirty=false` and redoing past it returns `dirty=true`. `canUndo/canRedo/undoDepth/redoDepth` track `CommandHistory`.
- Write-count: instrument the test (e.g. count `project.json` mtimes or wrap save) to assert that N mutations cause 0 disk writes until an explicit save, then exactly one write per `saveProject`.
- `saveProjectAs`: target produces a complete independent bundle (manifest + copied `Assets/`), the original is unchanged and re-openable, the active path switches to the new bundle, and the call rejects (throws, no copy) when the target already exists or is nested inside the source bundle.

## Notes For Agents
- Realtime/safety: this is non-realtime session/authoring code on the UI/control thread. No audio-thread work here — but every mutation must produce a manifest that `validateProjectManifest` accepts so that P05/P09 can build RT-safe validated graph snapshots from it; never let an invalid manifest reach a save or a snapshot consumer.
- Apply-then-validate pattern: prefer applying the command onto a working copy (or via `ICommand::apply`) and calling `validateProjectManifest` on the result before committing to `document_->mutableManifest()`; on failure, discard and throw so the live manifest is untouched (mirrors the existing pre-save guard at `ApplicationSession.cpp:1351`).
- Validation already enforces the invariants you must respect: clip->track and clip->asset existence (`ProjectManifest.cpp:1078-1083`), MIDI reference->clip (`1108`), routing endpoints (`1120-1128`), unique plugin ids + plugin->track (`1149-1166`), automation target resolution by `AutomationTargetKind` (`1176-1190`), and duplicate-id checks throughout. Author commands to satisfy these, not to re-implement them.
- The `CommandHistory` is per-command and already supports `serialize`/`commandFromSerialized`/`replaySerializedCommands` (Command.hpp:1031-1035) — keep new commands serializable so P24's MCP edit tools and any project-level command-log replay work. Add new commands to `registeredProjectCommandNames()`.
- `ApplicationSessionStatus` carries FirstTrack-specific fields (`firstTrackReady`, `firstTrackEditable`, `starter*` at `ApplicationSession.hpp:28-55`); keep them for template projects but stop gating generic editability on them. Generic counts (`trackCount`, `clipCount`, `pluginCount`, `automationLaneCount`, `markerCount`) already exist on the status struct.
- Assets bundle copy: assets resolve as `projectRoot / relativePath` under the `"Assets"` base dir (`Assets.cpp:168`, `244-246`); `saveProjectAs` must copy that tree and any future sidecars. Use `std::filesystem::copy_file`/`copy` with overwrite guards, mirroring `Assets.cpp:282`.
- Mapping to existing tasks docs: P07 delivers the substance of deferred `docs/tasks/005-session-model-project-format.md` and `docs/tasks/006-undoable-command-layer.md` at product grade, and is the precondition the whole `docs/tasks/027-034` UI/MCP line assumes. It demotes the "first-track bootstrap" surface that `docs/tasks/027-production-daw-shell.md` treats as a decision item — generic create/author becomes the default and the canned starter becomes an opt-in template.
