# P19 Interactive DAW shell: command manager, native menus, document lifecycle, and live data binding

> **Area:** daw-ui · **Severity:** blocker · **Effort:** L · **Phase:** UI & Interaction — Application Shell
> **Depends on:** P07 · **Blocks:** P20, P21, P25, P30, P31, P34

## Objective
Replace the five static, read-only `StatusPanel` `juce::Label` regions (frozen after a single `refresh()`, transport hardcoded `"Stopped"`, leaking a temp untitled bundle on launch) with a real interactive application shell: a focusable `juce::Component` region hierarchy, one `juce::ApplicationCommandManager` + `CommandTarget` tree that every menu/key/button routes through, and a live binding lifecycle (a ~30 Hz `juce::Timer` plus invalidate-on-command). Ship a native macOS menu bar with the ten conventional menus whose enablement is driven by `ApplicationSessionStatus`, a full document lifecycle (New in-memory / Open / Save / Save-As / Close-with-prompt / Recent / autosave / crash-recovery / edited indicator), undo/redo affordances, and `KeyPressMappingSet`-based shortcut routing with preference-supplied custom equivalents. Delete the orphaned `main.mm`. This is the production shell every downstream UI task docks into — not a placeholder.

## Dependencies
- P07 — Generic command-authored multitrack session API (add/remove/rename tracks, audio+MIDI clips, sends, automation, markers) with dirty-tracking and Save-As

## Decisions To Make
- Command host topology: a single `ApplicationCommandTarget` on `MainComponent` with a `getNextCommandTarget()` chain through the five regions, vs. each region as an independent target. Recommend the chain so focused-region commands (cut/copy/paste, zoom) resolve contextually while global commands (file/transport) always reach the root.
- Recovery model boundary with P08: P19 wires the recovery *prompt* against `recoverLastProject`, but the autosave cadence, sidecar payloads, and snapshot format are owned by P08 (binary sidecar + autosave/recovery). Decide whether P19 ships a thin manifest-only autosave timer now or defers all writing to P08 and only consumes its recovery entry point. Recommend: consume P08's entry point; if P19 lands first, gate autosave behind a feature flag and only persist the manifest.
- "New = no temp file" guarantee: `createProject` does not write to disk (`ProjectDocument::createEmpty` constructs in memory), but recents and recovery need a real path. Decide the placeholder-path policy for an unsaved New document (empty path until first Save-As, with menu items that require a path disabled) vs. a sandbox scratch path.
- macOS menu specifics: which commands belong under the App menu (About, Preferences, Quit, Services) vs. File; Preferences window scope for P19 (shell-only: keyboard map + recents) vs. deferring the audio/MIDI/plugin preference panes to P20/P22.
- Visual system baseline (027's open decision): colors, spacing, typography, component states, minimum window size, and panel docking/resize behavior. P19 sets the shared `juce::LookAndFeel` and dock contract that P21/P22/P23 build on.
- Async `FileChooser` vs. modal: macOS sandbox + notarization favor `launchAsync` with a completion lambda. Settle the re-entrancy/state-machine handling so a chooser open during a dirty-close prompt cannot double-fire.

## Compromises And Tradeoffs
- Accepted: the five regions render bound *data* and route *commands* but are not yet the full arrangement/mixer/piano-roll editors — those are P21/P22/P23. P19 delivers interactive, state-driven panels (no frozen labels) and the dock/command/menu spine they plug into.
- Accepted: transport binding is a ~30 Hz `juce::Timer` polling `status().transportPlaying`/`playheadSample`, not a push from the audio thread. The real transport bar and UI-to-engine playback bridge is P20; P19 must not put any work on the audio callback and must read only the published status snapshot.
- Rejected MVP shortcut: keeping `StatusPanel` labels with a one-shot `refresh()` (covers_gaps: "Replace read-only StatusPanel labels with a real interactive DAW shell scaffold"). Rejected: launching into a hardwired first-track starter (`createFirstTrackProject`) — the product shell opens clean and offers New/Open/Recent (covers_gaps: "no bootstrap commands").
- Rejected: leaving the orphaned `main.mm` and `main.cpp` smoke binary as the de-facto UI proof. `main.mm` is deleted; the smoke target `lamusica_daw_smoke` stays as a headless session harness but is no longer the menu/route source of truth.
- Risk later tasks must revisit: autosave/recovery payload format is P08's; if P19's interim manifest-only autosave ships, P08 must migrate it. The MCP-to-running-DAW bridge (P25) and addressable undo registry depend on this command manager exposing stable command ids — keep the `CommandIDs` enum and the session command-id mapping stable. Accessibility (P30) requires every region to be a real focusable component with an accessible name now, not a `Label`.

## Notes For Agents
- Realtime safety: the shell touches the engine only by reading the published `ApplicationSessionStatus` snapshot and by authoring commands through `ApplicationSession`. No allocation/locks/file-IO/logging on any audio callback. The 30 Hz transport timer reads `status()` values (`transportPlaying`, `playheadSample`, `renderFrames` in `ApplicationSession.hpp`), it does not call into the engine render path. The UI->engine playback push is explicitly P20's job.
- Session API is already shaped for this shell: `createProject`/`openProject`/`saveProject`/`closeProject`/`recoverLastProject`, `recentProjects()`, `routeMenuCommand()` (with `MenuCommandRoute{command,panel,handled,enabled}`), `focusPanel()`/`focusedPanel()`, `setKeyboardShortcut()`, `undoLastEdit()`/`redoLastEdit()`, and `status().{dirty,canUndo,canRedo,undoDepth,redoDepth,hasOpenProject}` all exist (`libs/session/include/lamusica/session/ApplicationSession.hpp`; impls `libs/session/src/ApplicationSession.cpp:594,619,1917,1929,1943,1960,1964`). P19 is mostly *binding* this surface to JUCE, not adding session logic. The `ApplicationPanel` enum (Browser/Timeline/Inspector/Mixer/Transport) and `ApplicationPreferences.keyboardShortcuts` are the focus/shortcut contract.
- Dirty semantics depend on P07: today every setter force-clears `status_.dirty` after a `document_->save()` (e.g. `ApplicationSession.cpp:646-648`). P07 removes that save-on-every-edit and makes mutations set dirty until Save/undo-to-clean. Do not build the edited-indicator gating until P07's dirty state machine is in place, or it will read a permanently-clean flag.
- JUCE wiring: `juce::ApplicationCommandManager`, `ApplicationCommandTarget`, `MenuBarModel`/`setMacMainMenu`, `KeyPressMappingSet`, and `juce::FileChooser` all live in `juce_gui_basics`, pulled in transitively by the already-linked `juce::juce_gui_extra` (`apps/daw/CMakeLists.txt:9`). The `Info.plist.in` already declares `NSMicrophoneUsageDescription`/`NSAppleEventsUsageDescription` and `LSMinimumSystemVersion 14.0` — no plist change needed for the shell itself.
- Tests run headless in CI (no display). Keep `shell_tests.cpp` driving the command/document layer through `ApplicationSession` (the pattern `bootstrap_tests.cpp` already uses at lines 2542-2586 and the `routeMenuCommand` assertions at 2891+, 3411-3505). Any `juce::Component`-level construction in a test must avoid requiring a window/peer; prefer asserting command enablement/perform side effects on the session.
- Mapping to existing task docs: P19 **directly implements** `docs/tasks/027-production-daw-shell.md` (production DAW shell deliverables and decisions) and **resolves its open `main.mm`-deletion decision** by deleting it. It supersedes the StatusPanel-era `apps/daw/src/main_juce.cpp` baseline introduced around 026 (`docs/tasks/026-pin-juce-production-app.md`). The downstream editor UIs that 028-033 sketch (timeline 028, clip editing 029, MIDI/piano-roll/drums 030, plugin UI 031, mixer/automation 032, browser/import/export 033) become P21/P22/P23 and dock into this shell; do not implement them here.
